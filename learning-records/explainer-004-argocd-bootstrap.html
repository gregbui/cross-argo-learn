ArgoCD App-of-Apps Bootstrap

ArgoCD is the GitOps engine that syncs infrastructure manifests from Git to your cluster. The App-of-Apps pattern creates a hierarchical bootstrap chain: one "root" Application manages child Applications, which manage the actual resources. This explainer shows the concrete YAML for bootstrapping the entire Crossplane + KubeVirt stack from Git.

What we've covered so far
You know XRDs, Compositions, Composition Functions, and Provider installation. Now we tie it all together with ArgoCD — the layer that makes the whole stack GitOps-driven.

1. The App-of-Apps Pattern

An ArgoCD Application declares a Git repository path, a target revision, and a destination cluster/namespace. An App-of-Apps is an Application whose source points to a directory containing other Application manifests:

Bootstrap Hierarchy
Git Repo → Root App manages Infra App syncs Crossplane → VMs

The hierarchy for our stack:

Root Application (argocd namespace)
├── Infrastructure Application (crossplane-system namespace)
│   ├── Crossplane core
│   ├── Provider: KubeVirt
│   ├── CompositionFunctions: patch-and-transform, go-templating
│   ├── XRD: XVirtualMachine
│   ├── Composition: vm-kubevirt
│   └── ProviderConfig: default
└── Workloads Application (workloads namespace)
    ├── dev/web-server/claim.yaml
    ├── dev/db-server/claim.yaml
    ├── staging/web-server/claim.yaml
    └── prod/web-server/claim.yaml

2. Repository Structure

Here's the recommended directory layout for the Git repository:

infrastructure-as-code/
├── argocd/
│   ├── root-application.yaml          # The root App-of-Apps
│   ├── infrastructure-app.yaml        # Infrastructure bootstrap
│   └── workloads-app.yaml             # Workload Claims
├── crossplane/
│   ├── provider-kubevirt.yaml         # Provider package
│   ├── provider-config.yaml           # ProviderConfig
│   ├── composition-functions.yaml     # CompositionFunctions
│   └── rbac/
│       ├── serviceaccount.yaml
│       ├── clusterrole.yaml
│       └── clusterrolebinding.yaml
├── xrd/
│   └── xvirtualmachine.yaml           # XRD definition
├── compositions/
│   ├── vm-kubevirt.yaml               # VM Composition
│   └── vm-patch-config.yaml           # FunctionConfig for patches
└── workloads/
    ├── dev/
    │   ├── web-server/
    │   │   └── claim.yaml
    │   └── db-server/
    │       └── claim.yaml
    ├── staging/
    │   └── web-server/
    │       └── claim.yaml
    └── prod/
        └── web-server/
            └── claim.yaml
Directory organization rationale
- argocd/: ArgoCD Application manifests (the bootstrap layer) - crossplane/: Crossplane infrastructure (providers, functions, RBAC) - xrd/: XRD definitions - compositions/: Composition manifests and FunctionConfigs - workloads/: Per-environment VM Claims This separation lets ArgoCD Applications target specific directories and keeps concerns isolated.

3. Root Application (App-of-Apps)

The root Application points to the argocd/ directory, which contains the child Application manifests:

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: root
  namespace: argocd
  labels:
    environment: all
spec:
  project: infrastructure
  source:
    repoURL: https://github.com/your-org/infrastructure-as-code.git
    targetRevision: main
    path: argocd
  destination:
    server: https://kubernetes.default.svc
    namespace: argocd
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
    - CreateNamespace=true

This Application doesn't sync resources directly. It syncs the child Applications defined in argocd/. ArgoCD detects the child Applications and creates them automatically.

4. Infrastructure Application

Syncs all Crossplane infrastructure manifests to the cluster. This is the foundation — nothing works until this Application is healthy.

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: crossplane-infrastructure
  namespace: argocd
  labels:
    environment: infrastructure
spec:
  project: infrastructure
  source:
    repoURL: https://github.com/your-org/infrastructure-as-code.git
    targetRevision: main
    path: crossplane
  destination:
    server: https://kubernetes.default.svc
    namespace: crossplane-system
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
    - CreateNamespace=true
    - PruneLast=true
Sync options explained
CreateNamespace=true — auto-creates the crossplane-system namespace if it doesn't exist.
PruneLast=true — prunes resources last, ensuring Crossplane has time to clean up managed resources before its own CRDs are removed. Critical for safe provider upgrades.

5. Workloads Application

Syncs VM Claims from Git. This is where the actual VM provisioning happens. For production, you typically want manual approval to prevent accidental VM creation.

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: workloads
  namespace: argocd
  labels:
    environment: workloads
spec:
  project: infrastructure
  source:
    repoURL: https://github.com/your-org/infrastructure-as-code.git
    targetRevision: main
    path: workloads
  destination:
    server: https://kubernetes.default.svc
    namespace: workloads
  syncPolicy:
    automated:
      prune: true
      selfHeal: false    # No self-heal — don't auto-delete VMs
    syncOptions:
    - CreateNamespace=true
    - PruneLast=true
Why no selfHeal for workloads?
Crossplane's reconciliation loop will try to recreate any VM that gets deleted. If ArgoCD also tries to self-heal, you get a conflict loop. For workloads, let Crossplane manage the lifecycle — ArgoCD's job is just to sync the Claim YAML from Git. Self-heal is safe for infrastructure (providers, XRDs) but risky for workloads (VMs, databases).

6. Sync Waves: Ordering the Bootstrap

Sync waves control the order in which ArgoCD syncs resources. For our stack, the order matters:

  1. Wave 0: RBAC (ServiceAccount, ClusterRole, ClusterRoleBinding)
  2. Wave 1: Crossplane core (if not pre-installed)
  3. Wave 2: Providers and CompositionFunctions
  4. Wave 3: ProviderConfig
  5. Wave 4: XRDs
  6. Wave 5: Compositions
  7. Wave 6: Workload Claims

Implement sync waves using the argocd.argoproj.io/sync-wave annotation:

# RBAC - Wave 0
apiVersion: v1
kind: ServiceAccount
metadata:
  name: crossplane-kubevirt-provider
  namespace: crossplane-system
  annotations:
    argocd.argoproj.io/sync-wave: "0"

---
# Provider - Wave 2
apiVersion: pkg.crossplane.io/v1beta1
kind: Provider
metadata:
  name: crossplane-community-provider-kubevirt
  namespace: crossplane-system
  annotations:
    argocd.argoproj.io/sync-wave: "2"
spec:
  package: crossplane/community-provider-kubevirt:v0.18.0

---
# XRD - Wave 4
apiVersion: apiextensions.crossplane.io/v1
kind: CompositeResourceDefinition
metadata:
  name: xvirtualmachines.infra.example.com
  annotations:
    argocd.argoproj.io/sync-wave: "4"

---
# Composition - Wave 5
apiVersion: apiextensions.crossplane.io/v1
kind: Composition
metadata:
  name: vm-kubevirt
  annotations:
    argocd.argoproj.io/sync-wave: "5"

---
# Workload Claim - Wave 6
apiVersion: infra.example.com/v1alpha1
kind: VirtualMachineClaim
metadata:
  name: web-server
  namespace: workloads
  annotations:
    argocd.argoproj.io/sync-wave: "6"
Sync wave strategy
ArgoCD syncs resources in ascending wave order. Resources in the same wave are synced in parallel. Use wave -1 for cleanup (prune) operations and wave 999 for final verification steps. Negative waves run before positive ones.

7. The Complete Bootstrap Flow

Here's the complete sequence from a Git commit to a running VM:

End-to-End Bootstrap
Developer commits claim.yaml to Git
↓
ArgoCD detects change (polls or webhook)
↓
ArgoCD syncs claim.yaml to cluster
↓
Crossplane claim controller creates XVirtualMachine XR
↓
XR controller invokes Composition pipeline
↓
Composition renders VirtualMachine + DataVolume managed resources
↓
KubeVirt provider creates VM and DV CRs in cluster
↓
KubeVirt controller provisions VM on OpenShift

8. Drift Detection and Self-Healing

ArgoCD continuously monitors cluster state against Git. For the Crossplane stack, drift behavior differs by layer:

LayerDrift SourceSelf-Heal Behavior
Infrastructure (Providers, XRDs, Compositions) Manual changes, ArgoCD sync Self-heal: ArgoCD restores Git state
Crossplane managed resources Crossplane reconciliation Let Crossplane manage — don't fight its reconciliation
Workload Claims Developer Git commit Self-heal: ArgoCD restores Git state
VirtualMachine CRs (created by Composition) Crossplane + KubeVirt Don't sync — these are managed by Crossplane, not ArgoCD
Critical: Don't sync Crossplane-managed resources
ArgoCD should never sync VirtualMachine, DataVolume, or VirtualMachineInstance CRs. These are created by Crossplane's Composition pipeline. If ArgoCD tries to manage them, you'll get sync conflicts between ArgoCD's declarative sync and Crossplane's reconciliation. Only sync Claims, XRDs, Compositions, and Provider manifests — the inputs to Crossplane, not its outputs.

9. Try This: Design the Sync Policy

Exercise
Your team wants to add a security namespace with its own set of VM Claims. The security team requires:
  1. Manual approval for all syncs to the security namespace
  2. No self-heal (human must approve every change)
  3. Sync only when the security team's branch is merged to main
How would you configure the ArgoCD Application for this namespace? What changes to the repository structure would you make?

Exercise 004: ArgoCD Sync Policy for Security Namespace — Answer Key

Repository Structure Change

Add a security/ directory alongside dev/, staging/, and prod/:

workloads/
├── dev/
├── staging/
├── prod/
└── security/          # New: separate branch, manual approval
    └── firewall/
        └── claim.yaml

ArgoCD Application Configuration

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: workloads-security
  namespace: argocd
  labels:
    environment: security
    approval-required: "true"
spec:
  project: infrastructure
  source:
    repoURL: https://github.com/your-org/infrastructure-as-code.git
    targetRevision: refs/heads/security-team
    path: workloads/security
  destination:
    server: https://kubernetes.default.svc
    namespace: security
  syncPolicy:
    automated:
      prune: true
      selfHeal: false
    syncOptions:
    - CreateNamespace=true
    - PruneLast=true
    # Manual approval required
    syncOptions:
    - PrunePropagationPolicy=foreground
Key configuration points
  • targetRevision: refs/heads/security-team — points to a dedicated branch, not main. Changes only sync when this branch is merged.
  • selfHeal: false — ArgoCD won't auto-correct drift. A human must review and sync.
  • path: workloads/security — isolated directory for security workloads.

In the ArgoCD UI, the Application will show as OutOfSync until a human clicks Sync and approves. You can also use ArgoCD's approval workflow with Webhooks for CI/CD integration.

Alternative: Using ArgoCD Approvals

For even stricter control, use ArgoCD's sync-wave approvals:

# In the claim.yaml
apiVersion: infra.example.com/v1alpha1
kind: VirtualMachineClaim
metadata:
  name: firewall
  namespace: security
  annotations:
    argocd.argoproj.io/sync-wave: "6"
    argocd.argoproj.io/require-sync-approval: "true"

This requires explicit approval in the ArgoCD UI before the resource is synced, adding a human-in-the-loop gate on top of the branch protection.
