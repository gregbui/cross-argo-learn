VM Configuration Data: Passing KV Pairs to Linux VMs

You need to pass key-value pairs (puppet profile names, app config, feature flags) into VMs at provisioning time. Puppet code already reads from a known file on each VM — you cannot change the puppet code or the file it reads.

The question: what's the most efficient and flexible way to get data from your Crossplane XR spec into that existing puppet config file on the VM?

1. The Problem

Constraints:

  • Puppet code reads from a known file path on every VM (e.g. /etc/puppet/profiles.yaml)
  • Cannot modify the puppet code
  • Cannot change the file format puppet expects
  • Each VM needs its own config (different profiles, different app settings)
  • Data must be Git-tracked via the XR

This means the solution is a pipeline:

XR spec (YAML, Git-tracked) │ │ Composition transforms... ▼ Puppet config file format (JSON/YAML/INI — whatever puppet reads) │ │ cloud-init write_files drops it... ▼ Known path on VM (/etc/puppet/profiles.yaml) │ │ Puppet reads it at next run ▼ VM configured

2. Approaches Compared

Approach Efficiency Flexibility Puppet Changes Complexity
1. cloud-init write_files → known path ★★★★★ ★★★★☆ None Low
2. Guest agent push (post-boot) ★★☆☆☆ ★★★★☆ None Medium
3. Puppet ENC API ★★★☆☆ ★★★★★ None High
4. Cron job on VM polling K8s API ★★☆☆☆ ★★★★☆ None Medium

Approach 1 wins for first-boot provisioning. Zero network calls, instant availability, no puppet changes. Approaches 2-4 are alternatives if you need runtime updates.

3. Recommended: cloud-init write_files → Known Path

The composition transforms XR spec fields into the exact file format puppet reads, and cloud-init drops that file at the known path. No puppet code changes.

┌─────────────────────────────────────────────────────┐ │ XR Spec (Git-tracked) │ │ spec.configData: │ │ puppet_profile: ["base", "webserver"] │ │ app_environment: production │ │ db_host: db.internal.example.org │ │ features: │ │ dark_mode: true │ │ beta_api: false │ └──────────────────────┬──────────────────────────────┘ │ Composition transforms │ XR → puppet file format ▼ ┌────────────────┐ │ Composition │ │ generates │ │ user-data: │ │ write_files: │ │ /etc/puppet/ │ │ profiles.yaml │ └────────┬───────┘ │ Dropped at boot ▼ ┌────────────────┐ │ VM Boot │ │ /etc/puppet/ │ │ profiles.yaml │ │ (puppet file) │ └────────┬───────┘ │ Puppet runs ▼ ┌────────────────┐ │ Puppet reads │ │ known file → │ │ applies config│ └────────────────┘

4. XRD Schema for Config Data

Define a configData field on the XR. The structure is flexible — users pass whatever they need, and the composition maps it to the puppet file format.

# xrd/xvirtualmachine.yaml
apiVersion: apiextensions.crossplane.io/v1
kind: XRD
metadata:
  name: xvirtualmachines.example.org
spec:
  versions:
    - name: v1alpha1
      schema:
        openAPIV3Schema:
          type: object
          properties:
            spec:
              type: object
              properties:
                # ── Config data for puppet ──
                configData:
                  type: object
                  description: >
                    Key-value pairs passed to the VM. The composition
                    transforms these into the puppet config file format.
                  nullable: true
                  properties:
                    puppet_profile:
                      type: array
                      items:
                        type: string
                      description: >
                        List of puppet profiles to apply.
                        Example: ["base", "webserver", "monitoring"]
                    app_environment:
                      type: string
                      description: Application environment (dev/staging/prod)
                    app_config:
                      type: object
                      description: >
                        Application-specific key-value pairs.
                      nullable: true
                    features:
                      type: object
                      description: Feature flags
                      nullable: true
                    custom:
                      type: object
                      description: >
                        Arbitrary KV pairs. Mapped directly to the
                        puppet config file.
                      nullable: true

                # ── Sensitive data ──
                secretData:
                  type: object
                  description: >
                    Sensitive values. Injected separately from
                    non-sensitive config.
                  nullable: true
                  properties:
                    db_password:
                      type: string
                    api_key:
                      type: string

Usage — user creates a VM with config data:

# workloads/dev/web-server-03.yaml
apiVersion: example.org/v1alpha1
kind: VirtualMachineClaim
metadata:
  name: web-server-03
  namespace: workloads-dev
spec:
  parameters:
    imageTemplate: centos-stream-9
    hostname: web-server-03
    cpu: 2
    memory: 4Gi

    configData:
      puppet_profile:
        - base
        - webserver
        - monitoring
      app_environment: production
      app_config:
        db_host: db.internal.example.org
        db_port: "5432"
        workers: 4
        log_level: info
      features:
        dark_mode: true
        beta_api: false
      custom:
        nginx_worker_connections: "2048"
        redis_maxmemory: "256mb"

    secretData:
      db_password: "s3cr3t"
      api_key: "ak-123456"

5. Composition: Transform XR → Puppet File Format

The composition reads the XR spec and generates the puppet config file content. The exact transformation depends on what format puppet reads. Here are the three most common cases:

5.1 If puppet reads YAML

# compositions/xvirtualmachine-pipeline.yaml
- step: transform-config
  functionRef:
    name: crossplane-function-patch-and-transform
  inputs:
    - apiVersion: pt.fn.crossplane.io/v1
      kind: Resources
      resources:
        - name: configmap
          base:
            apiVersion: v1
            kind: ConfigMap
            metadata:
              name: '{{ .XR.metadata.name }}-puppet-config'
              namespace: '{{ .XR.metadata.namespace }}'
            data:
              # This is the file puppet reads — generated from XR spec
              profiles.yaml: |
                # Auto-generated by Crossplane — do not edit
                # Source: {{ .XR.metadata.namespace }}/{{ .XR.metadata.name }}
                ---
                puppet_profile:
                {{- range .XR.spec.configData.puppet_profile }}
                  - {{ . | quote }}
                {{- end }}
                {{- if .XR.spec.configData.app_environment }}
                app_environment: {{ .XR.spec.configData.app_environment | quote }}
                {{- end }}
                {{- if .XR.spec.configData.app_config }}
                app_config:
                {{- range $key, $value := .XR.spec.configData.app_config }}
                  {{ $key }}: {{ $value | toPrettyJson }}
                {{- end }}
                {{- end }}
                {{- if .XR.spec.configData.features }}
                features:
                {{- range $key, $value := .XR.spec.configData.features }}
                  {{ $key }}: {{ $value }}
                {{- end }}
                {{- end }}
                {{- if .XR.spec.configData.custom }}
                custom:
                {{- range $key, $value := .XR.spec.configData.custom }}
                  {{ $key }}: {{ $value | toPrettyJson }}
                {{- end }}
                {{- end }}

5.2 If puppet reads JSON

            data:
              # JSON format — puppet reads this
              profiles.json: |
                {{- $root := dict -}}
                {{- if .XR.spec.configData.puppet_profile -}}
                  {{- $_ := set $root "puppet_profile" .XR.spec.configData.puppet_profile -}}
                {{- end -}}
                {{- if .XR.spec.configData.app_environment -}}
                  {{- $_ := set $root "app_environment" .XR.spec.configData.app_environment -}}
                {{- end -}}
                {{- if .XR.spec.configData.app_config -}}
                  {{- $_ := set $root "app_config" .XR.spec.configData.app_config -}}
                {{- end -}}
                {{- if .XR.spec.configData.features -}}
                  {{- $_ := set $root "features" .XR.spec.configData.features -}}
                {{- end -}}
                {{- if .XR.spec.configData.custom -}}
                  {{- $_ := set $root "custom" .XR.spec.configData.custom -}}
                {{- end -}}
                {{ $root | toPrettyJson }}

5.3 If puppet reads INI / key=value

            data:
              # INI format
              profiles.ini: |
                # Auto-generated by Crossplane
                [puppet]
                {{- range .XR.spec.configData.puppet_profile }}
                profile = {{ . }}
                {{- end }}
                {{- if .XR.spec.configData.app_environment }}
                app_environment = {{ .XR.spec.configData.app_environment }}
                {{- end }}

                [app]
                {{- range $key, $value := .XR.spec.configData.app_config }}
                {{ $key }} = {{ $value }}
                {{- end }}

                [features]
                {{- range $key, $value := .XR.spec.configData.features }}
                {{ $key }} = {{ $value }}
                {{- end }}

                [custom]
                {{- range $key, $value := .XR.spec.configData.custom }}
                {{ $key }} = {{ $value }}
                {{- end }}

6. Injecting via cloud-init

cloud-init drops the generated config file at the known puppet path. The composition creates a Secret containing the user-data, and KubeVirt injects it at boot.

# compositions/xvirtualmachine-pipeline.yaml — continued
- step: inject-cloudinit
  functionRef:
    name: crossplane-function-patch-and-transform
  inputs:
    - apiVersion: pt.fn.crossplane.io/v1
      kind: Resources
      resources:
        - name: cloudinit-secret
          base:
            apiVersion: v1
            kind: Secret
            metadata:
              name: '{{ .XR.metadata.name }}-cloudinit-secret'
              namespace: '{{ .XR.metadata.namespace }}'
            type: Opaque
            stringData:
              user-data: |
                #cloud-config
                hostname: {{ .XR.spec.guest.hostname | default .XR.metadata.name }}

                users:
                  - name: cloud-user
                    groups: wheel
                    sudo: ['ALL=(ALL) NOPASSWD:ALL']
                    ssh-authorized-keys:
                    {{- range .XR.spec.guest.linux.sshKeys }}
                      - {{ . }}
                    {{- end }}

                # Drop the puppet config file at the known path
                write_files:
                  - path: /etc/puppet/profiles.yaml
                    content: |
                      # Auto-generated by Crossplane — do not edit
                      # Source: {{ .XR.metadata.namespace }}/{{ .XR.metadata.name }}
                      ---
                      puppet_profile:
                      {{- range .XR.spec.configData.puppet_profile }}
                        - {{ . | quote }}
                      {{- end }}
                      {{- if .XR.spec.configData.app_environment }}
                      app_environment: {{ .XR.spec.configData.app_environment | quote }}
                      {{- end }}
                      {{- if .XR.spec.configData.app_config }}
                      app_config:
                      {{- range $key, $value := .XR.spec.configData.app_config }}
                        {{ $key }}: {{ $value | toPrettyJson }}
                      {{- end }}
                      {{- end }}
                      {{- if .XR.spec.configData.features }}
                      features:
                      {{- range $key, $value := .XR.spec.configData.features }}
                        {{ $key }}: {{ $value }}
                      {{- end }}
                      {{- end }}
                      {{- if .XR.spec.configData.custom }}
                      custom:
                      {{- range $key, $value := .XR.spec.configData.custom }}
                        {{ $key }}: {{ $value | toPrettyJson }}
                      {{- end }}
                      {{- end }}
                    permissions: '0644'

                # Drop sensitive data
                - path: /etc/puppet/secrets.env
                  content: |
                    DB_PASSWORD={{ .XR.spec.secretData.db_password | default "" }}
                    API_KEY={{ .XR.spec.secretData.api_key | default "" }}
                  permissions: '0600'

                runcmd:
                  - systemctl enable qemu-guest-agent
                  - systemctl start qemu-guest-agent
                  - chown -R cloud-user:wheel /etc/puppet/
                  - chmod 0644 /etc/puppet/profiles.yaml
                  - chmod 0600 /etc/puppet/secrets.env
                  - systemctl enable puppet
                  - systemctl start puppet

                final_message: "Config applied to {{ .XR.metadata.name }}"

7. Handling Sensitive Data

Non-sensitive config goes inline in cloud-init user-data. Sensitive values (DB passwords, API keys) need special handling:

7.1 Option A: Inline in user-data (simple, less secure)

For non-production or low-risk secrets, inline them in the user-data Secret. They're base64-encoded in K8s but not encrypted at rest.

# In cloud-init user-data:
- path: /etc/puppet/secrets.env
  content: |
    DB_PASSWORD={{ .XR.spec.secretData.db_password }}
    API_KEY={{ .XR.spec.secretData.api_key }}
  permissions: '0600'

7.2 Option B: External Secrets Operator (recommended for prod)

For production, use the OpenShift External Secrets Operator to pull from Vault, AWS Secrets Manager, etc. The composition references the external secret:

# compositions/xvirtualmachine-pipeline.yaml
- step: external-secrets
  functionRef:
    name: crossplane-function-patch-and-transform
  inputs:
    - apiVersion: pt.fn.crossplane.io/v1
      kind: Resources
      resources:
        - name: db-password-secret
          base:
            apiVersion: external-secrets.io/v1beta1
            kind: ExternalSecret
            metadata:
              name: '{{ .XR.metadata.name }}-db-password'
              namespace: '{{ .XR.metadata.namespace }}'
            spec:
              secretStoreRef:
                name: vault-backend
                kind: SecretStore
              target:
                name: '{{ .XR.metadata.name }}-db-password'
                creationPolicy: Owner
              data:
                - secretKey: db_password
                  remoteRef:
                    key: production/db/password
        - name: api-key-secret
          base:
            apiVersion: external-secrets.io/v1beta1
            kind: ExternalSecret
            metadata:
              name: '{{ .XR.metadata.name }}-api-key'
              namespace: '{{ .XR.metadata.namespace }}'
            spec:
              secretStoreRef:
                name: vault-backend
                kind: SecretStore
              target:
                name: '{{ .XR.metadata.name }}-api-key'
                creationPolicy: Owner
              data:
                - secretKey: api_key
                  remoteRef:
                    key: production/api/key

Then in cloud-init, reference the created K8s Secret:

                - path: /etc/puppet/secrets.env
                  contentFrom:
                    secretRef:
                      name: '{{ .XR.metadata.name }}-db-password'
                      key: db_password
                  permissions: '0600'

8. Updating Config Without Reboot

If you need to update config on a running VM (not just at boot), you have two options:

8.1 Puppet runs periodically (default)

If puppet is running on a schedule (every 30min default), it will re-read the config file on each run. To trigger an immediate run:

# Via guest agent (requires guest agent running)
oc virt guest-exec <vm-name> -n workloads-dev -- puppet agent -t

# Via SSH
oc port-forward <vm-name> 2222:22 -n workloads-dev
ssh -p 2222 cloud-user@localhost "sudo puppet agent -t"

8.2 Update the XR → ArgoCD syncs → puppet picks it up

# Update the VirtualMachineClaim
oc edit virtualmachineclaim web-server-03 -n workloads-dev
# Change configData.puppet_profile, save
# ArgoCD detects drift, syncs
# Puppet runs on next cycle, reads new file
# VM reconfigured

This gives you near-real-time config updates without rebooting the VM.

9. Troubleshooting

9.1 Puppet config file not present

# Check if cloud-init ran
oc rsh <vm-debug-pod>
systemctl status cloud-init
cat /var/log/cloud-init.log | tail -20

# Check if file exists
ls -la /etc/puppet/profiles.yaml

# Check cloud-init user-data that was injected
oc get secret <vm-name>-cloudinit-secret -n workloads-dev \
  -o jsonpath='{.data.user-data}' | base64 -d

9.2 Puppet not reading the file

# Verify file format is correct
cat /etc/puppet/profiles.yaml
python3 -c "import yaml; yaml.safe_load(open('/etc/puppet/profiles.yaml'))"

# Check puppet logs
sudo cat /var/log/puppetlabs/puppet/puppet.log | grep -i "profile"

# Force puppet to run and check output
sudo puppet agent -t --verbose

9.3 File permissions wrong

# Check permissions
ls -la /etc/puppet/profiles.yaml
stat -c '%a %U %G' /etc/puppet/profiles.yaml

# Should be 0644 for config, 0600 for secrets
# If wrong, fix:
sudo chmod 0644 /etc/puppet/profiles.yaml
sudo chown root:root /etc/puppet/profiles.yaml

9.4 Sensitive data not injected

# Check if the secret was created
oc get secret <vm-name>-db-password -n workloads-dev

# Check ExternalSecret status
oc get externalsecret <vm-name>-db-password -n workloads-dev -o yaml | grep -A5 status

# Check if secrets.env has content
cat /etc/puppet/secrets.env
