VM Configuration Data: Passing KV Pairs to Linux VMs

You need to pass key-value pairs from your Crossplane composition into the VM at provisioning time — things like puppet profile names, environment variables, application config, feature flags. The question is: what's the most efficient and flexible mechanism?

This explainer compares four approaches, recommends a best practice, and shows how to wire it all together in a Crossplane + KubeVirt stack.

1. The Problem

When a VM boots, it needs configuration data. In a traditional setup, you might use:

  • A configuration management tool (puppet, ansible, chef)
  • Environment variables
  • Config files dropped into standard locations
  • An external API the VM polls at boot

In a Crossplane + KubeVirt stack, the data lives in Kubernetes (XR spec, ConfigMaps, Secrets) and needs to get into the VM's filesystem or environment at first boot. The mechanism must be:

  • Efficient — no extra network calls, no polling, no runtime dependency
  • Flexible — supports nested data, secrets, per-VM overrides
  • Idempotent — safe to re-run, doesn't break existing VMs
  • Auditable — data is versioned in Git via the XR

2. Four Approaches Compared

Approach Efficiency Flexibility Complexity Secrets Git-auditable
1. cloud-init write_files + config drive ★★★★★ ★★★★☆ Low Via Secret Yes (XR)
2. Puppet ENC (external API) ★★★☆☆ ★★★★★ High Via API Partial
3. Environment file (envvars) ★★★★☆ ★★★☆☆ Low Via Secret Yes (XR)
4. Guest agent file push (post-boot) ★★☆☆☆ ★★★★☆ Medium Via Secret Yes (XR)

Approach 1: cloud-init write_files (Recommended)

Drop config files directly into the VM via cloud-init's write_files directive. The data is embedded in the user-data at provisioning time — zero network calls, instant availability.

Best for: puppet profiles, application configs, feature flags, environment variables. Works with any config management tool.

Approach 2: Puppet ENC

Puppet agent queries an external API ( ENC) at boot to get its node classification and variables. The ENC can read from Kubernetes, a database, or any backend.

Best for: large fleets where config changes frequently and you need dynamic classification. Adds an external dependency and latency.

Approach 3: Environment files

Drop /etc/default/ or /etc/sysconfig/ style env files via cloud-init. Simple but limited to flat key=value pairs. No nesting, no complex types.

Best for: simple environment variables, no config management tool involved.

Approach 4: Guest agent file push

Push files into the running VM using the QEMU guest agent (virt-copy-in). Happens after the VM is booted.

Best for: post-provisioning config updates, not first-boot. Adds latency and a runtime dependency on the guest agent.

3. Recommended: cloud-init + Hiera YAML Drop

The best approach for a puppet-based stack: cloud-init write_files that drops a Hiera YAML data file. Here's why:

  • Zero network calls — data is baked into the VM at boot via cloud-init
  • Nested data support — YAML supports maps, lists, nested structures
  • Puppet-native — Hiera is puppet's built-in data lookup system
  • Per-VM overrides — each VM gets its own Hiera data file
  • Git-auditable — data lives in the XR spec, tracked in Git
  • Secrets safe — sensitive values go in K8s Secrets, injected via cloud-init secretRef

4. XRD Schema for Config Data

Define a configData field in your XRD that accepts structured key-value pairs. Use a generic object type so users can pass any nested structure:

# xrd/xvirtualmachine.yaml — configData field
spec:
  versions:
    - name: v1alpha1
      schema:
        openAPIV3Schema:
          type: object
          properties:
            spec:
              type: object
              properties:
                # ── OS-level config (passed via cloud-init) ──
                configData:
                  type: object
                  description: >
                    Key-value pairs passed to the VM at provisioning time.
                    Supports nested structures. Mapped to Hiera YAML by
                    default, but can be used by any config system.
                  nullable: true
                  properties:
                    puppet:
                      type: object
                      description: Puppet-specific config
                      properties:
                        profile:
                          type: array
                          items:
                            type: string
                          description: List of puppet profiles to apply
                        hiera_override:
                          type: object
                          description: >
                            Custom Hiera key-value pairs that override
                            the global hierarchy.
                          nullable: true
                    app:
                      type: object
                      description: Application-specific config
                      nullable: true
                    env:
                      type: object
                      description: >
                        Flat environment variables. Written to
                        /etc/default/ or /etc/sysconfig/ files.
                      nullable: true
                    features:
                      type: object
                      description: Feature flags
                      nullable: true
                    custom:
                      type: object
                      description: >
                        Arbitrary key-value pairs. All keys become
                        YAML files under /etc/vm-config/.
                      nullable: true

                # ── Secrets (injected via K8s Secret) ──
                secretData:
                  type: object
                  description: >
                    Sensitive config data. Injected as a K8s Secret
                    and referenced in cloud-init user-data.
                  nullable: true
                  properties:
                    db_password:
                      type: string
                    api_key:
                      type: string
                    ssh_private_key:
                      type: string

Usage example — user creates a VM with config data:

# workloads/dev/web-server-03.yaml
apiVersion: example.org/v1alpha1
kind: VirtualMachineClaim
metadata:
  name: web-server-03
  namespace: workloads-dev
spec:
  parameters:
    imageTemplate: centos-stream-9
    hostname: web-server-03
    cpu: 2
    memory: 4Gi

    # Config data for puppet and apps
    configData:
      puppet:
        profile:
          - base
          - webserver
          - monitoring
        hiera_override:
          environment: production
          datacenter: dc1
          rack: r05
      app:
        environment: production
        log_level: info
        db_host: db.internal.example.org
        db_port: 5432
        workers: 4
      features:
        dark_mode: true
        beta_api: false
        new_dashboard: false
      env:
        APP_ENV: production
        APP_LOG_LEVEL: info
        APP_WORKERS: "4"
      custom:
        nginx::worker_connections: "2048"
        nginx::keepalive_timeout: "65"
        redis::maxmemory: "256mb"

    # Sensitive data (goes into K8s Secret)
    secretData:
      db_password: "{{ .externalSecret.dbPassword }}"
      api_key: "{{ .externalSecret.apiKey }}"

5. Composition: Generating Config Data

The composition reads the XR's configData and secretData and generates two resources:

  1. A ConfigMap with the non-sensitive config as YAML files
  2. A Secret with the sensitive config

Both are referenced by cloud-init user-data.

5.1 ConfigMap with YAML files

Use the PatchAndTransform function to generate a ConfigMap where each config section becomes a file:

# compositions/xvirtualmachine-pipeline.yaml
- step: generate-configmap
  functionRef:
    name: crossplane-function-patch-and-transform
  inputs:
    - apiVersion: pt.fn.crossplane.io/v1
      kind: Resources
      resources:
        - name: vm-configmap
          base:
            apiVersion: v1
            kind: ConfigMap
            metadata:
              name: '{{ .XR.metadata.name }}-config'
              namespace: '{{ .XR.metadata.namespace }}'
            data:
              puppet-hiera.yaml: |
                # Hiera override data for this VM
                ---
                puppet::profile:
                {{- range .XR.spec.configData.puppet.profile }}
                  - {{ . | quote }}
                {{- end }}
                {{- if .XR.spec.configData.puppet.hiera_override }}
                {{- range $key, $value := .XR.spec.configData.puppet.hiera_override }}
                {{ $key }}: {{ $value | toPrettyJson }}
                {{- end }}
                {{- end }}
              app-config.yaml: |
                ---
                {{- if .XR.spec.configData.app }}
                {{ .XR.spec.configData.app | toPrettyJson }}
                {{- end }}
              features.yaml: |
                ---
                {{- if .XR.spec.configData.features }}
                {{ .XR.spec.configData.features | toPrettyJson }}
                {{- end }}
              envvars: |
                # Environment variables for this VM
                # Source this file: source /etc/default/vm-envvars
                {{- if .XR.spec.configData.env }}
                {{- range $key, $value := .XR.spec.configData.env }}
                export {{ $key }}={{ $value | quote }}
                {{- end }}
                {{- end }}
              custom-configs.yaml: |
                ---
                {{- if .XR.spec.configData.custom }}
                {{ .XR.spec.configData.custom | toPrettyJson }}
                {{- end }}

5.2 Secret for sensitive data

        - name: vm-secret
          base:
            apiVersion: v1
            kind: Secret
            metadata:
              name: '{{ .XR.metadata.name }}-secret'
              namespace: '{{ .XR.metadata.namespace }}'
            type: Opaque
            stringData:
              db_password: '{{ .XR.spec.secretData.db_password | default "" }}'
              api_key: '{{ .XR.spec.secretData.api_key | default "" }}'
              ssh_private_key: '{{ .XR.spec.secretData.ssh_private_key | default "" }}'

6. Injecting into cloud-init

The cloud-init user-data reads from the ConfigMap and Secret, then drops files into the VM's filesystem. Here's the full pattern:

# cloud-init user-data template (used in composition)
#cloud-config

hostname: {{ .XR.spec.guest.hostname | default .XR.metadata.name }}

users:
  - name: cloud-user
    groups: wheel
    sudo: ['ALL=(ALL) NOPASSWD:ALL']
    ssh-authorized-keys:
    {{- range .XR.spec.guest.linux.sshKeys }}
      - {{ . }}
    {{- end }}

# Drop config files into the VM
write_files:
  # Puppet Hiera override (per-VM)
  - path: /etc/puppetlabs/puppet/hieradata/vm-override.yaml
    content: |
      ---
      # Auto-generated by Crossplane — do not edit
      # Source XR: {{ .XR.metadata.namespace }}/{{ .XR.metadata.name }}
      # Generated: {{ now | date "2006-01-02T15:04:05Z" }}
      puppet::profile:
      {{- range .XR.spec.configData.puppet.profile }}
        - {{ . | quote }}
      {{- end }}
      {{- if .XR.spec.configData.puppet.hiera_override }}
      {{- range $key, $value := .XR.spec.configData.puppet.hiera_override }}
      {{ $key }}: {{ $value | toPrettyJson }}
      {{- end }}
      {{- end }}
    permissions: '0644'

  # Application config
  - path: /etc/vm-config/app-config.yaml
    content: |
      ---
      {{- if .XR.spec.configData.app }}
      {{ .XR.spec.configData.app | toPrettyJson }}
      {{- end }}
    permissions: '0644'

  # Feature flags
  - path: /etc/vm-config/features.yaml
    content: |
      ---
      {{- if .XR.spec.configData.features }}
      {{ .XR.spec.configData.features | toPrettyJson }}
      {{- end }}
    permissions: '0644'

  # Environment variables
  - path: /etc/default/vm-envvars
    content: |
      # Environment variables — source this file
      {{- if .XR.spec.configData.env }}
      {{- range $key, $value := .XR.spec.configData.env }}
      export {{ $key }}={{ $value | quote }}
      {{- end }}
      {{- end }}
    permissions: '0644'

  # Custom configs (one file per key)
  {{- if .XR.spec.configData.custom }}
  {{- range $key, $value := .XR.spec.configData.custom }}
  - path: /etc/vm-config/custom/{{ $key }}
    content: |
      {{ $value | toPrettyJson }}
    permissions: '0644'
  {{- end }}
  {{- end }}

  # Sensitive data (from Secret reference)
  - path: /etc/vm-config/secrets.env
    content: |
      # Sensitive config — read-only for root
      # Generated by Crossplane
      {{- if .XR.spec.secretData.db_password }}
      DB_PASSWORD={{ .XR.spec.secretData.db_password }}
      {{- end }}
      {{- if .XR.spec.secretData.api_key }}
      API_KEY={{ .XR.spec.secretData.api_key }}
      {{- end }}
    permissions: '0600'

runcmd:
  # Enable guest agent
  - systemctl enable qemu-guest-agent
  - systemctl start qemu-guest-agent

  # Set ownership on config files
  - chown -R cloud-user:wheel /etc/vm-config/
  - chmod -R 0644 /etc/vm-config/
  - chmod 0600 /etc/vm-config/secrets.env

  # Create a ready marker
  - echo "ready" > /etc/VM_READY

  # Signal puppet to run (if puppet is installed)
  - systemctl enable puppet
  - systemctl start puppet

final_message: "Config data applied to {{ .XR.metadata.name }}"

7. Puppet Integration

With the Hiera override file in place, puppet reads it as part of its hierarchy. Here's how to wire it up:

7.1 Puppet Hiera Configuration

# /etc/puppetlabs/puppet/hiera.yaml (in base image)
---
version: 5
defaults:
  datadir: hieradata
  data_backend: yaml

hierarchy:
  - name: "Per-VM override (Crossplane)"
    paths:
      - "vm-override.yaml"
    data_backend: yaml

  - name: "Environment"
    paths:
      - "%{environment}.yaml"
    data_backend: yaml

  - name: "Common"
    paths:
      - "common.yaml"
    data_backend: yaml

  - name: "Global defaults"
    paths:
      - "_common.yaml"
    data_backend: yaml

7.2 Puppet Module Usage

Inside a puppet module, consume the config data via Hiera lookups:

# manifests/init.pp
class vm_config (
  Array[String] $profile = [],
  Hash        $app       = {},
  Hash        $features  = {},
  Hash        $env       = {},
) {

  # Profiles from Hiera (set by Crossplane)
  $profiles = $profile ? {
    []    => hiera('puppet::profile', ['base']),
    default => $profile,
  }

  # Apply profiles
  include $profiles

  # App config
  if !empty($app) {
    file { '/etc/vm-config/app-config.yaml':
      content => yaml_encode($app),
      owner   => 'root',
      group   => 'root',
      mode    => '0644',
    }
  }

  # Feature flags
  if $features['dark_mode'] {
    include vm_config::dark_mode
  }
  if $features['beta_api'] {
    include vm_config::beta_api
  }

  # Environment variables
  if !empty($env) {
    file { '/etc/default/vm-envvars':
      content => join(keys($env), "\n"),
      owner   => 'root',
      group   => 'root',
      mode    => '0644',
    }
  }
}

8. Alternatives: When to Use Something Else

Scenario Better Approach Why
Config changes frequently (hourly) Puppet ENC API cloud-init is one-time; ENC is dynamic
No config management tool Env vars via cloud-init Simpler than Hiera for flat configs
Need runtime config updates Guest agent file push Push to running VM without reboot
Large teams, many VMs External secret manager Vault/Sealed Secrets for secret rotation
Multi-cloud (not just KubeVirt) UserData format standard cloud-init works on AWS, GCP, Azure too

8.1 Puppet ENC Alternative

If you need dynamic config (changes without rebuilding VMs), deploy an ENC that reads from Kubernetes:

#!/bin/bash
# /etc/puppetlabs/puppet/environments/production/manifests/site.pp
# Uses ENC to classify nodes

node default {
  # Get node data from Kubernetes API
  $node_data = inline_template(<<ERB>>
    <%=
      require 'open3'
      stdout, stderr, status = Open3.capture3(
        "curl -s http://enc-service.vm-config.svc:8080/lookup/#{Facts['networking']['hostname']}"
      )
      JSON.parse(stdout)
    >>

    # Apply profiles from ENC
    $node_data['puppet::profile'].each |$profile| {
      include $profile
    }

    # Set app config
    include vm_config
    $app_config = $node_data['app'] || {}
    $features = $node_data['features'] || {}
  ERB
}

This adds a runtime dependency on the ENC service. Use it only if you need config to change without VM recreation.

9. Troubleshooting

9.1 Config files not appearing in VM

# Check cloud-init status
oc rsh <vm-debug-pod>
systemctl status cloud-init
journalctl -u cloud-init

# Check if write_files succeeded
ls -la /etc/vm-config/
ls -la /etc/puppetlabs/puppet/hieradata/vm-override.yaml

# Check cloud-init log for errors
cat /var/log/cloud-init.log | grep -i error
cat /var/log/cloud-init-output.log

9.2 Puppet not picking up Hiera data

# Check Hiera hierarchy
sudo hiera --debug puppet::profile

# Verify the file exists and is valid YAML
cat /etc/puppetlabs/puppet/hieradata/vm-override.yaml
python3 -c "import yaml; yaml.safe_load(open('/etc/puppetlabs/puppet/hieradata/vm-override.yaml'))"

# Check puppet logs
sudo cat /var/log/puppetlabs/puppet/puppet.log | grep -i hiera

9.3 Secret values not injected

# Check if the Secret was created
oc get secret <vm-name>-secret -n workloads-dev -o yaml

# Check if cloud-init user-data references it correctly
oc get secret <vm-name>-cloudinit-secret -n workloads-dev -o jsonpath='{.data.user-data}' | base64 -d | grep -A5 secrets

# Verify permissions inside VM
ls -la /etc/vm-config/secrets.env
stat -c '%a %U %G' /etc/vm-config/secrets.env

9.4 YAML serialization issues

# If values contain special characters, use toPrettyJson in templates
# Bad: {{ .XR.spec.configData.env.APP_CMD }}  # breaks on spaces
# Good: {{ .XR.spec.configData.env.APP_CMD | quote }}  # wraps in quotes
# Best: {{ .XR.spec.configData.env.APP_CMD | toPrettyJson }}  # full JSON

# Debug by printing the generated user-data
oc get secret <vm-name>-cloudinit-secret -n workloads-dev -o jsonpath='{.data.user-data}' | base64 -d
