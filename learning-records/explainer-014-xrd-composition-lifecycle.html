Overview: The Four-Phase Lifecycle

Creating a VM workload via Crossplane follows a strict sequence. Each phase produces artifacts that the next phase consumes:

Phase 1
XRD schema
→
Phase 2
Composition
→
Phase 3
ArgoCD sync to cluster
→
Phase 4
Claim → running VM

Key principle: You cannot create a Claim until the XRD exists in the cluster. You cannot verify the XRD works until the Composition is deployed alongside it. ArgoCD is the delivery mechanism for Phases 1–3; kubectl (or Git) is the delivery mechanism for Phase 4.

Prerequisites
  • OpenShift cluster with KubeVirt operator installed
  • Crossplane installed with provider-kubevirt configured
  • ArgoCD running with an App-of-Apps bootstrap pointing to your Git repo
  • Local checkout of the infrastructure repo with crossplane/ directory

Phase 1 — Design the XRD Schema

The XRD defines the interface that workload teams will use. It is the contract between the infrastructure team (who writes the Composition) and the application team (who writes the Claim).

Step 1.1 — Create the XRD file

Create crossplane/xrd/virtualmachine.yaml:

apiVersion: apiextensions.crossplane.io/v1
kind: CompositeResourceDefinition
metadata:
  name: xrvirtualmachines.infra.example.com
spec:
  group: infra.example.com
  names:
    kind: XRVirtualMachine
    plural: xrvirtualmachines
  versions:
    - name: v1alpha1
      served: true
      referenceable: true
      schema:
        openAPIV3Schema:
          type: object
          properties:
            spec:
              type: object
              properties:
                cpu:
                  type: integer
                  description: Number of vCPUs
                memory:
                  type: string
                  description: RAM allocation (e.g., "8Gi")
                guestOS:
                  type: string
                  enum: [linux, windows]
                  description: Guest operating system type
                imageName:
                  type: string
                  description: Name of the base OS image (from ImageStream)
                storage:
                  type: object
                  properties:
                    size:
                      type: string
                      default: "20Gi"
                      description: Root disk size
                    storageClass:
                      type: string
                      description: Kubernetes StorageClass for the PVC
                  required: [size]
                network:
                  type: object
                  properties:
                    networkName:
                      type: string
                      description: OpenShift network name (default: default)
                    podCIDR:
                      type: string
                      description: Pod CIDR for the VM
                  required: [networkName]
                userData:
                  type: string
                  format: byte
                  description: Base64-encoded cloud-init or guest-custom-data
              required: [cpu, memory, guestOS, imageName, storage, network]
            status:
              type: object
              properties:
                conditions:
                  type: array
                  items:
                    type: object
                    properties:
                      type:
                        type: string
                      status:
                        type: string
                      reason:
                        type: string
                vmName:
                  type: string
                  description: Name of the provisioned VirtualMachine
                vmStatus:
                  type: string
                  description: Current VirtualMachine phase
Design decisions in this XRD
  • group: infra.example.com — your organization's domain. Use a real domain in production.
  • referenceable: true — allows other Compositions to reference this XR by name (needed for cross-references).
  • status.vmName and status.vmStatus — the Composition writes these back so Claim users can see VM state without knowing the underlying CR.
  • userData is byte format — the Composition will base64-encode the guest-custom-data before passing it to KubeVirt.

Step 1.2 — Verify the XRD is valid

Before deploying, validate the schema locally:

# Apply the XRD to the cluster (or commit to Git if using ArgoCD)
kubectl apply -f crossplane/xrd/virtualmachine.yaml

# Verify the XRD was created
kubectl get xrvirtualmachines.infra.example.com

# Check the XRD's conditions
kubectl describe xrd xrvirtualmachines.infra.example.com

Look for Established: True in the conditions. If it's False, check the reason field — it will tell you if the schema has validation errors.

Common pitfall

If you change the XRD after it has been used, Crossplane will block the update. You must either:

  • Delete all existing Claims and XRs (destructive), or
  • Add new fields as optional with defaults (safe), or
  • Create a new version (v1beta1) and set the old version to served: false.

Step 1.3 — Test the XRD schema with a dry-run Claim

Create a sample Claim to verify the schema accepts valid input:

# crossplane/claim/virtualmachine-sample.yaml
apiVersion: infra.example.com/v1alpha1
kind: VirtualMachine
metadata:
  name: sample-vm
spec:
  cpu: 2
  memory: "4Gi"
  guestOS: linux
  imageName: rhel-9.2-base
  storage:
    size: "20Gi"
    storageClass: ocs-storagecluster-ceph-rbd
  network:
    networkName: default
# Dry-run apply to validate schema acceptance
kubectl apply -f crossplane/claim/virtualmachine-sample.yaml --dry-run=client

# Check that the Claim was created (it will sit in Pending until the Composition exists)
kubectl get virtualmachine sample-vm

At this point the Claim will show CompositeResourceRef: <none> and Ready: False — that's expected. The Composition hasn't been deployed yet.

Phase 2 — Write the Composition

The Composition is the recipe. It tells Crossplane how to turn an XRVirtualMachine into actual KubeVirt resources (VirtualMachine, DataVolume, PVC).

Step 2.1 — Create the Composition file

Create crossplane/composition/virtualmachine.yaml. This example uses the declarative (patchAndSync) pipeline pattern with a Composition Function, which gives you conditional logic and data transforms that pure YAML patches can't express:

apiVersion: apiextensions.crossplane.io/v1
kind: Composition
metadata:
  name: xrvirtualmachine-linux
  labels:
    infra.example.com/composition-type: pipeline
    infra.example.com/guest-os: linux
spec:
  writeConnectionSecretsToNamespace: crossplane-system
  compositeResourceSelector:
    matchLabels:
      infra.example.com/guest-os: linux
  functionRef:
    name: crossplane-contrib-function-patch-and-transform
  patchAndTransform:
    - step: set-infrastructure
      patches:
        # Render the VirtualMachine CR
        - type: fromCompositeFieldSet
          from: spec
          to: metadata.annotations
          transforms:
            - type: string
              string:
                fmt: '{{ .cpu }}-{{ .memory }}-{{ .guestOS }}'
                type: Format

        # Render the DataVolume for the base image
        - type: patchAndSelect
          patch:
            type: fromCompositeFieldSet
            from: spec.storage
            to: spec
          to: spec.resources[0]
          selectedForResources:
            - kind: DataVolume
              apiVersion: cdi.kubevirt.io/v1beta1

        # Render the PVC (if not using DataVolume, this is where you'd create a PVC)
        - type: patchAndSelect
          patch:
            type: fromCompositeFieldSet
            from: spec.storage
            to: spec
          to: spec.resources[1]
          selectedForResources:
            - kind: PersistentVolumeClaim
              apiVersion: v1
Why patchAndTransform?

The patchAndTransform pattern (via the function-patch-and-transform function) is the most widely used Composition pattern because it:

  • Supports conditional rendering (only create a DataVolume if the image isn't cached)
  • Allows data transforms (base64-encoding, string formatting) inline
  • Is simpler than writing a Go or Lua function
  • Is the default pattern in most Crossplane examples

Use a custom Go/Lua function only when you need loops, external API calls, or complex data flows.

Step 2.2 — The full Composition with all managed resources

Here's the complete Composition for a Linux VM with a DataVolume, VirtualMachine, and proper connections:

apiVersion: apiextensions.crossplane.io/v1
kind: Composition
metadata:
  name: xrvirtualmachine-linux
  labels:
    infra.example.com/composition-type: pipeline
    infra.example.com/guest-os: linux
spec:
  writeConnectionSecretsToNamespace: crossplane-system
  compositeResourceSelector:
    matchLabels:
      infra.example.com/guest-os: linux
  functionRef:
    name: crossplane-contrib-function-patch-and-transform
  patchAndTransform:
    - step: create-managed-resources
      resources:
        # ── Resource 0: DataVolume (base image import) ──
        - name: datavolume
          base:
            apiVersion: cdi.kubevirt.io/v1beta1
            kind: DataVolume
            spec:
              storage: {}
          patches:
            - type: fromCompositeFieldSet
              from: spec
              to: spec.storage.storageClass
            - type: fromCompositeFieldSet
              from: spec.storage.size
              to: spec.storage.size
            - type: fromCompositeFieldSet
              from: spec.imageName
              to: spec.source.http.url
              transforms:
                - type: string
                  string:
                    fmt: 'https://image-registry.example.com/%s.qcow2'

        # ── Resource 1: VirtualMachine ──
        - name: virtualmachine
          base:
            apiVersion: kubevirt.io/v1
            kind: VirtualMachine
            spec:
              runStrategy: Halted
              template:
                spec:
                  domain:
                    cpu:
                      cores: 1
                    devices: {}
                    machines:
                      machineType: pc-q35-rhel9.2.0
                  terminationGracePeriodSeconds: 30
                  networks:
                    - name: default-network
                      pod: {}
                  volumes: []
          patches:
            - type: fromCompositeFieldSet
              from: spec.cpu
              to: spec.template.spec.domain.cpu.cores
            - type: fromCompositeFieldSet
              from: spec.memory
              to: spec.template.spec.domain.memory.guest
            - type: fromCompositeFieldSet
              from: spec.guestOS
              to: spec.template.spec.domain.devices.automaticCPUPlacement
              transforms:
                - type: map
                  map:
                    linux: true
                    windows: true
            - type: fromCompositeFieldSet
              from: spec.userData
              to: spec.template.spec.guestCustomData
              transforms:
                - type: string
                  string:
                    encode: Base64
            - type: fromCompositeFieldSet
              from: spec.network.networkName
              to: spec.template.spec.networks[0].name
              transforms:
                - type: string
                  string:
                    fmt: '%s-network'

        # ── Resource 2: PVC (fallback if no DataVolume) ──
        - name: pvc
          base:
            apiVersion: v1
            kind: PersistentVolumeClaim
            spec:
              accessModes:
                - ReadWriteOnce
              resources:
                requests:
                  storage: 20Gi
          patches:
            - type: fromCompositeFieldSet
              from: spec.storage.size
              to: spec.resources.requests.storage
            - type: fromCompositeFieldSet
              from: spec.storage.storageClass
              to: spec.storageClassName

Step 2.3 — Wire the connections between resources

Connections tell Crossplane how to link the DataVolume to the VirtualMachine. Add a connections section:

  connections:
    # Link DataVolume PVC to VirtualMachine volumes
    - from: datavolume
      to: virtualmachine
      patches:
        - type: toCompositeField
          from: status.atProvider.url
          to: spec.template.spec.volumes[0].persistentVolumeClaim.claimName
          transforms:
            - type: string
              string:
                fmt: '%s-dv'
    # Link PVC to VirtualMachine (fallback path)
    - from: pvc
      to: virtualmachine
      patches:
        - type: toCompositeField
          from: metadata.name
          to: spec.template.spec.volumes[0].persistentVolumeClaim.claimName
Connection resolution order

Crossplane resolves connections in sequence. The first matching patch wins. If a DataVolume is created, its PVC name flows to the VirtualMachine. If only a PVC exists (no DataVolume), the PVC name flows instead. This is why the fallback PVC resource exists.

Phase 3 — Deploy to the Cluster via ArgoCD

Now that you have the XRD and Composition files, you need to get them into the cluster. ArgoCD is the delivery mechanism.

Step 3.1 — Organize files in your Git repo

Structure your infrastructure repo so ArgoCD can find and sync the resources:

infra-repo/
├── argocd/
│   └── applications/
│       ├── crossplane-appofapps.yaml      # ArgoCD App-of-Apps bootstrap
│       └── crossplane-workloads.yaml      # ArgoCD app for Crossplane resources
├── crossplane/
│   ├── xrd/
│   │   └── virtualmachine.yaml            # XRD
│   ├── composition/
│   │   ├── virtualmachine-linux.yaml      # Composition (Linux)
│   │   └── virtualmachine-windows.yaml    # Composition (Windows)
│   ├── claim/
│   │   └── virtualmachine-sample.yaml     # Sample Claim (deployed separately)
│   └── providerconfig/
│       └── kubevirt.yaml                  # ProviderConfig for KubeVirt
└── README.md

Step 3.2 — Create the ArgoCD Application resource

Create argocd/applications/crossplane-workloads.yaml:

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: crossplane-infrastructure
  namespace: argocd
spec:
  project: crossplane
  source:
    repoURL: https://github.com/org/infra-repo.git
    targetRevision: main
    path: crossplane
  destination:
    server: https://kubernetes.default.svc
    namespace: crossplane-system
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true
      - PruneLast=true

Step 3.3 — Sync the XRD and Composition

# Sync via ArgoCD UI or CLI
argocd app sync crossplane-infrastructure

# Or apply directly for testing
kubectl apply -f crossplane/xrd/virtualmachine.yaml
kubectl apply -f crossplane/composition/virtualmachine-linux.yaml

Step 3.4 — Verify deployment

# Check the XRD is established
kubectl get xrvirtualmachines.infra.example.com
# Expected: xrvirtualmachines.infra.example.com   Established   True

# Check the Composition is ready
kubectl get composition xrvirtualmachine-linux
# Expected: xrvirtualmachine-linux   Ready   True

# Check the Composition Function pod is running
kubectl get pods -n crossplane-system -l app=crossplane-contrib-function-patch-and-transform
# Expected: 1/1 Running

# Verify the CRD was created by the XRD
kubectl api-versions | grep infra.example.com
# Expected: infra.example.com/v1alpha1
If the Composition shows Ready: False

Common causes:

  • The Composition Function pod is not running — check kubectl get pods -n crossplane-system
  • Schema mismatch — the XRD schema doesn't match the fields referenced in the Composition's patches
  • Missing label — the compositeResourceSelector.matchLabels don't match any XRs
  • Provider not installed — the provider-kubevirt CRDs (VirtualMachine, DataVolume) are not present

Phase 4 — Use the Claim to Provision a VM

Now the XRD and Composition are live. Workload teams can create Claims — instances of the XRD — and Crossplane will render and apply the Composition.

Step 4.1 — Create a Claim

Create crossplane/claim/virtualmachine-production.yaml:

apiVersion: infra.example.com/v1alpha1
kind: VirtualMachine
metadata:
  name: webserver-01
  labels:
    app: webserver
    env: production
spec:
  cpu: 4
  memory: "8Gi"
  guestOS: linux
  imageName: rhel-9.2-base
  storage:
    size: "50Gi"
    storageClass: ocs-storagecluster-ceph-rbd
  network:
    networkName: default
  userData: |
    #cloud-config
    chpasswd:
      list: |
        cloud-user:{{ .password }}
      expire: false
    users:
      - name: cloud-user
        ssh-authorized-keys:
          - ssh-rsa AAAA... user@host
    runcmd:
      - systemctl enable httpd
      - systemctl start httpd

Step 4.2 — Apply the Claim

# Apply the Claim (via Git or directly)
kubectl apply -f crossplane/claim/virtualmachine-production.yaml

# Watch the Claim progress
kubectl get virtualmachine webserver-01 -w

Step 4.3 — Trace the provisioning flow

Watch each stage as Crossplane processes the Claim:

# 1. Claim → XR (Crossplane creates a composite resource)
kubectl get xrvirtualmachine webserver-01

# 2. XR → Composition renders managed resources
kubectl get xr xrvirtualmachine.webserver-01 -o yaml | grep -A5 'resources:'

# 3. Check the DataVolume
kubectl get datavolume -n crossplane-system | grep webserver

# 4. Check the VirtualMachine
kubectl get virtualmachine -n crossplane-system | grep webserver

# 5. Check the PVC
kubectl get pvc -n crossplane-system | grep webserver

The typical timeline:

Stage What happens Typical duration
Claim → XR Crossplane controller creates the XR < 1 second
Composition render Function renders DataVolume + VM + PVC manifests 1–5 seconds
DataVolume import CDI pulls the base image from the registry 1–10 minutes (image size dependent)
VM creation KubeVirt creates the VM CR, virt-launcher pod spins up 10–60 seconds
Guest agent cloud-init runs, guest-agent connects 1–5 minutes

Step 4.4 — Verify the running VM

# Check the Claim's status
kubectl get virtualmachine webserver-01 -o yaml | grep -A10 status:

# Check the XR's status (mirrors the VM state)
kubectl get xrvirtualmachine webserver-01 -o yaml | grep -A10 status:

# Check the VirtualMachine phase
kubectl get virtualmachine webserver-01 -n crossplane-system -o jsonpath='{.status.phase}'

# Check the DataVolume phase
kubectl get datavolume webserver-01 -n crossplane-system -o jsonpath='{.status.phase}'

# SSH into the VM (if networking allows)
ssh cloud-user@<vm-ip>

Troubleshooting the Lifecycle

When something goes wrong, follow this diagnostic sequence:

Troubleshooting checklist

Symptom Check Fix
Claim shows Ready: False, no XR created kubectl describe claim <name> Check XRD is Established: True
XR created but no managed resources kubectl get xr <name> -o yaml Check Composition is Ready: True and labels match
DataVolume stuck in WaitForConsumer kubectl get dv <name> -n crossplane-system PVC consumer not linked — check Composition connections
VirtualMachine stuck in Stopped kubectl get vm <name> -n crossplane-system RunStrategy is Halted — edit VM to set runStrategy: Running
Composition Function pod crashing kubectl logs -n crossplane-system -l app=crossplane-contrib-function-patch-and-transform Check schema mismatch or missing fields in patches
cloud-init not applying kubectl get vm <name> -o yaml | grep guestCustomData userData must be base64-encoded; check Composition transform
ArgoCD shows sync failure argocd app diff crossplane-infrastructure Check YAML syntax and API version compatibility

Debugging tools

# Watch reconciliation events
kubectl get events -n crossplane-system --field-selector reason=Reconciling

# Check Crossplane controller logs
kubectl logs -n crossplane-system -l app=crossplane -f

# Trace a specific XR
kubectl get xr xrvirtualmachine.webserver-01 -o yaml

# Check managed resource conditions
kubectl get managed -n crossplane-system | grep webserver
kubectl describe managed <managed-resource-name> -n crossplane-system

# Check the Composition Function logs
kubectl logs -n crossplane-system -l app=crossplane-contrib-function-patch-and-transform -f

Exercise: Create a Windows VM XRD and Composition

You've walked through the Linux VM lifecycle. Now apply what you've learned:

  1. Create a new XRD version v1beta1 that adds a gpu field (boolean) and a dataDisks array
  2. Write a Composition xrvirtualmachine-windows that:
    • Sets runStrategy: Always instead of Halted
    • Adds a guestCustomData field with PowerShell/DSC bootstrap instead of cloud-init
    • Creates additional PVCs for data disks from the dataDisks array
  3. Deploy both to your cluster via ArgoCD
  4. Create a Claim for a Windows VM with 2 data disks and verify the provisioning flow
Click to reveal the answer

1. XRD v1beta1 with gpu and dataDisks

    versions:
      - name: v1alpha1
        served: true
        referenceable: true
        # ... (existing v1alpha1 schema)
      - name: v1beta1
        served: true
        referenceable: true
        schema:
          openAPIV3Schema:
            type: object
            properties:
              spec:
                type: object
                properties:
                  # ... (all v1alpha1 fields)
                  gpu:
                    type: boolean
                    description: Enable GPU passthrough
                  dataDisks:
                    type: array
                    items:
                      type: object
                      properties:
                        size:
                          type: string
                        storageClass:
                          type: string
                    description: Additional data disks

2. Windows Composition highlights

    # Set runStrategy
    - type: fromCompositeFieldSet
      from: spec.guestOS
      to: spec.runStrategy
      transforms:
        - type: map
          map:
            linux: Halted
            windows: Always

    # PowerShell guestCustomData
    - type: fromCompositeFieldSet
      from: spec.userData
      to: spec.template.spec.guestCustomData
      transforms:
        - type: string
          string:
            encode: Base64
            fmt: |
              <powershell>
              Set-ExecutionPolicy Bypass -Scope Process -Force
              [System.Text.Encoding]::UTF8.GetString([System.Convert]::FromBase64String('{{ .userData }}')) | Out-File C:\guest-config.ps1
              C:\guest-config.ps1
              </powershell>

3. Data disk PVCs in the Composition

    # Loop over dataDisks array (requires a Go/Lua function or manual expansion)
    # With patchAndTransform, you'd manually list each disk:
    - name: datadisk-0
      base:
        apiVersion: v1
        kind: PersistentVolumeClaim
        spec:
          accessModes: ["ReadWriteOnce"]
          resources: { requests: { storage: 100Gi } }
      patches:
        - type: fromCompositeFieldSet
          from: spec.dataDisks[0].size
          to: spec.resources.requests.storage
        - type: fromCompositeFieldSet
          from: spec.dataDisks[0].storageClass
          to: spec.storageClassName

    # Connect to VM volumes
    - from: datadisk-0
      to: virtualmachine
      patches:
        - type: toCompositeField
          from: metadata.name
          to: spec.template.spec.volumes[1].persistentVolumeClaim.claimName

Key insight

The patchAndTransform function doesn't support true loops over arrays. For variable-length lists like dataDisks, you have two options:

  1. Manual expansion — list each disk index explicitly in the Composition (simple but rigid)
  2. Custom Go/Lua function — iterate over the array and generate resources dynamically (flexible but more complex)

The XRD is the interface, the Composition is the implementation, ArgoCD is the delivery mechanism, and the Claim is the instance. Master this cycle and you can provision any infrastructure resource — not just VMs — through the same pattern.
