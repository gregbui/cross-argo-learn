Explainer 13: Storage for VMs — DataVolumes, PVC Lifecycle, and Storage Classes

VM storage is the most operationally fragile part of a Crossplane + KubeVirt stack. Unlike containers, where ephemeral storage is the default and persistent storage is the exception, VMs require persistent disks to function. A misconfigured storage class, a reclaim policy that wipes a disk on delete, or a DataVolume stuck in Pending can silently prevent VM provisioning — or worse, destroy data when a developer deletes a claim.

This explainer covers the storage layer end-to-end: how OpenShift exposes storage classes, how KubeVirt's DataVolume CRD manages VM disk images, how PVCs flow through their lifecycle, and how Crossplane compositions orchestrate all of it.

1. OpenShift Storage Classes for VMs

OpenShift ships with a set of storage profiles — one per provisioner type — that describe what storage is available on the cluster. These are exposed as StorageClass objects, and they are the first decision point when provisioning VM disks.

Common storage classes on OpenShift

StorageClass Provisioner Best for Performance
ocs-storagecluster-cephfs rook-ceph.cephfs.csi.ceph.com Shared storage, multi-WI Moderate
ocs-storagecluster-rbd rook-ceph.rbd.csi.ceph.com High-performance block High
local-storage local-storage Low-latency, node-local Highest
nfs-client cluster.local/nfs-client-provisioner File sharing, backups Low-Moderate
vsc-viper-5a-fast csi.vsc.emc.com EMC VMAX/vMAX V5A Very High
Key concept

StorageClass is the Kubernetes abstraction for storage provisioning. It tells the cluster: (1) which provisioner to use, (2) whether to allow volume expansion, (3) what the reclaim policy is, and (4) what parameters to pass to the provisioner. For VMs, the storage class directly determines I/O performance, snapshot capability, and cross-node migration support.

StorageProfile CRD

OpenShift also creates a StorageProfile CRD for each storage class. This is consumed by KubeVirt's DataVolume controller to determine:

  • Whether the storage class supports ReadWriteMany (needed for shared disk scenarios)
  • Whether the storage class supports ReadWriteOncePod (needed for certain snapshot workflows)
  • What clone strategy to use (VolumeSnapshot vs. direct copy)
  • The default dataSource for blank PVCs

You can inspect the storage profile for a given class:

oc get storageprofile ocs-storagecluster-rbd -o yaml

# Key fields:
# - storageClassParameters: raw string from StorageClass
# - volumeBindingMode: Immediate or WaitForFirstConsumer
# - reclaimPolicy: Retain or Delete
# - isDefault: true/false
# - cloneStrategy: volumeSnapshot or copy
Recommendation

For production VMs, prefer ocs-storagecluster-rbd (Ceph RBD) for general workloads and local-storage for latency-sensitive workloads (databases, domain controllers). Avoid NFS for primary VM disk storage — it works but adds significant overhead.

2. PVC Lifecycle Deep Dive

A PVC goes through a well-defined lifecycle. Understanding each phase is critical because Crossplane compositions often create PVCs directly, and failures at any phase can leave the XR stuck in Creating.

The 6 phases of a PVC

  1. Pending — The PVC exists in the API server but has not been bound to a PV yet. The storage provisioner is evaluating whether it can fulfill the request.
  2. Provisioning — The provisioner has accepted the request and is creating the underlying storage (Ceph RBD image, NFS directory, etc.).
  3. Bound — The PV has been created and bound to the PVC. The storage is ready to be used.
  4. InUse — A Pod or VirtualMachineInstance is actively using the PV. For KubeVirt, this is when a VirtualMachineInstance references the PVC.
  5. Released — The consumer has been deleted, but the PV has not yet been reclaimed. The data is still on disk.
  6. Available — The PV has been reclaimed and is ready for a new PVC binding.

Reclaim policies

The reclaimPolicy on a StorageClass determines what happens when a PVC is deleted:

Policy Behavior VM Use Case
Delete (default) PV and underlying storage are deleted Dev/test VMs, disposable workloads
Retain PV remains in Released state, underlying storage preserved Production VMs, VMs with persistent data
Recycle (deprecated) Runs rm -rf /the-data — not recommended None
Critical

Never use Delete reclaim policy for production VM storage. If a developer deletes a VirtualMachineClaim, a Delete reclaim policy will wipe the entire disk. Use Retain and implement explicit cleanup via a Composition's deletionPolicy or a separate lifecycle management layer.

Volume expansion

OpenShift supports allowVolumeExpansion: true on storage classes. When enabled, a PVC can be resized while it is bound and in use. KubeVirt propagates the resize to the underlying hypervisor.

# Expand a PVC in place
kubectl patch pvc vm-disk-my-vm -p '{"spec":{"resources":{"requests":{"storage":"50Gi"}}}}'

# Verify expansion
kubectl get pvc vm-disk-my-vm -o jsonpath='{.status.capacity.storage}'

In a Crossplane composition, you would update the PVC's spec.resources.requests.storage field and let the controller reconcile. The DataVolume controller handles the resize propagation to KubeVirt.

3. DataVolumes: The KubeVirt Storage Abstraction

A DataVolume (DV) is a KubeVirt CRD that manages the lifecycle of a VM's disk image. It is the bridge between raw PVCs and actual VM disk content. Without a DataVolume, you have a blank PVC — no OS, no image, nothing to boot from.

What a DataVolume does

  • Downloads an image from a URL (HTTP, HTTPS, OVA/OVF, CDI upload)
  • Clones from an existing PVC or VolumeSnapshot
  • Blanks a PVC with a minimal disk (for empty VMs)
  • Uploads images via the CDI upload proxy (for air-gapped environments)
  • Tracks progress via status.phase and status.progress

DataVolume phases

Phase Meaning
Empty Just created, no action yet
Pending Waiting for PVC to be provisioned
BucketClone Cloning from a Ceph object store bucket
Importing Downloading from a URL
Populating Running a population operation (clone, upload, etc.)
Ready Disk image is fully loaded and ready for VM use
Failed Something went wrong — check status.reason
Timeout Population exceeded spec.storage.size timeout

DataVolume types

# 1. HTTP download (most common for base images)
apiVersion: cdi.kubevirt.io/v1beta1
kind: DataVolume
metadata:
  name: dv-centos-stream-9
spec:
  source:
    http:
      url: "https://example.com/images/centos-stream-9.qcow2"
  storage:
    requests:
      storage: 20Gi
    storageClassName: ocs-storagecluster-rbd
  storageClass:
    - name: "cdi.kubevirt.io/storage.provisioner"
      value: "ocs-storagecluster-cephfs"

# 2. Clone from existing PVC
apiVersion: cdi.kubevirt.io/v1beta1
kind: DataVolume
metadata:
  name: dv-vm-clone
spec:
  source:
    pvc:
      name: base-centos-stream-9
      namespace: vm-import
  storage:
    requests:
      storage: 20Gi
    storageClassName: ocs-storagecluster-rbd

# 3. Clone from VolumeSnapshot
apiVersion: cdi.kubevirt.io/v1beta1
kind: DataVolume
metadata:
  name: dv-from-snapshot
spec:
  source:
    snapshot:
      name: vm-snapshot-2026-06-24
      namespace: vm-system
      kind: VolumeSnapshot
  storage:
    requests:
      storage: 20Gi
    storageClassName: ocs-storagecluster-rbd

# 4. Blank PVC (empty disk)
apiVersion: cdi.kubevirt.io/v1beta1
kind: DataVolume
metadata:
  name: dv-blank-disk
spec:
  storage:
    requests:
      storage: 100Gi
    storageClassName: ocs-storagecluster-rbd

# 5. CDI Upload (air-gapped / local image)
apiVersion: cdi.kubevirt.io/v1beta1
kind: DataVolume
metadata:
  name: dv-upload
spec:
  source:
    upload: {}
  storage:
    requests:
      storage: 50Gi
    storageClassName: ocs-storagecluster-rbd
Connection to Explainer 8

Recall from explainer-008 (OS Image Management) that we stored base images in an OCI registry and used a BaseImage CRD. The DataVolume is the runtime artifact that consumes those images. The composition function should translate a BaseImage reference into the appropriate DataVolume source — typically a PVC clone from a pre-populated base image PVC, not a fresh HTTP download.

CDI configuration

The Containerized Data Importer (CDI) is installed as part of OpenShift Virtualization. Its configuration lives in the cdi namespace and controls global behavior:

apiVersion: cdi.kubevirt.io/v1beta1
kind: CDI
metadata:
  name: cdi
spec:
  config:
    # Max concurrent imports/exports
    podResourceConstraints:
      high:
        cpu: "2"
        memory: "4Gi"
      medium:
        cpu: "1"
        memory: "2Gi"
      low:
        cpu: "500m"
        memory: "1Gi"
    # Timeout for large image imports
    dataImportCronTTL: 2h
    # HTTP proxy settings for air-gapped environments
    uploadProxyURLOverride: "https://cdi-upload-proxy.openshift-cnv.svc:443"
Resource constraints

CDI pods (importer, uploader, cloner) consume significant CPU and memory. In a busy cluster with many concurrent VM provisions, you may need to increase podResourceConstraints.high or scale the CDI deployment. Watch for DataVolume stuck in Pending — this is often CDI resource exhaustion, not a storage class problem.

4. Crossplane Composition: PVCs + DataVolumes

This is where Crossplane's composition function ties everything together. The composition must create both a DataVolume (to manage the disk image) and a PVC (to provide the storage claim that the VirtualMachine references). The order matters: the DataVolume populates the PVC, and the VirtualMachine can only reference the PVC once the DataVolume is Ready.

Composition function logic

package main

import (
    "context"
    "fmt"

    cdiv1beta1 "kubevirt.io/containerized-data-importer-api/pkg/apis/core/v1beta1"
    corev1 "k8s.io/api/core/v1"
    "sigs.k8s.io/controller-runtime/pkg/client"
    "sigs.k8s.io/controller-runtime/pkg/function"
)

// ComposeStorage creates the DataVolume and PVC for a VM.
// It returns the PVC name so the VirtualMachine can reference it.
func ComposeStorage(ctx context.Context, r *composition.Runtime, req *composition.Request) (*composition.Response, error) {
    xr := req.XR

    // Determine base image source from XR spec
    baseImageRef := xr.Spec.BaseImageRef
    diskSize := xr.Spec.DiskSize
    storageClass := xr.Spec.StorageClass

    // 1. Create or patch the DataVolume
    dv := &cdiv1beta1.DataVolume{
        ObjectMeta: metav1.ObjectMeta{
            Name:      fmt.Sprintf("dv-%s", xr.GetName()),
            Namespace: xr.GetNamespace(),
            OwnerReferences: []metav1.OwnerReference{
                *metav1.NewControllerRef(xr, xr.GroupVersionKind()),
            },
        },
        Spec: cdiv1beta1.DataVolumeSpec{
            Source: cdiv1beta1.DataVolumeSource{
                PVC: &cdiv1beta1.DataVolumeSourcePVC{
                    Name:      baseImageRef.PVCName,
                    Namespace: baseImageRef.PVCNamespace,
                },
            },
            Storage: &corev1.ResourceRequirements{
                Requests: corev1.ResourceList{
                    corev1.ResourceStorage: diskSize,
                },
            },
            StorageClassName: &storageClass,
        },
    }

    // 2. Create or patch the PVC (the DV controller will bind it)
    pvc := &corev1.PersistentVolumeClaim{
        ObjectMeta: metav1.ObjectMeta{
            Name:      fmt.Sprintf("vm-disk-%s", xr.GetName()),
            Namespace: xr.GetNamespace(),
            OwnerReferences: []metav1.OwnerReference{
                *metav1.NewControllerRef(xr, xr.GroupVersionKind()),
            },
        },
        Spec: corev1.PersistentVolumeClaimSpec{
            AccessModes: []corev1.PersistentVolumeAccessMode{
                corev1.ReadWriteOnce,
            },
            Resources: corev1.ResourceRequirements{
                Requests: corev1.ResourceList{
                    corev1.ResourceStorage: diskSize,
                },
            },
            StorageClassName: &storageClass,
            // After DV is Ready, the VM will use this PVC
            DataSource: &corev1.TypedLocalObjectReference{
                Name: fmt.Sprintf("dv-%s", xr.GetName()),
                Kind: "DataVolume",
            },
        },
    }

    return &composition.Response{
        Resources: map[string]*composition.Resource{
            "dataVolume": {
                Base:   dv,
                Managed: true,
            },
            "pvc": {
                Base:   pvc,
                Managed: true,
            },
        },
    }, nil
}

Using patchAndSync for DataVolume readiness

The composition function must wait for the DataVolume to reach Ready before the VirtualMachine can be created. Use the composition runtime's patchAndSync pattern to poll the DV status:

// After creating the DataVolume, patch the XR status to reflect DV state
dv := &cdiv1beta1.DataVolume{}
if err := r.Client.Get(ctx, client.ObjectKey{Name: dvName, Namespace: ns}, dv); err != nil {
    return nil, fmt.Errorf("get datavolume: %w", err)
}

// Set XR condition based on DV phase
switch dv.Status.Phase {
case cdiv1beta1.Ready:
    // DV is ready — XR can proceed to create the VirtualMachine
    xr.Status.AtProvider.DataVolumeReady = true
case cdiv1beta1.Failed, cdiv1beta1.Timeout:
    // Mark XR as errored
    xr.Status.AtProvider.ErrorMessage = fmt.Sprintf(
        "DataVolume failed: %s", dv.Status.Phase,
    )
default:
    // Still provisioning — XR stays in Creating
    xr.Status.AtProvider.DataVolumeProgress = dv.Status.Progress
}
Composition ordering

The composition function should create resources in this order: (1) DataVolume, (2) PVC (with dataSource pointing to the DV), (3) VirtualMachine. The PVC's dataSource field tells KubeVirt/CDI to wait for the DataVolume to complete before the VM tries to use the disk. This is the critical dependency chain.

Composition XRD schema for storage

Extend your XRD to include storage configuration:

spec:
  connectionSecretKeys:
    - sshPublicKey
  descriptions:
    name: VM storage configuration
  properties:
    baseImageRef:
      description: Reference to the base image to clone from
      properties:
        pvcName:
          type: string
        pvcNamespace:
          type: string
      type: object
    diskSize:
      description: Size of the VM disk
      type: string
      pattern: "^\\d+(Gi|Ti)$"
    storageClass:
      description: OpenShift storage class for VM disk
      type: string
      enum:
        - ocs-storagecluster-rbd
        - ocs-storagecluster-cephfs
        - local-storage
        - vsc-viper-5a-fast
    additionalDisks:
      description: Additional disks beyond the boot disk
      items:
        properties:
          size:
            type: string
          storageClass:
            type: string
          mountPath:
            type: string
        type: object
      type: array

5. Storage Class Selection in Compositions

Not all VMs need the same storage class. The composition should support selecting a storage class based on the VM's workload profile. This can be driven by the XRD schema or by labels on the VirtualMachineClaim.

Performance profiles

# Low-tier VMs (dev/test) — use shared CephFS
storageClass: ocs-storagecluster-cephfs
reclaimPolicy: Delete

# Standard VMs (production workloads) — use Ceph RBD
storageClass: ocs-storagecluster-rbd
reclaimPolicy: Retain

# High-performance VMs (databases) — use local storage
storageClass: local-storage
reclaimPolicy: Retain

# Enterprise VMs (domain controllers) — use enterprise SAN
storageClass: vsc-viper-5a-fast
reclaimPolicy: Retain

Composition function: auto-select storage class

func selectStorageClass(xr *crossplanev1.CompositeResource) string {
    // Explicit override takes precedence
    if xr.Spec.StorageClass != "" {
        return xr.Spec.StorageClass
    }

    // Auto-select based on labels
    labels := xr.GetLabels()
    switch {
    case labels["workload-type"] == "database":
        return "local-storage"
    case labels["workload-type"] == "enterprise":
        return "vsc-viper-5a-fast"
    case labels["tier"] == "production":
        return "ocs-storagecluster-rbd"
    default:
        return "ocs-storagecluster-cephfs"
    }
}
Best practice

Let the claim author override the storage class via the XRD schema, but provide sensible defaults in the composition function. This gives teams flexibility while preventing accidental use of the wrong storage class for production workloads.

6. Multi-DC Storage Considerations

In your 3-DC topology (from docs/topology-multi-dc-ove-clusters.md), storage adds significant complexity:

Per-cluster storage independence

Each OVE cluster has its own storage infrastructure (Ceph cluster, NFS server, or SAN). A DataVolume created on cluster A cannot be consumed by a VM on cluster B. This means:

  • Base images must exist on every cluster. You need a base image pipeline that pushes images to each cluster's CDI upload proxy, or a shared image registry that each cluster pulls from.
  • Storage classes differ per cluster. Cluster A might have ocs-storagecluster-rbd while cluster B has vsc-viper-5a-fast. The composition function must be cluster-aware.
  • Storage class names may differ. Don't hardcode storage class names — use a label or annotation on the cluster to select the appropriate class.

Image distribution strategy

# Strategy 1: Shared OCI registry (recommended)
# Each cluster pulls base images from the same registry
# The composition function references the image by digest
# CDI clones from a local PVC that was pre-populated

# Strategy 2: CDI upload to each cluster
# Push the base image to each cluster's CDI upload proxy
# Requires authenticated access to each cluster's upload endpoint

# Strategy 3: DataImportCron (CDI scheduled imports)
# Schedule periodic imports of base images to each cluster
# Automates the "keep base images up to date" problem

DataImportCron

CDI's DataImportCron CRD automates periodic image imports. This is the recommended approach for keeping base images fresh across multiple clusters:

apiVersion: cdi.kubevirt.io/v1beta1
kind: DataImportCron
metadata:
  name: dicron-centos-stream-9
  namespace: vm-system
spec:
  source:
    http:
      url: "https://example.com/images/centos-stream-9-latest.qcow2"
  schedule: "0 2 1 * *"  # 2 AM on the 1st of every month
  storageClass: ocs-storagecluster-rbd
  storage:
    requests:
      storage: 20Gi
  retentionPolicy: KeepLast 5
  template:
    spec:
      storage:
        requests:
          storage: 20Gi
        storageClassName: ocs-storagecluster-rbd

When the cron fires, CDI downloads the image and stores it in a PVC. The composition function then clones from this pre-populated PVC instead of downloading on-demand, which is much faster and more reliable.

Multi-DC caveat

DataImportCron runs per cluster. You need one DataImportCron per base image per cluster. In a 3-DC topology with 9 clusters, that's 27 DataImportCrons for a single base image. Automate this with a composition that creates DataImportCrons on each target cluster, or use a central management cluster that pushes images to all clusters.

7. Operational Concerns

Monitoring DataVolumes

Watch for these failure modes:

Problem Symptom Fix
CDI resource exhaustion DV stuck in Pending Scale CDI deployment, increase podResourceConstraints
Network timeout during import DV in Failed with timeout error Increase spec.timeout, check network to image source
Storage class not found DV in Failed with "storageclass not found" Verify storage class exists on target cluster
Insufficient storage capacity PVC stuck in Pending, no PV provisioned Check storage capacity, expand storage cluster
Image format unsupported DV in Failed with "unsupported format" Convert image to qcow2 or raw format

Storage capacity planning

# Check CDI pod resource usage
oc top pods -n cdi

# Check PVC usage across namespaces
oc get pvc -A --sort-by=.spec.resources.requests.storage

# Check storage cluster capacity (Ceph)
ceph df

# Check CDI upload proxy health
oc get pods -n cdi | grep upload

# Monitor DataVolume phases
oc get datavolume -A --sort-by=.status.phase

Snapshot and backup workflows

For production VMs, implement a snapshot-and-restore workflow:

# 1. Create a VolumeSnapshot of the VM's disk
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshot
metadata:
  name: vm-snapshot-backup
  namespace: vm-system
spec:
  volumeSnapshotClassName: csi-rbdplugin-snapclass
  source:
    persistentVolumeClaimName: vm-disk-my-vm

# 2. Use the snapshot as a DataVolume source for cloning
apiVersion: cdi.kubevirt.io/v1beta1
kind: DataVolume
metadata:
  name: dv-from-snapshot
spec:
  source:
    snapshot:
      name: vm-snapshot-backup
      namespace: vm-system
      kind: VolumeSnapshot
  storage:
    requests:
      storage: 20Gi
    storageClassName: ocs-storagecluster-rbd

# 3. Or use KubeVirt VirtualMachineSnapshot (full VM snapshot)
apiVersion: kubevirt.io/v1
kind: VirtualMachineSnapshot
metadata:
  name: full-vm-snapshot
  namespace: vm-system
spec:
  source:
    apiGroup: kubevirt.io
    kind: VirtualMachine
    name: my-vm
Integration point

This connects to the "Disaster recovery" topic from the roadmap. A Crossplane composition could manage a VirtualMachineSnapshot CRD that triggers periodic snapshots, and a separate composition could restore from a snapshot when a VM is corrupted. The VirtualMachineSnapshot CRD captures the entire VM state — disk, memory (optional), and configuration — making it more comprehensive than a bare VolumeSnapshot.

Reclaim policy enforcement

Enforce Retain reclaim policy for production storage classes. Add a validation webhook or an OPA/Gatekeeper policy:

# OPA/Gatekeeper constraint
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sStorageClass
metadata:
  name: require-retain-policy
spec:
  match:
    kinds:
      - apiGroups: ["storage.k8s.io"]
        kinds: ["StorageClass"]
  parameters:
    reclaimPolicy: Retain

In a Crossplane composition, you can also enforce this by explicitly setting the reclaim policy in the composed PV:

# Composition function: ensure reclaim policy
pv := &corev1.PersistentVolume{
    Spec: corev1.PersistentVolumeSpec{
        PersistentVolumeReclaimPolicy: corev1.PersistentVolumeRetain,
        // ...
    },
}

8. Summary

Storage for VMs in a Crossplane + KubeVirt stack is a multi-layered problem:

  • StorageClass defines the backing storage (Ceph RBD, NFS, local, enterprise SAN) and its behavior (reclaim policy, expansion support, volume binding mode)
  • PVC is the storage claim that the VM references — it goes through a 6-phase lifecycle from Pending to Bound to Released
  • DataVolume is the KubeVirt abstraction that populates the PVC with a disk image (from a URL, a clone, a snapshot, or an upload)
  • CDI is the runtime engine that executes DataVolume operations — it consumes CPU/memory and can become a bottleneck
  • Crossplane composition orchestrates the dependency chain: DataVolume → PVC → VirtualMachine, with the composition function polling DV readiness
  • DataImportCron automates base image updates per cluster, critical for multi-DC deployments

The most common failure modes are CDI resource exhaustion (stuck DataVolumes), wrong reclaim policies (accidental data loss), and storage class mismatches across clusters (multi-DC). All three are preventable with proper composition design and operational monitoring.

Exercise 1: Storage class selection

Question: Your team wants to provision a PostgreSQL VM and a development test VM. Which storage classes would you use for each, and why?

Answer: PostgreSQL → local-storage or vsc-viper-5a-fast (low latency, high IOPS for database workloads). Development test VM → ocs-storagecluster-cephfs (shared storage, lower cost, Delete reclaim policy is acceptable for disposable workloads).

Exercise 2: DataVolume lifecycle

Question: A DataVolume is stuck in Pending for 30 minutes. What are the top 3 things to check?

Answer: (1) CDI pod resource exhaustion — check oc top pods -n cdi and CDI deployment replicas. (2) Storage class not found — verify the storage class exists on the target cluster. (3) PVC provisioning failure — check if the underlying storage provisioner (Ceph, NFS, etc.) has capacity and is healthy.

Exercise 3: Reclaim policy enforcement

Question: A developer accidentally deleted a VirtualMachineClaim for a production VM. The storage class had Delete reclaim policy. What happened, and how do you prevent this?

Answer: The PV and underlying storage were deleted — all data is gone. To prevent: (1) Set Retain reclaim policy on production storage classes. (2) Implement a composition-level deletionPolicy: Orphan that detaches the PVC from the XR before the XR is deleted. (3) Add OPA/Gatekeeper constraints that reject Delete reclaim policy on production storage classes.
