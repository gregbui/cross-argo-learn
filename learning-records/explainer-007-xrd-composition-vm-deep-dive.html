1. The XRD-to-Composition Relationship

Before diving into schemas and patches, let's be precise about how XRDs and Compositions relate. This is the architecture that everything else builds on.

┌─────────────┐ ┌────────────────┐ ┌─────────────────┐ │ User / CI │────▶│ Claim │────▶│ XR │ │ Git commit │ │ VirtualMachine │ │ XVirtualMachine │ │ │ │ Claim │ │ (auto-created) │ └─────────────┘ └────────────────┘ └──────┬──────────┘ │ │ Composition │ crossplane.io/xrd label match ▼ ┌──────────────────┐ │ Composition │ │ vm-kubevirt │ │ │ │ Renders: │ │ • VirtualMachine │ │ • DataVolume │ │ • PVC │ └────────┬─────────┘ │ │ Provider ▼ ┌──────────────┐ │ KubeVirt │ │ Provider │ │ (controller) │ └──────┬───────┘ │ ▼ ┌──────────────┐ │ OpenShift │ │ (libvirt) │ └──────────────┘

The binding mechanism

A Composition is linked to an XRD by the crossplane.io/xrd label on the Composition's metadata:

metadata:
  name: vm-kubevirt
  labels:
    crossplane.io/xrd: xvirtualmachines.infra.example.com

Crossplane's composition-controller watches for XRD changes. When it sees an XRD, it scans all Compositions for matching labels. A Composition can also use compositeResourceSelector to match XRs by label instead:

spec:
  compositeResourceSelector:
    matchLabels:
      infra.example.com/kubevirt: "true"
Design choice
Label-based (recommended): One XRD → many Compositions (e.g., vm-kubevirt, vm-vmware). Flexible but requires the XR to carry the right label.
Direct reference: XRD's spec.defaultCompositionName pins one Composition. Simpler but no fallback.

Versioning

An XRD can declare multiple versions. Each version can have its own Composition:

spec:
  versions:
  - name: v1alpha1
    served: true
    referenceable: true
  - name: v1beta1
    served: true
    referenceable: true

When a Claim is created, Crossplane creates an XR in the referenceable version. If only v1alpha1 is referenceable, all Claims produce XVirtualMachine/v1alpha1 XRs regardless of the Claim's API version.

2. XRD Schema Design for VM Workloads

The XRD schema is the contract between the infrastructure team (who defines it) and the application teams (who fill it in). Good schema design prevents misconfiguration before it happens.

2.1 Required vs. Optional fields

Use required arrays to enforce minimums. For VMs, compute and storage are almost always required:

spec:
  properties:
    compute:
      type: object
      required:
      - vcpus
      - memory
      properties:
        vcpus:
          type: integer
          minimum: 1
          maximum: 64
        memory:
          type: string
          pattern: "^[0-9]+(Mi|Gi|Ti)$"
    storage:
      type: object
      required:
      - size
      - image
      properties:
        size:
          type: string
          pattern: "^[0-9]+(Mi|Gi|Ti)$"
        image:
          type: string
          description: "DataVolume source: http URL, registry path, or PVC name"
OpenAPI v3 validation
Crossplane uses OpenAPI v3 schema for validation. Use minimum/maximum for numbers, pattern for strings, enum for fixed sets, and minItems/maxItems for arrays. These validate at apply time — before Crossplane even starts reconciling.

2.2 Default values via Composition

XRDs cannot declare default values. Instead, the Composition provides them via patches. This is a common source of confusion — the XRD says "this field is optional" and the Composition fills in sensible defaults:

# In the Composition's function config:
patches:
- fromFieldPath: "status.availabilityZone"
  toFieldPath: "spec.template.spec.domain.devices.disks[0].disk"
  policy:
    fromFieldPath: optional  # ← If XR doesn't set it, skip the patch

For fields that always need a default, the Composition's base resource can set them directly — they only get overridden if the XR provides a value.

2.3 Composite types (grouping related fields)

For complex VM specs, group related fields into composite types. This makes the schema self-documenting:

properties:
  compute:
    type: object
    properties:
      vcpus: { type: integer }
      memory: { type: string }
      gpu:
        type: object
        properties:
          model: { type: string }
          devices: { type: integer }
  storage:
    type: object
    properties:
      size: { type: string }
      storageClass: { type: string }
      image: { type: string }
      additionalVolumes:
        type: array
        items:
          type: object
          properties:
            name: { type: string }
            size: { type: string }
            storageClass: { type: string }
  network:
    type: object
    properties:
      macAddress: { type: string }
      interfaces:
        type: array
        items:
          type: object
          properties:
            model: { type: string }
            bridge: { type: object }
            masquerade: { type: object }
  lifecycle:
    type: object
    properties:
      running:
        type: boolean
        default: true
      evictionStrategy:
        type: string
        enum: [LiveMigrate, External, Halt, Migrate]
      shutdown:
        type: object
        properties:
          signal:
            type: string
            enum: [Ssh, Acpi]

2.4 Status shape

The XRD status declares what Crossplane reports back to the user. For VMs, the most useful status fields are:

status:
  type: object
  properties:
    conditions:
      type: array
      items:
        type: object
        properties:
          type: { type: string }
          status: { type: string }
          reason: { type: string }
          lastTransitionTime: { type: string }
    endpoint:
      type: string
      description: "VM IP address or hostname"
    ready:
      type: boolean
      description: "Whether all managed resources are Ready"

Crossplane auto-manages the Ready condition based on whether all managed resources report Ready. You can add additional conditions via the Composition.

2.5 Referenceable versions

Setting referenceable: true on a version allows other resources to reference this XR's fields via connections. For VM provisioning chains (e.g., a "VM cluster" XRD that references individual VM XRs), this is essential:

versions:
- name: v1alpha1
  served: true
  referenceable: true   # ← Allows other XRDs to reference this XR's spec/status

3. Composition Patterns for VMs

A Composition for VMs typically manages three resource types: VirtualMachine, DataVolume, and optionally PersistentVolumeClaim. The pattern for wiring these together is the core skill.

3.1 The three managed resource types

DataVolume

apiVersion: cdi.kubevirt.io/v1beta1

Handles disk image lifecycle: download from URL, import from registry, or clone from existing PVC. Creates the backing PVC that the VM boots from.

Key fields: spec.source, spec.pvc, spec.storage

VirtualMachine

apiVersion: kubevirt.io/v1

Declares VM compute, storage, and network config. KubeVirt's controller translates this into a running VM via libvirt.

Key fields: spec.running, spec.template.spec.domain, spec.template.spec.volumes

PersistentVolumeClaim

apiVersion: v1

Optional — the DataVolume usually creates its own PVC. Use explicit PVCs for additional disks beyond the boot disk.

Key fields: spec.accessModes, spec.resources.requests.storage, spec.storageClassName

ProviderConfig

apiVersion: kubevirt.crossplane.io/v1alpha1

Not a managed resource — it's the credential/config that tells the KubeVirt provider how to reach the OpenShift API.

Key fields: spec.credentials, spec.serviceAccountName

3.2 Declarative vs. Pipeline mode

For VM workloads, both modes work. The choice depends on complexity:

CriteriaDeclarativePipeline
Best for Simple VMs (fixed image, single disk, standard networking) Complex VMs (conditional configs, cloud-init, multiple disks)
Number of managed resources Up to ~10 Unlimited (loop via Go-Templating)
Conditional logic Limited (policy-based, enum transforms) Full (Go templates: if/else, ranges, Sprig functions)
DataVolume injection Connection from DV name to VM volume Template renders both DV and VM, patches link them
Learning curve Lower — pure YAML patches Higher — need to understand function pipeline
Maintenance Changes require Composition re-sync Functions can be updated independently
Recommendation for this stack
Start with declarative mode for simple VMs. Move to pipeline mode when you need cloud-init injection, conditional eviction strategies, or dynamic disk provisioning. Most production VM stacks end up using both — declarative for standard workloads, pipeline for custom ones.

3.3 The composition rendering flow

Regardless of mode, the rendering flow is the same:

1. Crossplane receives an XR (or sees a Claim → creates XR)
2. Finds the matching Composition (by label or selector)
3. Renders the Composition:
   a. For each resource in the Composition, create a Managed Resource template
   b. Apply patches: map XR spec fields → managed resource spec fields
   c. Apply connections: pass outputs from one managed resource → another
   d. Apply function pipeline steps (if pipeline mode)
4. Apply all rendered managed resources to the cluster
5. The KubeVirt provider reconciles each managed resource against the KubeVirt API
6. Status flows back: KubeVirt → Provider → Composition → XR → Claim

4. Declarative Composition: Full VM + DataVolume Example

This is the simplest working Composition. It creates a DataVolume (to download an OS image) and a VirtualMachine (to run it), wired together via a connection.

4.1 The Composition

apiVersion: apiextensions.crossplane.io/v1
kind: Composition
metadata:
  name: vm-kubevirt-declarative
  labels:
    crossplane.io/xrd: xvirtualmachines.infra.example.com
    infra.example.com/composition-style: declarative
spec:
  compositeResourceSelector:
    matchLabels:
      infra.example.com/kubevirt: "true"
  resources:
  # ── Resource 1: DataVolume (disk image) ──
  - name: datavolume
    base:
      apiVersion: cdi.kubevirt.io/v1beta1
      kind: DataVolume
      metadata:
        labels:
          infra.example.com/managed-by: crossplane
      spec:
        storage: {}  # Will be patched from XR spec
        source: {}   # Will be patched from XR spec
    patches:
    # Map XR storage.size → DV storage request
    - fromFieldPath: "storage.size"
      toFieldPath: "spec.storage.resources.requests.storage"
    # Map XR storage.image → DV source URL
    - fromFieldPath: "storage.image"
      toFieldPath: "spec.source.http.url"
      transforms:
      - type: string
        string:
          fmt: "https://example.com/images/%s"
          # Only if image doesn't already start with http/https
    # Map XR storage.storageClass → DV storage class
    - fromFieldPath: "storage.storageClass"
      toFieldPath: "spec.storage.storageClassName"
    # Set the DV name to include the XR name for traceability
    - fromFieldPath: "metadata.name"
      toFieldPath: "metadata.generateName"
      transforms:
      - type: string
        string:
          fmt: "vm-%s-"

  # ── Resource 2: VirtualMachine ──
  - name: virtualmachine
    base:
      apiVersion: kubevirt.io/v1
      kind: VirtualMachine
      metadata:
        labels:
          infra.example.com/managed-by: crossplane
      spec:
        running: true
        template:
          spec:
            domain:
              devices:
                disks:
                - name: disk0
                  disk:
                    bus: virtio
              evictionStrategy: LiveMigrate
            networks:
            - name: default-network
              pod: {}
            terminationGracePeriodSeconds: 30
            volumes:
            - name: disk0
              dataVolume:
                name: placeholder-dv
    patches:
    # Map XR compute.vcpus → VM vCPUs
    - fromFieldPath: "compute.vcpus"
      toFieldPath: "spec.template.spec.domain.cpu.cores"
    # Map XR compute.memory → VM memory
    - fromFieldPath: "compute.memory"
      toFieldPath: "spec.template.spec.domain.resources.requests.memory"
    # Conditional: if XR sets evictionStrategy, use it
    - fromFieldPath: "lifecycle.evictionStrategy"
      toFieldPath: "spec.template.spec.domain.evictionStrategy"
      policy:
        fromFieldPath: optional
    # Connection: pass DV name into VM volume reference
    - fromFieldPath: "metadata.name"
      toFieldPath: "spec.template.spec.volumes[0].dataVolume.name"
      connections:
      - fromResource: datavolume
        toFieldPath: metadata.name
        transforms:
        - type: string
          string:
            fmt: "vm-%s-"
    # Connection: set running from XR
    - fromFieldPath: "lifecycle.running"
      toFieldPath: "spec.running"
      policy:
        fromFieldPath: optional
    # Connection: set VM metadata name to XR name
    - fromFieldPath: "metadata.name"
      toFieldPath: "metadata.generateName"
      transforms:
      - type: string
        string:
          fmt: "vm-%s-"

4.2 How the connection works

The tricky part of VM compositions is wiring the DataVolume name into the VirtualMachine's volume reference. Here's the flow:

XR metadata.name = "my-web-server"
         │
         ▼
DataVolume metadata.generateName = "vm-my-web-server-"
         │
         │ Crossplane resolves this to "vm-my-web-server-abc12"
         │
         ▼
VirtualMachine spec.template.spec.volumes[0].dataVolume.name
    = "vm-my-web-server-abc12"   ← resolved from DV's actual name
Timing caveat
Crossplane renders all managed resources in a single pass. The DataVolume and VirtualMachine are created simultaneously. The DV will be in Pending state while it downloads the image, and the VM will be created but not yet running until the DV becomes Succeeded. KubeVirt handles this — it waits for the backing PVC before booting.

4.3 The resulting Claim

With the XRD and Composition in place, a user creates a simple Claim:

apiVersion: infra.example.com/v1alpha1
kind: VirtualMachineClaim
metadata:
  name: web-server-1
  namespace: default
  labels:
    infra.example.com/kubevirt: "true"
spec:
  compute:
    vcpus: 2
    memory: "4Gi"
  storage:
    size: "50Gi"
    storageClass: ocs.external.ceph.rook.io
    image: "centos-stream-9.qcow2"
  network: {}
  lifecycle:
    running: true
    evictionStrategy: LiveMigrate

Crossplane creates an XVirtualMachine/web-server-1 XR, the Composition renders a DataVolume and VirtualMachine, and KubeVirt provisions the VM. Done in ~3 minutes (mostly image download time).

5. Pipeline Composition: Full VM + DataVolume Example

Pipeline mode gives you imperative logic. This is where you'd go for cloud-init injection, conditional resource creation, or dynamic disk provisioning.

5.1 Pipeline structure

apiVersion: apiextensions.crossplane.io/v1
kind: Composition
metadata:
  name: vm-kubevirt-pipeline
  labels:
    crossplane.io/xrd: xvirtualmachines.infra.example.com
    infra.example.com/composition-style: pipeline
spec:
  compositeResourceSelector:
    matchLabels:
      infra.example.com/kubevirt: "true"
  mode: Pipeline
  pipeline:
  # ── Step 1: Render base VM template ──
  - step: render-vm-template
    functionRef:
      name: crossplane-function-go-templating
    inputs:
      resources:
      - name: vm
        base:
          apiVersion: kubevirt.io/v1
          kind: VirtualMachine
          metadata:
            labels:
              infra.example.com/managed-by: crossplane
          spec:
            running: true
            template:
              spec:
                domain:
                  cpu:
                    cores: 1
                  devices:
                    disks:
                    - name: bootdisk
                      disk:
                        bus: virtio
                    # Additional disks injected by later steps
                  evictionStrategy: LiveMigrate
                networks:
                - name: default-network
                  pod: {}
                terminationGracePeriodSeconds: 30
                volumes:
                - name: bootdisk
                  dataVolume:
                    name: ""
                # Guest agent and cloud-init injected here
        desired:
          apiVersion: crossplane.io/v1alpha1
          kind: ResourceList
          resources:
          - name: vm
            ref:
              name: vm
            path: /
      - name: providerconfig
        base:
          apiVersion: kubevirt.crossplane.io/v1alpha1
          kind: ProviderConfig
          spec:
            credentialSource:
              serviceAccount: crossplane-kubevirt-provider

  # ── Step 2: Patch VM fields from XR spec ──
  - step: patch-vm-spec
    functionRef:
      name: crossplane-function-patch-and-transform
    inputs:
      functionConfigRef:
        name: vm-patch-config
      resources:
      - name: vm
        patches:
        - fromFieldPath: "compute.vcpus"
          toFieldPath: "spec.template.spec.domain.cpu.cores"
        - fromFieldPath: "compute.memory"
          toFieldPath: "spec.template.spec.domain.resources.requests.memory"
        - fromFieldPath: "lifecycle.evictionStrategy"
          toFieldPath: "spec.template.spec.domain.evictionStrategy"
          policy:
            fromFieldPath: optional

  # ── Step 3: Render DataVolume ──
  - step: render-datavolume
    functionRef:
      name: crossplane-function-go-templating
    inputs:
      resources:
      - name: dv
        base:
          apiVersion: cdi.kubevirt.io/v1beta1
          kind: DataVolume
          metadata:
            labels:
              infra.example.com/managed-by: crossplane
          spec:
            storage: {}
            source: {}
        desired:
          apiVersion: crossplane.io/v1alpha1
          kind: ResourceList
          resources:
          - name: dv
            ref:
              name: dv
            path: /
      - name: vm
        base:
          apiVersion: kubevirt.io/v1
          kind: VirtualMachine

  # ── Step 4: Patch DataVolume from XR spec ──
  - step: patch-dv-spec
    functionRef:
      name: crossplane-function-patch-and-transform
    inputs:
      functionConfigRef:
        name: dv-patch-config
      resources:
      - name: dv
        patches:
        - fromFieldPath: "storage.size"
          toFieldPath: "spec.storage.resources.requests.storage"
        - fromFieldPath: "storage.storageClass"
          toFieldPath: "spec.storage.storageClassName"
        - fromFieldPath: "storage.image"
          toFieldPath: "spec.source.http.url"

  # ── Step 5: Link DV to VM ──
  - step: link-dv-to-vm
    functionRef:
      name: crossplane-function-patch-and-transform
    inputs:
      functionConfigRef:
        name: vm-dv-link-config
      resources:
      - name: vm
        patches:
        - fromFieldPath: "status.atProvider.name"
          toFieldPath: "spec.template.spec.volumes[0].dataVolume.name"
          connections:
          - fromPath: status.atProvider.name
            fromResource: dv

  # ── Step 6: Apply cloud-init if specified ──
  - step: inject-cloudinit
    functionRef:
      name: crossplane-function-go-templating
    inputs:
      resources:
      - name: vm
        base:
          apiVersion: kubevirt.io/v1
          kind: VirtualMachine
        desired:
          apiVersion: crossplane.io/v1alpha1
          kind: ResourceList
          resources:
          - name: vm
            ref:
              name: vm
            path: /spec/template/spec/volumes
          - name: cloudinit-dv
            ref:
              name: cloudinit-dv
            path: /
            remove: true  # Only keep if cloud-init is defined

5.2 Patch-and-Transform function configs

Each Patch-and-Transform step references a FunctionConfig that holds the patch definitions:

apiVersion: v1
kind: ConfigMap
metadata:
  name: vm-patch-config
data:
  function.crossplane.io/resource-patches: |
    - fromFieldPath: "compute.vcpus"
      toFieldPath: "spec.template.spec.domain.cpu.cores"
    - fromFieldPath: "compute.memory"
      toFieldPath: "spec.template.spec.domain.resources.requests.memory"
Why separate FunctionConfigs?
Storing patches in ConfigMaps means you can update patch logic without touching the Composition. ArgoCD can sync FunctionConfigs independently, enabling gradual rollout of patch changes.

5.3 Go-Templating for cloud-init

The most powerful VM-specific use case for Go-Templating is injecting cloud-init user-data:

apiVersion: v1
kind: ConfigMap
metadata:
  name: cloudinit-function-config
data:
  function.crossplane.io/resources: |
    - name: vm
      base:
        apiVersion: kubevirt.io/v1
        kind: VirtualMachine
      desired:
        apiVersion: crossplane.io/v1alpha1
        kind: ResourceList
        resources:
        - ref:
            name: vm
          path: /
        - name: cloudinit-dv
          ref:
            name: cloudinit-dv
          path: /
          remove: true
---
  # In the Go template:
  {{- if .resources.XVirtualMachine.claim.spec.cloudInit }}
  {{- $userData := .resources.XVirtualMachine.claim.spec.cloudInit.userData }}
  {{- $dvName := printf "%s-cloudinit" (include "crossplane.metadata.name" .resources.XVirtualMachine.claim) }}
  - name: cloudinit-dv
    apiVersion: cdi.kubevirt.io/v1beta1
    kind: DataVolume
    metadata:
      name: {{ $dvName }}
    spec:
      pvc:
        name: {{ $dvName }}
        accessModes: ["ReadWriteOnce"]
        resources:
          requests:
            storage: 100Mi
      source:
        http:
          url: data:text/plain;base64,{{ $userData | b64enc }}
  - ref:
      name: vm
    patches:
    - path: spec.template.spec.volumes
      valueFrom:
        insertAfter:
          name: bootdisk
          dataVolume:
            name: {{ $dvName }}
    - path: spec.template.spec.domain.devices.disks
      valueFrom:
        insertAfter:
          name: bootdisk
          disk:
            bus: virtio
          name: cloudinitdisk
          disk:
            cdrom:
              bus: sata
          name: cloudinitdisk

6. Connection Patterns: Wiring DataVolume to VirtualMachine

Connections are how Crossplane passes data between managed resources. For VMs, the most critical connection is wiring the DataVolume's name (and later its PVC) into the VirtualMachine's volume specification.

6.1 Declarative connections

In declarative mode, connections are declared inline on the patch:

- fromFieldPath: "spec.template.spec.volumes[0].dataVolume.name"
  connections:
  - fromPath: metadata.name
    fromResource: datavolume

This tells Crossplane: "After the DataVolume is created, read its metadata.name and write it into the VM's volume reference."

6.2 Pipeline connections

In pipeline mode, connections are declared in the FunctionConfig:

patches:
- fromFieldPath: "spec.template.spec.volumes[0].dataVolume.name"
  connections:
  - fromPath: status.atProvider.name
    fromResource: dv

Using status.atProvider.name (the provider's reported name) is more reliable than metadata.name because it reflects the actual reconciled state.

6.3 Connection patterns reference

PatternFromToWhen to use
DV → VM name DV metadata.name VM volumes[].dataVolume.name Always — VM needs to know which DV to boot from
VM → XR status VM status.printableStatus XR status.endpoint Report VM running/stopped state back to user
ProviderConfig ref XR spec.providerConfigName Managed resource spec.providerConfigRef.name Multi-tenant setups with different auth per namespace
XR → all resources XR metadata.name All managed resource metadata.labels Traceability — label all resources with XR name

6.4 The connection resolution order

1. Crossplane renders all managed resource templates (base + patches)
2. For each connection:
   a. Read the source field from the source managed resource
   b. If the source field is not yet available (resource not Ready),
      defer the connection until the next reconciliation
   c. Write the value to the target field on the target managed resource
3. Apply all managed resources to the cluster
4. On next reconciliation:
   a. Read actual status from the KubeVirt provider
   b. Resolve any deferred connections
   c. Re-apply managed resources if connection values changed
Deferred connections
If a connection source field isn't available yet (e.g., the DV hasn't been created), Crossplane defers the connection and retries on the next reconciliation cycle (default ~1 minute). This is why VM provisioning takes a few minutes — the DV must become Ready before the VM can reference it.

7. ProviderConfig and Multi-Tenancy

ProviderConfig is how the KubeVirt provider knows how to talk to OpenShift. Its placement and scoping have important design implications.

7.1 ProviderConfig structure

apiVersion: kubevirt.crossplane.io/v1alpha1
kind: ProviderConfig
metadata:
  name: default
spec:
  credentials:
    source: Secret
    secretRef:
      name: kubevirt-provider-credentials
      namespace: crossplane-system
      key: credentials
  # OR for in-cluster auth:
  serviceAccountName: crossplane-kubevirt-provider

7.2 Scoping strategies

StrategyScopeUse case
Single ProviderConfig Cluster-wide (name: "default") Single team, single OpenShift cluster
Namespace-scoped One per namespace (dev, staging, prod) Separate auth per environment
XR-referenced Each XR specifies spec.providerConfigRef.name Mixing providers (KubeVirt + VMware on same cluster)

7.3 In-cluster vs. external auth

# In-cluster (recommended for OpenShift):
# The provider uses the Crossplane pod's ServiceAccount
# to authenticate to the OpenShift API server.
# No secrets needed — OpenShift RBAC handles authorization.

# External (for remote clusters):
# ProviderConfig holds a kubeconfig or token.
# Useful if Crossplane runs on a different cluster
# than the one running KubeVirt.
OpenShift recommendation
Use in-cluster auth with a ServiceAccount. The KubeVirt provider pod runs in crossplane-system and uses its SA to authenticate to OpenShift. Grant the SA permissions via a ClusterRole/ClusterRoleBinding that allows get/list/create/update/patch on virtualmachines, datavolumes, and persistentvolumeclaims in the target namespaces.

8. Status Reporting and Conditions

Status reporting is how Crossplane tells the user what's happening. For VM workloads, meaningful status is critical for debugging and monitoring.

8.1 Auto-managed conditions

Crossplane automatically manages the Ready condition based on managed resource status:

status:
  conditions:
  - type: Ready
    status: True    # All managed resources are Ready
    reason: CompositeReady
  - type: Synced
    status: True
    reason: Synced

8.2 Custom conditions from the Composition

You can add custom conditions via the Composition. For VMs, useful custom conditions include:

# In the Composition's pipeline step:
- step: report-vm-status
  functionRef:
    name: crossplane-function-patch-and-transform
  inputs:
    functionConfigRef:
      name: vm-status-config
    resources:
    - name: xr
      patches:
      # Report VM's running state
      - fromFieldPath: "resources.virtualmachine.status.conditions[0].status"
        toFieldPath: "status.conditions[?type=='VMRunning'].status"
      # Report DataVolume's download progress
      - fromFieldPath: "resources.datavolume.status.phase"
        toFieldPath: "status.conditions[?type=='DiskReady'].reason"

8.3 Status fields to expose

For VM workloads, the most useful status fields to report back:

  • status.endpoint — VM IP address (from KubeVirt's status.interfaces)
  • status.conditions[?type=='VMRunning'].status — Running, Stopped, or Paused
  • status.conditions[?type=='DiskReady'].reason — Succeeded, Pending, or CreateError
  • status.ready — Crossplane's auto-managed aggregate

9. Common Pitfalls and Anti-Patterns

9.1 XRD immutability

Critical

Once an XRD is created, its spec.versions[].schema is immutable. You cannot change the schema of an existing version. To modify the schema:

  1. Create a new version (e.g., v1beta1)
  2. Update the Composition to support both versions
  3. Migration: users create Claims in the new version
  4. Old Claims continue using the old version until manually migrated
Don't do this
Editing an XRD's schema after XRs or Claims exist will cause reconciliation failures. Always plan your schema evolution before the first Claim is created.

9.2 KubeVirt field immutability

Important

Many KubeVirt fields are immutable after the VM is created. Notably:

  • spec.template.spec.domain.devices.disks — cannot add/remove disks
  • spec.template.spec.domain.cpu — cannot change vCPU topology
  • spec.template.spec.volumes — cannot change volume names

If the XR spec changes these fields, Crossplane will patch the VirtualMachine, but KubeVirt will reject the patch. The XR will show a ReconcileError condition.

Mitigation
Mark KubeVirt-immutable fields as immutable: true in the XRD schema. This catches the error at Claim apply time rather than at reconciliation time.

9.3 DataVolume timing races

Important

If the DataVolume takes a long time to download an image (large ISO files), the VirtualMachine will be created but remain in Stopped state waiting for the DV to complete. This is normal, but can confuse monitoring.

Mitigation: Set spec.running: false in the Composition's base VM template, and use a patch to set running: true only after the DV is Ready. Or, use the spec.waitForFirstConsumer storage class.

9.4 Overly broad ProviderConfig permissions

Critical

Granting the provider SA */* permissions is a common shortcut that becomes a security risk. The minimum permissions for the KubeVirt provider:

resources:
- virtualmachines
- virtualmachineinstances
- datavolumes
- persistentvolumeclaims
verbs:
- get
- list
- watch
- create
- update
- patch
- delete

9.5 Using ArgoCD self-heal on Crossplane-managed resources

Critical

Never set ArgoCD's syncPolicy.autoSync on namespaces managed by Crossplane. ArgoCD will try to revert Crossplane's patches, and Crossplane will re-apply them, creating a sync loop.

ArgoCD should only manage: XRDs, Compositions, ProviderConfigs, ProviderConfigs, and Claims. It should ignore the managed resources that Crossplane creates (VirtualMachine, DataVolume, PVC).

10. Exercise: Design a Multi-Claim XRD

Design an XRD that supports three distinct claim types for the same underlying infrastructure:

  • WebServer — 2 vCPUs, 4Gi RAM, 50Gi disk, public networking
  • Database — 4 vCPUs, 16Gi RAM, 200Gi disk, private networking, SSD storage
  • Workstation — 8 vCPUs, 32Gi RAM, 500Gi disk, GPU, public + private networking

Requirements:

  1. All three claim types share the same underlying XVirtualMachine
  2. Each claim type has different required fields
  3. The Composition must handle all three types

Approach: Single XRD with discriminant field

Use a workloadType field in the spec as a discriminant. The Composition uses this to branch its patch logic.

apiVersion: apiextensions.crossplane.io/v1
kind: CompositeResourceDefinition
metadata:
  name: xvirtualmachines.infra.example.com
spec:
  group: infra.example.com
  names:
    kind: XVirtualMachine
    plural: xvirtualmachines
  claimNames:
    kind: VirtualMachineClaim
    plural: virtualmachineclaims
  versions:
  - name: v1alpha1
    served: true
    referenceable: true
    schema:
      openAPIV3Schema:
        type: object
        properties:
          spec:
            type: object
            required:
            - workloadType
            - compute
            - storage
            properties:
              workloadType:
                type: string
                enum: [WebServer, Database, Workstation]
              compute:
                type: object
                properties:
                  vcpus:
                    type: integer
                    minimum: 1
                    maximum: 64
                  memory:
                    type: string
                  gpu:
                    type: object
                    properties:
                      model: { type: string }
                      devices: { type: integer }
                # WebServer: no GPU
                # Database: no GPU
                # Workstation: GPU required
              storage:
                type: object
                properties:
                  size: { type: string }
                  storageClass: { type: string }
                  image: { type: string }
                # WebServer: standard storage class
                # Database: SSD class required
                # Workstation: high-performance class
              network:
                type: object
                properties:
                  public: { type: boolean }
                  private: { type: boolean }
                  macAddress: { type: string }
                # WebServer: public=true, private=false
                # Database: public=false, private=true
                # Workstation: public=true, private=true
              lifecycle:
                type: object
                properties:
                  running: { type: boolean }
                  evictionStrategy:
                    type: string
                    enum: [LiveMigrate, External, Halt, Migrate]
                  shutdown:
                    type: object
                    properties:
                      signal: { type: string }

Composition strategy

Use pipeline mode with Go-Templating because the Composition needs conditional logic based on workloadType:

# In the Go-Templating function:
{{- if eq .resources.XVirtualMachine.claim.spec.workloadType "Database" }}
  # Use SSD storage class
  # Set higher default vCPUs
  # Disable public networking
  # Add read-only replica volumes
{{- else if eq .resources.XVirtualMachine.claim.spec.workloadType "Workstation" }}
  # Add GPU device
  # Enable both public and private networking
  # Set higher memory defaults
{{- else }}
  # WebServer: standard config
  # Enable public networking
  # Standard storage class
{{- end }}

Alternative: Multiple XRDs

Instead of one XRD with a discriminant, you could create three separate XRDs:

  • XWebServer — with WebServerClaim
  • XDatabase — with DatabaseClaim
  • XWorkstation — with WorkstationClaim

Each XRD has its own tightly-scoped schema. Each has its own Composition with no conditionals. More XRDs to maintain, but each is simpler and easier to validate.

Tradeoff
Single XRD = fewer CRDs, more conditional logic in Composition.
Multiple XRDs = more CRDs, simpler Compositions, better per-type validation.
