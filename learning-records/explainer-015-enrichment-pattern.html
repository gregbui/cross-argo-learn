Enrichment Pattern — External Controllers for Crossplane XRs

The enrichment pattern uses external controllers — Kubernetes controllers that are not part of Crossplane's core reconciliation loop — to add data, metadata, and capabilities to Composite Resources (XRs) after the Composition has rendered its managed resources. This pattern bridges the gap between what Crossplane provisions and what the real world requires: DNS names, IP addresses, monitoring endpoints, security labels, service discovery entries, and more.

This explainer covers the why, how, and where of enrichment — from the conceptual model through concrete implementations with external controllers that enrich XRs for VM workloads.

Motivation: Why Enrichment?

Crossplane's Composition renders managed resources — a VirtualMachine, a DataVolume, a PVC. But a production-ready VM workload needs more than those three CRs. It needs:

  • A DNS record pointing to the VM's IP address
  • A monitoring endpoint registered in Prometheus
  • A network policy allowing specific traffic flows
  • A security scan result tagged on the XR
  • A service discovery entry in Consul or etcd
  • A certificate from Vault for TLS termination

These resources are external to the Composition. They live in separate controllers, separate namespaces, or even separate clusters. The enrichment pattern provides the mechanism to wire them into the XR lifecycle.

Without enrichment
Claim ──▶ XR ──▶ Composition ──▶ VirtualMachine ──▶ DataVolume ──▶ PVC

The VM runs, but nothing knows about it. No DNS, no monitoring, no security tags.

With enrichment
Claim ──▶ XR ──▶ Composition ──▶ VirtualMachine ──▶ DataVolume ──▶ PVC │ │ (enrichment) ▼ ┌─────────────────────────┐ │ External Controllers │ │ ┌───────────────────┐ │ │ │ DNS Controller │──▶ A record → vm-01.infra.local │ └───────────────────┘ │ │ ┌───────────────────┐ │ │ │ Monitoring Ctrl │──▶ ServiceMonitor → Prometheus │ └───────────────────┘ │ │ ┌───────────────────┐ │ │ │ Security Ctrl │──▶ VulnerabilityReport → XR.status │ └───────────────────┘ │ └─────────────────────────┘

The XR is enriched with DNS, monitoring, and security data — all managed by external controllers.

The core problem enrichment solves

Crossplane's Composition is great at provisioning — creating the resources it knows about. But it is not designed for:

What Crossplane does well What Crossplane doesn't do well
Creating Kubernetes-native resources Creating external DNS records (Route53, Cloudflare)
Managing CRDs it defines Integrating with services outside the cluster
Declarative resource rendering Fetching data from external APIs (IPAM, PKI, CMDB)
Reconciling managed resources Enriching XRs with cross-cutting concerns

Enrichment fills this gap by delegating cross-cutting concerns to specialized external controllers while keeping the XR as the single source of truth.

Conceptual Model

The enrichment pattern has three roles:

1. The Enrichment Source (the XR)

The XR is the source of truth for what needs enriching. It provides:

  • Labels — selectors that tell controllers which XRs to watch
  • Annotations — configuration hints for enrichment controllers
  • Status fields — data that external controllers write back
  • Spec fields — parameters that determine what enrichment is needed

2. The Enrichment Controller

The controller is a Kubernetes controller (not necessarily a Crossplane provider) that:

  • Watches for XRs (or their managed resources) matching specific labels
  • Reads XR spec/status to determine what enrichment is needed
  • Creates or manages external resources (DNS records, certificates, etc.)
  • Writes enrichment results back to the XR (status, annotations, or a dedicated enrichment CR)

3. The Enrichment Target (what gets created)

The target is the external resource or data that the controller creates:

  • CRDs — DNSRecord, ServiceMonitor, Certificate, NetworkPolicy
  • XR status fields — enriched IP, DNS name, scan results
  • Connection secrets — enriched with externally-obtained data
  • Annotations — metadata tags from external systems
Key distinction: Enrichment vs. Composition rendering

A Composition renders resources directly — the VirtualMachine, DataVolume, PVC are created by the Composition's patchAndTransform. Enrichment is indirect — the external controller observes the XR and creates supplementary resources on its own. The Composition does not create the DNS record; the DNS controller does.

Pattern Variants

There are three main ways enrichment manifests in Crossplane, each with different trade-offs:

Variant 1: Status Enrichment

External controllers write data directly into the XR's status subresource. The XR's XRD must define the status fields the controller is allowed to write.

# External controller writes:
status:
  enrichedIP: "10.0.1.42"
  dnsName: "vm-01.infra.local"
  scanStatus: "clean"

Best for: Simple data that the Claim consumer needs to read.

Variant 2: Resource Enrichment

External controllers create separate CRs that reference the XR via cross-references. The XR doesn't hold the data; the external controller manages it.

# Controller creates:
apiVersion: dns.infra.example.com/v1
kind: DNSRecord
metadata:
  name: vm-01-dns
spec:
  xrRef: xrvirtualmachine/webserver-01
  recordName: vm-01.infra.local

Best for: Resources with their own lifecycle (TTL, propagation, external API calls).

Variant 3: Connection Secret Enrichment

External controllers inject enriched data into the connection secret that Crossplane writes to the Claim's namespace. The secret contains both Crossplane-managed data and externally-enriched data.

# Connection secret contains:
data:
  endpoint: "vm-01.infra.local"    # from DNS controller
  ip: "10.0.1.42"                  # from IPAM controller
  ca-cert: "LS0t..."               # from PKI controller
  password: "c2VjcmV0"             # from Crossplane

Best for: Passing enriched data to workload pods that consume the resource.

When to use each variant

Scenario Recommended variant Reason
VM IP address for the Claim consumer Status enrichment Simple read; no separate lifecycle
DNS record with TTL and propagation tracking Resource enrichment Needs its own CRD and reconciliation
Connection data for a consuming application Connection secret enrichment Secret is the natural delivery mechanism
Security scan results visible in the XR Status enrichment Queryable; part of XR observability
Certificate management with external PKI Resource enrichment Certificates have their own lifecycle (renewal, rotation)

Enrichment Sources

External controllers need to know when to act and what to act on. Here are the common triggers:

Trigger 1: Label-based selection

The most common approach. The Composition adds a specific label to the XR, and the external controller watches for that label:

# Composition adds this label to the XR:
metadata:
  labels:
    infra.example.com/enrich: "dns,monitoring,security"

# External controller watches for this label:
# In the controller's reconciliation loop:
xrList := &infrav1.XRVirtualMachineList{}
client.List(ctx, xrList, client.MatchingLabels{
    "infra.example.com/enrich": "dns",
})

Trigger 2: Annotation-based configuration

Annotations tell the controller how to enrich, not just whether:

metadata:
  annotations:
    infra.example.com/dns-zone: "infra.local"
    infra.example.com/dns-ttl: "300"
    infra.example.com/monitoring-endpoint: "/metrics"
    infra.example.com/security-scan: "required"

Trigger 3: Status-based triggering

The controller watches for the XR to reach a specific condition before enriching. For example, a DNS controller waits for the VirtualMachine to be Running before creating an A record:

# Controller checks XR status before enriching:
if xr.Status.Conditions[0].Type == "Ready" &&
   xr.Status.Conditions[0].Status == "True" {
    // Create DNS record
} else {
    // Delete DNS record if VM is deleted
}

Trigger 4: Cross-reference from managed resources

The controller watches the managed resources created by the Composition (VirtualMachine, DataVolume) and enriches based on their state. This is the most robust trigger because it doesn't depend on XR labels:

# Controller watches VirtualMachine CRs created by Crossplane:
vmList := &kubevirtv1.VirtualMachineList{}
client.List(ctx, vmList, client.MatchingLabels{
    "crossplane.io/composite": "xrvirtualmachine/webserver-01",
})
Crossplane's managed resource label

Crossplane automatically adds crossplane.io/composite: <group>/<kind>/<name> to every managed resource it creates. External controllers can use this label to find which XR a managed resource belongs to — no custom labels needed.

Enrichment Targets

Where does the enriched data go? The target determines how the Claim consumer accesses it.

Target 1: XR Status (most common)

The XRD must declare the status fields the external controller is authorized to write:

spec:
  versions:
    - name: v1alpha1
      schema:
        openAPIV3Schema:
          type: object
          properties:
            status:
              type: object
              properties:
                enrichedDNS:
                  type: string
                  description: DNS name assigned by external DNS controller
                enrichedIP:
                  type: string
                  description: IP address assigned by IPAM controller
                securityScan:
                  type: object
                  properties:
                    status:
                      type: string
                      enum: [pending, scanning, clean, vulnerable]
                    findings:
                      type: integer
                    lastScan:
                      type: string
                      format: date-time

The external controller writes to /status using a StatusWriter:

xr := &infrav1.XRVirtualMachine{}
client.get(ctx, types.NamespacedName{Name: name}, xr)

xr.Status.EnrichedDNS = "vm-01.infra.local"
xr.Status.EnrichedIP = "10.0.1.42"
xr.Status.SecurityScan.Status = "clean"

client.Status().Update(ctx, xr)

Target 2: XR Annotations

For metadata that shouldn't be part of the schema (dynamic, unstructured data):

metadata:
  annotations:
    # Enrichment controllers write here:
    enriched.dns/record-id: "abc-123"
    enriched.security/scan-hash: "sha256:deadbeef"
    enriched.monitoring/sla-class: "gold"

Target 3: Dedicated Enrichment CRs

For complex enrichment that needs its own lifecycle, the controller creates a CR that references the XR:

apiVersion: enriched.infra.example.com/v1
kind: XREnrichment
metadata:
  name: webserver-01-enrichment
spec:
  xrRef:
    apiVersion: infra.example.com/v1alpha1
    kind: XRVirtualMachine
    name: webserver-01
  dns:
    recordName: vm-01.infra.local
    recordType: A
    ttl: 300
  monitoring:
    endpoint: /metrics
    port: 8080
    scrapeInterval: 15s
  security:
    scanStatus: clean
    lastScan: "2024-01-15T10:30:00Z"
    vulnerabilities: 0

Target 4: Connection Secrets

External controllers can inject data into the connection secret that Crossplane creates:

# Crossplane creates the base secret:
apiVersion: v1
kind: Secret
metadata:
  name: crossplane-kubevirt-webserver-01
type: Opaque
data:
  kubeconfig: <base64>

# External controller enriches it:
kubectl patch secret crossplane-kubevirt-webserver-01 \
  --type=merge \
  -p '{"data":{"dns-name":"dm0tMDEuaW5mcmEubG9jYWw=","ip-address":"MTAuMC4xLjQy="}}'
Connection secret write permissions

External controllers need update permission on the connection secret. If the secret is in crossplane-system (the default), the controller needs a ClusterRoleBinding or a ServiceAccount with appropriate RBAC. If the secret is in the Claim's namespace, the controller needs namespace-level permissions.

Implementation: Building an Enrichment Controller

Let's walk through building a concrete enrichment controller: a DNS enrichment controller that creates DNS A records when VMs become ready and deletes them when VMs are deleted.

Step 1: Define the enrichment scope

The controller will:

  • Watch VirtualMachine CRs created by Crossplane (via the crossplane.io/composite label)
  • Create DNS A records when the VM reaches Running phase
  • Delete DNS records when the VM is deleted
  • Write the DNS name back to the XR's status

Step 2: Controller reconciliation loop

package controller

import (
    "context"
    "fmt"

    corev1 "k8s.io/api/core/v1"
    metav1 "k8s.io/apimachinery/pkg/apis/meta/v1"
    "k8s.io/apimachinery/pkg/runtime"
    "k8s.io/apimachinery/pkg/types"
    "sigs.k8s.io/controller-runtime"
    "sigs.k8s.io/controller-runtime/pkg/builder"
    "sigs.k8s.io/controller-runtime/pkg/client"
    "sigs.k8s.io/controller-runtime/pkg/controller/controllerutil"
    "sigs.k8s.io/controller-runtime/pkg/handler"
    logf "sigs.k8s.io/controller-runtime/pkg/log"
    "sigs.k8s.io/controller-runtime/pkg/reconcile"

    kubevirtv1 "github.com/kubevirt/kubevirt/pkg/apis/core/v1"
    infrav1 "github.com/org/infra-api/api/infra.example.com/v1alpha1"
)

// DNSReconciler watches VirtualMachine CRs and creates DNS records
type DNSReconciler struct {
    client.Client
    Scheme *runtime.Scheme
    DNSZone string
    TTL    int32
}

func (r *DNSReconciler) Reconcile(ctx context.Context, req controller.Request) (controller.Result, error) {
    log := logf.FromContext(ctx)

    // 1. Get the VirtualMachine
    vm := &kubevirtv1.VirtualMachine{}
    if err := r.Get(ctx, req.NamespacedName, vm); err != nil {
        return controller.Result{}, client.IgnoreNotFound(err)
    }

    // 2. Find the parent XR via the Crossplane label
    xrName, found := vm.Labels["crossplane.io/composite"]
    if !found {
        // Not a Crossplane-managed VM, skip
        return controller.Result{}, nil
    }

    // Parse the XR reference: "xrvirtualmachine/webserver-01"
    parts := strings.Split(xrName, "/")
    if len(parts) != 2 {
        log.Error(fmt.Errorf("invalid crossplane.io/composite label"), "skipping VM", "vm", req.Name)
        return controller.Result{}, nil
    }

    // 3. Get the XR
    xr := &infrav1.XRVirtualMachine{}
    if err := r.Get(ctx, types.NamespacedName{Name: parts[1]}, xr); err != nil {
        return controller.Result{}, client.IgnoreNotFound(err)
    }

    // 4. Check if the XR is ready
    if !isXRReady(xr) {
        // Delete DNS record if it exists
        r.ensureDNSRecord(ctx, xr, vm, false)
        return controller.Result{}, nil
    }

    // 5. Determine the VM's IP address
    vmIP := getVMIP(vm)
    if vmIP == "" {
        return controller.Result{RequeueAfter: 10 * time.Second}, nil
    }

    // 6. Create or update the DNS record
    r.ensureDNSRecord(ctx, xr, vm, true)

    // 7. Write enriched DNS name back to XR status
    dnsName := fmt.Sprintf("%s.%s", parts[1], r.DNSZone)
    xr.Status.EnrichedDNS = dnsName
    if err := r.Client.Status().Update(ctx, xr); err != nil {
        log.Error(err, "failed to update XR status")
        return controller.Result{}, err
    }

    return controller.Result{}, nil
}

func isXRReady(xr *infrav1.XRVirtualMachine) bool {
    for _, cond := range xr.Status.Conditions {
        if cond.Type == "Ready" && cond.Status == corev1.ConditionTrue {
            return true
        }
    }
    return false
}

func getVMIP(vm *kubevirtv1.VirtualMachine) string {
    // The VM's status.addresses contains the IP
    for _, addr := range vm.Status.Addresses {
        if addr.Type == "InternalIP" {
            return addr.Address
        }
    }
    return ""
}

Step 3: Set up the controller manager

func (r *DNSReconciler) SetupWithManager(mgr controller.Manager) error {
    return controller.NewControllerManagedBy(mgr).
        For(&kubevirtv1.VirtualMachine{},
            builder.WithPredicates(
                predicate.NewPredicateFuncs(func(obj client.Object) bool {
                    vm := obj.(*kubevirtv1.VirtualMachine)
                    _, found := vm.Labels["crossplane.io/composite"]
                    return found
                }),
            ),
        ).
        Watches(
            &source.Kind{Type: &infrav1.XRVirtualMachine{}},
            handler.EnqueueRequestsFromMapFunc(func(ctx context.Context, obj client.Object) []reconcile.Request {
                // Reconcile DNS for VMs when the XR changes
                xr := obj.(*infrav1.XRVirtualMachine)
                // Find related VMs via label selector
                vms := &kubevirtv1.VirtualMachineList{}
                r.List(ctx, vms, client.MatchingLabels{
                    "crossplane.io/composite": fmt.Sprintf("xrvirtualmachine/%s", xr.Name),
                })
                var requests []reconcile.Request
                for _, vm := range vms.Items {
                    requests = append(requests, reconcile.Request{
                        NamespacedName: types.NamespacedName{
                            Name:      vm.Name,
                            Namespace: vm.Namespace,
                        },
                    })
                }
                return requests
            }),
        ).
        Complete(r)
}

Example 6: IPAM — Static IP Address Enrichment

When a VM needs a static IP address managed by an external IPAM (IP Address Management) system — such as a dedicated IPAM service, a network controller, or cross-cluster address coordination — the enrichment pattern provides the bridge between Crossplane's VM provisioning and the IPAM's address assignment.

How it works

The enrichment controller watches for Crossplane-managed VirtualMachines that need an IP, requests a static address from the IPAM service, updates the VM spec with the assigned IP, and writes the result back to the XR status.

# Controller creates this CR to track the IP allocation:
apiVersion: ipam.infra.example.com/v1
kind: IPAddress
metadata:
  name: vm-01-ip
  labels:
    crossplane.io/composite: xrvirtualmachine/webserver-01
spec:
  xrRef:
    apiVersion: infra.example.com/v1alpha1
    kind: XRVirtualMachine
    name: webserver-01
  network: default
  requestedIP: ""  # assigned by IPAM controller
status:
  assignedIP: "10.0.1.42"
  gateway: "10.0.1.1"
  dnsName: "vm-01.infra.local"
  allocationStatus: allocated

# Controller patches the VirtualMachine spec with the IP:
# spec.template.spec.networks[0].multus:
#   networkName: default
# spec.template.spec.networks[0].podNetwork.v4IPNet: "10.0.1.42/24"

# Controller writes to XR status:
# xr.Status.EnrichedIP = "10.0.1.42"
# xr.Status.EnrichedGateway = "10.0.1.1"
# xr.Status.EnrichedDNS = "vm-01.infra.local"

Controller reconciliation (simplified)

func (r *IPAMReconciler) Reconcile(ctx context.Context, req controller.Request) (controller.Result, error) {
    vm := &kubevirtv1.VirtualMachine{}
    r.Get(ctx, req.NamespacedName, vm)

    xrName, found := vm.Labels["crossplane.io/composite"]
    if !found { return controller.Result{}, nil }

    xr := &infrav1.XRVirtualMachine{}
    r.Get(ctx, types.NamespacedName{Name: xrName}, xr)

    // Check if XR has requested an IP
    if xr.Spec.NeedsIP != true {
        return controller.Result{}, nil
    }

    // Check if IP is already assigned
    if xr.Status.EnrichedIP != "" {
        return controller.Result{}, nil
    }

    // Request static IP from IPAM service
    ip, gateway, err := r.ipamClient.RequestStaticIP(ctx, &ipam.Request{
        Network:  xr.Spec.Network.NetworkName,
        Hostname: xr.Name,
        Tags:     map[string]string{"crossplane": "true"},
    })
    if err != nil {
        return controller.Result{RequeueAfter: 30 * time.Second}, err
    }

    // Patch the VirtualMachine spec with the assigned IP
    vm.Spec.Template.Spec.Networks = append(vm.Spec.Template.Spec.Networks,
        kubevirtv1.PodNetwork{
            PodNetwork: kubevirtv1.PodNetworkAnnotations{
                Name: xr.Spec.Network.NetworkName,
                PodNetworkV4IPNet: ip.String() + "/24",
            },
        },
    )
    r.Update(ctx, vm)

    // Write enriched data to XR status
    xr.Status.EnrichedIP = ip.String()
    xr.Status.EnrichedGateway = gateway.String()
    xr.Status.EnrichedDNS = fmt.Sprintf("%s.%s", xr.Name, r.DNSZone)
    r.Status().Update(ctx, xr)

    return controller.Result{}, nil
}

Use enrichment Use Composition directly
IP managed by external IPAM service IP from a predefined pool in the XRD spec
IPs coordinated across multiple clusters IPs local to a single cluster
IP lifecycle includes reservation, release, reclaim IP lifecycle tied to VM lifecycle only
IP must be assigned after VM is created (dynamic) IP known before VM creation
IPAM enrichment as a hybrid pattern

IPAM enrichment often combines resource enrichment (the IPAddress CR tracks the allocation) with status enrichment (the IP is written to XR.status for the Claim consumer) and spec patching (the controller updates the VM spec with the assigned IP). This is a common pattern where enrichment reaches back into the managed resource's spec, not just the XR's status.

Example 7: Naming Service — Unique VM Name Enrichment

When a VM claim is submitted without a specific name (or with a human-readable intent name like webserver), an external naming service assigns a unique, convention-compliant name for the actual VirtualMachine resource. The enrichment controller bridges the claim's intent name with the naming service's unique assignment.

How it works

The enrichment controller watches for new XRVirtualMachine claims, calls the naming service to get a unique name, creates the VirtualMachine with that assigned name, and writes the result back to the XR status.

# Naming service API:
# POST /api/v1/names
# Body: { "intent": "webserver", "env": "production", "tier": "frontend" }
# Response: { "name": "prod-fe-webserver-7f3a2b", "namespace": "workloads" }

# Controller creates the VirtualMachine with the assigned name:
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: prod-fe-webserver-7f3a2b    # assigned by naming service
  namespace: workloads
  labels:
    crossplane.io/composite: xrvirtualmachine/webserver
    infra.example.com/naming-service: "enabled"
    infra.example.com/naming-version: "v2"

# Controller writes to XR status:
# xr.Status.EnrichedName = "prod-fe-webserver-7f3a2b"
# xr.Status.EnrichedNamespace = "workloads"
# xr.Status.EnrichedNamingVersion = "v2"

Controller reconciliation (simplified)

func (r *NamingReconciler) Reconcile(ctx context.Context, req controller.Request) (controller.Result, error) {
    vm := &kubevirtv1.VirtualMachine{}
    r.Get(ctx, req.NamespacedName, vm)

    xrName, found := vm.Labels["crossplane.io/composite"]
    if !found { return controller.Result{}, nil }

    xr := &infrav1.XRVirtualMachine{}
    r.Get(ctx, types.NamespacedName{Name: xrName}, xr)

    // Check if XR already has an assigned name
    if xr.Status.EnrichedName != "" {
        return controller.Result{}, nil
    }

    // Call naming service for a unique name
    namingReq := &naming.Request{
        Intent: xr.Name,              // e.g., "webserver"
        Env:    xr.Labels["env"],     // e.g., "production"
        Tier:   xr.Labels["tier"],    // e.g., "frontend"
    }
    assigned, err := r.namingClient.AssignName(ctx, namingReq)
    if err != nil {
        return controller.Result{RequeueAfter: 30 * time.Second}, err
    }

    // Update the VirtualMachine name (or recreate if already created)
    vm.Name = assigned.Name
    vm.Namespace = assigned.Namespace
    if err := r.Update(ctx, vm); err != nil {
        return controller.Result{}, err
    }

    // Write to XR status
    xr.Status.EnrichedName = assigned.Name
    xr.Status.EnrichedNamespace = assigned.Namespace
    xr.Status.EnrichedNamingVersion = assigned.Version
    r.Status().Update(ctx, xr)

    return controller.Result{}, nil
}

Key design considerations

Concern Approach
VM already created with claim name before enrichment Controller detects mismatch and recreates VM with assigned name, or patches the XR to use the assigned name going forward
Naming service unavailable Controller falls back to a deterministic naming formula (e.g., {intent}-{random-8chars}) as a backup
Name collision across namespaces Naming service returns both name and namespace; controller ensures uniqueness at the cluster level
Claim name vs. VM name confusion Always reference the XR by its original claim name; the enriched name is the VM's actual name, stored in status
When enrichment is the right choice for naming

Use the enrichment pattern when the naming service has business rules beyond simple formatting — e.g., collision avoidance across clusters, environment-aware conventions, tier-based prefixes, or version tracking. If the naming logic is just {claim-name}-{random-suffix}, handle it in the Composition with a patchAndTransform string format instead.

The key signal: if the naming service has its own API, state, and versioning, it deserves its own enrichment controller.

Sync ordering

Use ArgoCD sync waves to ensure enrichment controllers are running before Claims are applied:

# In enrichment controller deployment:
metadata:
  annotations:
    argocd.argoproj.io/sync-wave: "10"

# In Crossplane XRD/Composition:
metadata:
  annotations:
    argocd.argoproj.io/sync-wave: "20"

# In Claims:
metadata:
  annotations:
    argocd.argoproj.io/sync-wave: "30"
Sync wave ordering

Wave 10 (controllers) → Wave 20 (XRDs + Compositions) → Wave 30 (Claims). This ensures enrichment controllers are ready to observe XRs before any Claims are created. Without this ordering, a Claim might be created before the enrichment controller is running, causing a missed enrichment cycle.

2. Controller reconciliation (simplified)

func (r *MonitoringReconciler) Reconcile(ctx context.Context, req controller.Request) (controller.Result, error) {
    vm := &kubevirtv1.VirtualMachine{}
    r.Get(ctx, req.NamespacedName, vm)

    xrName, found := vm.Labels["crossplane.io/composite"]
    if !found { return controller.Result{}, nil }

    xr := &infrav1.XRVirtualMachine{}
    r.Get(ctx, types.NamespacedName{Name: xrName}, xr)

    // Check if VM is running
    if vm.Status.Running != nil && *vm.Status.Running &&
       vm.Status.Phase == kubevirtv1.Running {

        // Create ServiceMonitor
        sm := &monitoringv1.ServiceMonitor{
            ObjectMeta: metav1.ObjectMeta{
                Name:      fmt.Sprintf("%s-monitor", xr.Name),
                Namespace: "monitoring",
                Labels:    map[string]string{"crossplane.io/composite": xrName},
            },
            Spec: monitoringv1.ServiceMonitorSpec{
                Selector: metav1.LabelSelector{
                    MatchLabels: map[string]string{
                        fmt.Sprintf("vm-%s-monitor", xr.Name): "true",
                    },
                },
                Endpoints: []monitoringv1.Endpoint{
                    {
                        Port: "metrics",
                        Path: "/metrics",
                        Interval: "15s",
                    },
                },
            },
        }
        controllerutil.SetControllerReference(xr, sm, r.Scheme)
        r.Create(ctx, sm)

        // Write to XR status
        xr.Status.Monitoring.Endpoint = fmt.Sprintf("http://%s:8080/metrics", xr.Name)
        xr.Status.Monitoring.Enabled = true
        r.Status().Update(ctx, xr)
    }

    return controller.Result{}, nil
}

3. Key design decisions

  • The ServiceMonitor is created in the monitoring namespace, not the VM's namespace. This is because Prometheus typically scrapes from a centralized monitoring namespace.
  • The SetControllerReference call ensures the ServiceMonitor is garbage-collected when the XR is deleted.
  • The controller watches VM status changes, not just spec changes — this is critical because the VM transitions from Stopped to Running asynchronously.

Summary

The enrichment pattern extends Crossplane's provisioning capabilities by delegating cross-cutting concerns to external controllers. The pattern has three variants:

Status
Write to XR.status
Resource
Create external CRs
Secret
Enrich connection secrets

The key principles:

  • Trigger — External controllers watch for XRs or managed resources via labels, annotations, or status conditions
  • Target — Enrichment data goes to XR status, separate CRs, or connection secrets
  • Decouple — The Composition handles provisioning; enrichment controllers handle everything else
  • GitOps — Deploy enrichment controllers via ArgoCD with proper sync-wave ordering
  • RBAC — External controllers need explicit permissions to read XRs and write to their status

Enrichment turns a bare VM into a production-ready workload: it gets DNS, monitoring, security scanning, certificates, network policies, static IP addresses from IPAM, and unique names from a naming service — all managed by specialized external controllers that observe the XR and act independently.
