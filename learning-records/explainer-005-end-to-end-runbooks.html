Part 1: End-to-End Walkthrough

Scenario: A developer provisions a web server VM

A developer in the dev team wants to spin up a CentOS 9 VM with 4 vCPUs, 8Gi memory, and a 50Gi disk. Here's the complete chain of events:

Step 1: Developer commits a Claim to Git

# workloads/dev/web-server/claim.yaml
apiVersion: infra.example.com/v1alpha1
kind: VirtualMachineClaim
metadata:
  name: web-server
  namespace: dev
spec:
  compositionSelector:
    matchLabels:
      infra.example.com/kubevirt: "true"
  parameters:
    compute:
      vcpus: 4
      memory: "8Gi"
      liveMigrate: true
    storage:
      size: "50Gi"
      storageClass: ocs.external.ceph.rook.io
      image: "https://download.centos.org/9/images/CentOS-Stream-GenericCloud-9-latest.x86_64.qcow2"
    network:
      macAddress: "52:54:00:ab:cd:ef"

Step 2: ArgoCD detects and syncs the Claim

ArgoCD's polling interval (default 3 minutes) or a Git webhook detects the new commit. The workloads Application syncs the Claim to the dev namespace.

# ArgoCD logs
time="2026-06-19T10:00:00Z" level=info msg="Syncing item" application=workloads item="VirtualMachineClaim:dev/web-server"
time="2026-06-19T10:00:01Z" level=info msg="Created" name=web-server namespace=dev

Step 3: Crossplane claim controller creates the XR

Crossplane's claim-controller observes the new VirtualMachineClaim in the dev namespace and creates an XVirtualMachine XR:

apiVersion: infra.example.com/v1alpha1
kind: XVirtualMachine
metadata:
  name: xvirtualmachine-web-server
  namespace: dev
  ownerReferences:
  - apiVersion: infra.example.com/v1alpha1
    kind: VirtualMachineClaim
    name: web-server
    controller: true
spec:
  compositionSelector:
    matchLabels:
      infra.example.com/kubevirt: "true"
  compute:
    vcpus: 4
    memory: "8Gi"
    liveMigrate: true
  storage:
    size: "50Gi"
    storageClass: ocs.external.ceph.rook.io
    image: "https://download.centos.org/9/images/CentOS-Stream-9.qcow2"
  network:
    macAddress: "52:54:00:ab:cd:ef"

Step 4: XR controller invokes the Composition

Crossplane's xrd-controller observes the new XR, matches it to the vm-kubevirt Composition via the selector label, and runs the Composition pipeline:

  1. render-vm-template (Go-Templating): Renders the VirtualMachine CR with the XR's parameters
  2. patch-dv-reference (Patch-and-Transform): Creates the DataVolume CR with the image URL and disk size
  3. link-dv-to-vm (Patch-and-Transform): Links the DataVolume to the VirtualMachine

Step 5: Composition renders managed resources

The pipeline outputs two managed resources:

# VirtualMachine managed resource
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: web-server-vm
  namespace: dev
  labels:
    infra.example.com/managed-by: crossplane
    infra.example.com/xr-name: xvirtualmachine-web-server
spec:
  running: true
  evictionStrategy: LiveMigrate
  template:
    spec:
      domain:
        cpu:
          cores: 4
        resources:
          requests:
            memory: 8Gi
        devices:
          disks:
          - name: disk0
            disk:
              bus: virtio
          interfaces:
          - macAddress: "52:54:00:ab:cd:ef"
            bridge: {}
      networks:
      - name: default-network
        pod: {}
      terminationGracePeriodSeconds: 30
      volumes:
      - name: disk0
        dataVolume:
          name: dv-web-server

# DataVolume managed resource
apiVersion: cdi.kubevirt.io/v1beta1
kind: DataVolume
metadata:
  name: dv-web-server
  namespace: dev
  labels:
    infra.example.com/managed-by: crossplane
spec:
  source:
    http:
      url: "https://download.centos.org/9/images/CentOS-Stream-9.qcow2"
  pvc:
    accessModes:
    - ReadWriteMany
    resources:
      requests:
        storage: 50Gi
    storageClassName: ocs.external.ceph.rook.io

Step 6: KubeVirt provider creates the CRs

The KubeVirt provider controller observes the managed resources and creates the actual VirtualMachine and DataVolume CRs in the cluster. These are the same objects (Crossplane manages them in-place).

Step 7: KubeVirt provisions the VM

  1. DataVolume downloads the image: CDI controller downloads the CentOS qcow2 image and creates a PVC
  2. KubeVirt creates the VMI: The VirtualMachine controller creates a VirtualMachineInstance (VMI) CR
  3. libvirt runs the VM: The VMI controller provisions the VM using libvirt on an OpenShift node

Step 8: Status propagates back to the Claim

Crossplane observes the VirtualMachine's status and writes it to the XR, which propagates back to the Claim:

apiVersion: infra.example.com/v1alpha1
kind: VirtualMachineClaim
metadata:
  name: web-server
  namespace: dev
spec:
  compositionSelector:
    matchLabels:
      infra.example.com/kubevirt: "true"
  parameters:
    compute:
      vcpus: 4
      memory: "8Gi"
status:
  conditions:
  - type: Ready
    status: "True"
    reason: CompositionReconcileSucceeded
  - type: Synced
    status: "True"
    reason: CrossplaneReconcileSucceeded
  endpoint: "10.128.0.45"

The developer can now SSH to 10.128.0.45. The VM is running.

Complete Request Flow
Git commit: claim.yaml
↓
ArgoCD syncs to cluster
↓
Claim → XR (claim controller)
↓
XR → Composition pipeline
↓
Compose: VM + DataVolume CRs
↓
KubeVirt: DV downloads image
↓
KubeVirt: VMI created
↓

↓
Status → XR → Claim (IP: 10.128.0.45)

Part 2: Operational Runbooks

Runbook 1: Debugging a Stuck VM Provisioning

Symptom: The Claim shows Ready: False and the XR is stuck in Creating state.

# 1. Check the Claim status
oc get virtualmachineclaim web-server -n dev -o yaml

# 2. Check the XR status
oc get xvirtualmachine xvirtualmachine-web-server -n dev -o yaml

# 3. Check Composition reconciliation events
oc get events -n dev --field-selector involvedObject.kind=XVirtualMachine

# 4. Check if managed resources were created
oc get virtualmachine -n dev
oc get datavolume -n dev

# 5. If DV is stuck downloading, check CDI logs
oc logs -l cdi.kubevirt.io/datavolume=dv-web-server -n openshift-cdi

# 6. Check if the VM is actually running
oc get vmi -n dev

# 7. Check KubeVirt controller logs
oc logs -l kubevirt.io=virt-controller -n openshift-virtualization -c virt-controller
Common causes
  • DV stuck in Pending: Image URL unreachable from cluster nodes, or storage class not available
  • VM stuck in Stopped: KubeVirt controller can't schedule the VMI (node resources exhausted)
  • XR stuck in Creating: Composition has a patch error; check CompositionFunction logs
  • Claim not syncing: ArgoCD is blocked on manual approval or has a sync conflict

Runbook 2: Upgrading the KubeVirt Provider

# 1. Update the provider version in Git
# Edit: crossplane/provider-kubevirt.yaml
# Change: package: crossplane/community-provider-kubevirt:v0.18.0
# To:     package: crossplane/community-provider-kubevirt:v0.19.0

# 2. Commit and push
git add crossplane/provider-kubevirt.yaml
git commit -m "chore: upgrade provider-kubevirt to v0.19.0"
git push

# 3. ArgoCD detects the change and syncs
# Wait for the Application to show Synced

# 4. Verify the new provider is running
oc get providers -n crossplane-system

# 5. Verify existing VMs are unaffected
oc get virtualmachine -n dev
oc get vmi -n dev

# 6. Check for reconciliation errors
oc get events -n crossplane-system --sort-by='.lastTimestamp' | tail -20
Safe upgrade practices
  • Always test provider upgrades in dev before prod
  • Existing VMs are not affected by provider upgrades — the CRDs are backward-compatible within minor versions
  • Roll back by reverting the Git commit; ArgoCD will sync the previous version
  • For major version upgrades, read the migration notes in the provider's release changelog

Runbook 3: Rolling Back a Bad Claim

Scenario: A developer committed a Claim with wrong parameters (e.g., 1000Gi disk instead of 50Gi). The VM is already provisioning.

# Option A: Delete the Claim (removes the VM)
oc delete virtualmachineclaim web-server -n dev
# ArgoCD will detect the deletion and re-sync from Git
# Since the Claim file still exists in Git, ArgoCD will recreate it
# So you must also fix the Git file first!

# Option B: Patch the Claim in Git, then let ArgoCD sync
# 1. Edit workloads/dev/web-server/claim.yaml with correct values
# 2. Commit and push
# 3. ArgoCD syncs the corrected Claim
# 4. Crossplane reconciles the XR with new parameters
# Note: Some parameters (disk size) can't be changed on an existing VM
# You may need to delete and recreate the VM for certain changes

# Option C: Delete the XR directly (bypasses ArgoCD)
oc delete xvirtualmachine xvirtualmachine-web-server -n dev
# This removes the VM but leaves the Claim in Git
# Next ArgoCD sync will recreate the Claim and XR
Warning: Parameter immutability
Some VM parameters (disk size, image) cannot be changed after the VM is created. Changing these in the Claim will cause Crossplane to report a reconciliation error. The safe pattern is: delete the Claim → fix the Git file → ArgoCD recreates the Claim → new VM is provisioned. Existing data is lost.

Exercise 005: Adapt the Task Plan — Answer Key

Q1: OpenShift version compatibility

Check your OpenShift version against the KubeVirt provider compatibility matrix:

# Check OpenShift version
oc version

# Check KubeVirt operator version
oc get csv -n openshift-virtualization -l operators.coreos.com/kubevirt.openshift-virtualization-operator.openshift-io/

Generally, OpenShift 4.12+ supports KubeVirt 1.1+. The Crossplane KubeVirt provider v0.18+ works with KubeVirt 1.1+. If you're on an older OpenShift, you may need to upgrade first or use an older provider version.

Q2: Storage classes

List available storage classes:

oc get storageclass

Common options on OpenShift:

  • ocs.external.ceph.rbd — Ceph RBD (recommended for VMs)
  • ocs.external.cephfs — CephFS (less common for VMs)
  • local-storage — Local PV (fast but no migration)
  • nfs — NFS (slow for VMs, but works)

For VMs, Ceph RBD is the standard choice. Ensure the CDI (Containerized Data Importer) operator is installed and configured to use your storage class.

Q3: Multi-namespace RBAC

For multiple teams, the RBAC decision depends on trust level:

SetupRBAC Strategy
Single team, single namespace Namespace-scoped Role + RoleBinding
Multiple teams, shared cluster ClusterRole + ClusterRoleBinding (simplest)
Multiple teams, strict isolation Per-namespace Role + RoleBinding + ProviderConfig per namespace

Q4: Git workflow

Recommended workflow for XRD and Composition changes:

  1. Feature branch: Developer creates a branch for XRD/Composition changes
  2. Review: Infrastructure team reviews the PR (XRD changes affect all users)
  3. Test: Merge to main triggers ArgoCD sync to a test environment
  4. Validate: Test a VM provision with the new XRD/Composition
  5. Production: Changes are automatically synced to production via ArgoCD
XRD immutability warning
XRDs are immutable in Crossplane. To change an XRD schema, you must delete the XRD (which deletes all existing XRs and Claims), then recreate it with the new schema. Plan your XRD design carefully before the first commit. Use spec.xVersions to support multiple XRD versions if you anticipate schema evolution.
