VM Configuration Data for Windows VMs

Explainer-010 covers Linux VM configuration via cloud-init write_files and puppet YAML profiles. Windows VMs use different mechanisms: Guest Custom Data (cloud-init for Windows), WinRM/PowerShell, and custom data injection. This explainer covers how to pass KV pairs (puppet profiles, app config, feature flags) into Windows VMs at provisioning time.

1. Why Linux Approach Doesn't Work for Windows

The Linux approach (cloud-init write_files dropping YAML at a known path) doesn't work for Windows because:

  • cloud-init for Windows is different — supports a subset of Linux cloud-init features, uses different file paths
  • File paths are different — Windows uses C:\ paths, not /etc/
  • Puppet on Windows uses different formats — Hiera YAML works, but installation and service management differ
  • No shell — cloud-init runcmd runs PowerShell on Windows, not bash
  • Line endings — Windows expects CRLF, not LF
Linux (explainer-010) Windows (this explainer) ───────────────────────────────── ───────────────────────────────── cloud-init write_files Guest Custom Data (cloud-init) /etc/puppet/profiles.yaml C:\puppet\profiles.yaml bash runcmd PowerShell runcmd puppet agent -t puppet agent -t YAML config file YAML or JSON config file chmod 0644 icacls /grant

2. Windows Configuration Mechanisms

Four approaches for passing config data into Windows VMs:

Mechanism Best For Complexity Git-Auditable
1. Guest Custom Data (cloud-init) First-boot config, puppet profiles Low Yes (XR)
2. PowerShell Custom Data Script Complex bootstrapping, package installs Medium Yes (XR)
3. WinRM Remote Execution Post-boot config, runtime updates High Partial
4. Guest Agent File Push Post-boot config updates Medium Yes (XR)

Approach 1 (Guest Custom Data) is recommended for first-boot provisioning. Approach 3 (WinRM) is best for runtime updates.

3. Guest Custom Data (Cloud-init for Windows)

KubeVirt supports cloud-init for Windows via the guestInfo cloud-init source. This is a subset of Linux cloud-init features, optimized for Windows.

3.1 Supported Features

  • hostname — set the Windows hostname
  • users — create/configure local users (limited)
  • password — set Administrator password
  • runcmd — run PowerShell commands at boot
  • write_files — write files to disk (limited support)
  • final_message — log a message at boot completion
Warning
Guest Custom Data for Windows does NOT support: network configuration, package management, SSH keys, or most Linux-specific directives. Use PowerShell scripts for complex bootstrapping.

3.2 Guest Custom Data Structure

The Guest Custom Data is passed via the VirtualMachine spec's spec.dataVolumeTemplates or spec.template.spec.domain.devices.autoboot with a cloud-init source.

# KubeVirt VM spec for Windows Guest Custom Data
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: win-server-01
spec:
  dataVolumeTemplates:
    - metadata:
        name: win-server-01-cloudinit
      spec:
        pvc:
          accessModes:
            - ReadWriteOnce
          resources:
            requests:
              storage: 1Mi
        source:
          guestInfo:
            userData: |
              #cloud-config
              hostname: win-server-01
              password: {{ .XR.spec.secretData.admin_password }}
              chpasswd:
                expire: false
              runcmd:
                - powershell -Command "New-Item -ItemType Directory -Path 'C:\puppet' -Force"
                - powershell -Command "Set-Content -Path 'C:\puppet\profiles.yaml' -Value '{{ .XR.spec.configData.puppet_yaml }}'"
                - powershell -Command "Start-Service -Name puppet"
              final_message: "Windows VM {{ .XR.metadata.name }} configured"

4. XRD Schema for Windows Config

Add Windows-specific config fields to your XRD. The XR spec holds the config data, and the composition transforms it into Windows-compatible formats.

# xrd/xvirtualmachine.yaml — Windows config fields
apiVersion: apiextensions.crossplane.io/v1
kind: XRD
metadata:
  name: xvirtualmachines.example.org
spec:
  versions:
    - name: v1alpha1
      schema:
        openAPIV3Schema:
          type: object
          properties:
            spec:
              type: object
              properties:
                # ── OS type ──
                osType:
                  type: string
                  enum:
                    - linux
                    - windows
                  description: >
                    Operating system type. Determines which config
                    mechanism is used (cloud-init for Linux, Guest
                    Custom Data for Windows).
                  default: linux

                # ── Windows-specific config ──
                windowsConfig:
                  type: object
                  description: >
                    Configuration data for Windows VMs. Transformed
                    into Guest Custom Data by the composition.
                  nullable: true
                  properties:
                    puppet_profile:
                      type: array
                      items:
                        type: string
                      description: >
                        List of puppet profiles to apply.
                        Example: ["base", "webserver", "monitoring"]
                    app_environment:
                      type: string
                      description: Application environment (dev/staging/prod)
                    app_config:
                      type: object
                      description: >
                        Application-specific key-value pairs.
                      nullable: true
                    features:
                      type: object
                      description: Feature flags
                      nullable: true
                    custom:
                      type: object
                      description: >
                        Arbitrary KV pairs. Mapped to a YAML file
                        that puppet reads.
                      nullable: true
                    admin_password:
                      type: string
                      description: >
                        Administrator password. Injected via K8s Secret
                        for security.
                      nullable: true

                # ── Sensitive data ──
                secretData:
                  type: object
                  description: >
                    Sensitive values. Injected separately.
                  nullable: true
                  properties:
                    admin_password:
                      type: string
                    db_password:
                      type: string
                    api_key:
                      type: string

Usage — user creates a Windows VM with config data:

# workloads/dev/win-server-01.yaml
apiVersion: example.org/v1alpha1
kind: VirtualMachineClaim
metadata:
  name: win-server-01
  namespace: workloads-dev
spec:

        log_level: info

        dark_mode: true
        beta_api: false

        iis_worker_processes: "4"

    secretData:
      admin_password: "S3cur3P@ss"

5. Composition: Generating Windows Config

The composition reads the XR spec and generates Guest Custom Data for Windows. This includes the YAML config file content and PowerShell commands to write it to disk.

5.1 Generate YAML Config Content

# compositions/xvirtualmachine-pipeline.yaml
- step: transform-windows-config
  functionRef:
    name: crossplane-function-patch-and-transform
  inputs:
    - apiVersion: pt.fn.crossplane.io/v1
      kind: Resources
      resources:
        - name: configmap
          base:
            apiVersion: v1
            kind: ConfigMap
            metadata:
              name: '{{ .XR.metadata.name }}-windows-config'
              namespace: '{{ .XR.metadata.namespace }}'
            data:
              # Puppet YAML config for Windows
              profiles.yaml: |
                # Auto-generated by Crossplane — do not edit
                # Source: {{ .XR.metadata.namespace }}/{{ .XR.metadata.name }}
                ---
                puppet_profile:
                {{- range .XR.spec.windowsConfig.puppet_profile }}
                  - {{ . | quote }}
                {{- end }}
                {{- if .XR.spec.windowsConfig.app_environment }}
                app_environment: {{ .XR.spec.windowsConfig.app_environment | quote }}
                {{- end }}
                {{- if .XR.spec.windowsConfig.app_config }}
                app_config:
                {{- range $key, $value := .XR.spec.windowsConfig.app_config }}
                  {{ $key }}: {{ $value | toPrettyJson }}
                {{- end }}
                {{- end }}
                {{- if .XR.spec.windowsConfig.features }}
                features:
                {{- range $key, $value := .XR.spec.windowsConfig.features }}
                  {{ $key }}: {{ $value }}
                {{- end }}
                {{- end }}
                {{- if .XR.spec.windowsConfig.custom }}
                custom:
                {{- range $key, $value := .XR.spec.windowsConfig.custom }}
                  {{ $key }}: {{ $value | toPrettyJson }}
                {{- end }}
                {{- end }}

5.2 Generate PowerShell Bootstrap Script

        - name: bootstrap-script
          base:
            apiVersion: v1
            kind: ConfigMap
            metadata:
              name: '{{ .XR.metadata.name }}-bootstrap'
              namespace: '{{ .XR.metadata.namespace }}'
            data:
              bootstrap.ps1: |
                # PowerShell bootstrap script for Windows VM
                # Generated by Crossplane — do not edit
                # Source: {{ .XR.metadata.namespace }}/{{ .XR.metadata.name }}

                $ErrorActionPreference = 'Stop'

                # Create puppet directory
                Write-Host "Creating puppet directory..."
                New-Item -ItemType Directory -Path 'C:\puppet' -Force | Out-Null

                # Write profiles.yaml
                Write-Host "Writing profiles.yaml..."
                $yamlContent = @'
                {{ .XR.Spec.WindowsConfig.PuppetYaml | default "# No config" }}
                '@
                Set-Content -Path 'C:\puppet\profiles.yaml' -Value $yamlContent -Encoding UTF8

                # Set permissions (icacls for Windows)
                Write-Host "Setting file permissions..."
                icacls 'C:\puppet\profiles.yaml' /grant 'Administrators:F' /T

                # Enable and start puppet service
                Write-Host "Starting puppet service..."
                if (Get-Service -Name 'puppet' -ErrorAction SilentlyContinue) {
                    Set-Service -Name 'puppet' -StartupType Automatic
                    Start-Service -Name 'puppet'
                    Write-Host "Puppet service started"
                } else {
                    Write-Warning "Puppet service not found — install puppet first"
                }

                # Write ready marker
                Write-Host "VM configured at $(Get-Date)" | Out-File -FilePath 'C:\VM_READY' -Encoding UTF8

                Write-Host "Bootstrap complete"

6. Puppet on Windows

Puppet runs on Windows with some differences from Linux. The Hiera YAML format is the same, but installation and service management differ.

6.1 Puppet Installation in Base Image

Include Puppet installation in your Windows base image build (Packer provisioner):

# Packer Windows provisioner
provisioner "powershell" {
  inline = [
    # Download and install Puppet
    'Invoke-WebRequest -Uri "https://apt.puppet.com/puppet7-release-windows.ps1" -OutFile "C:\puppet-install.ps1"',
    'powershell -ExecutionPolicy Bypass -File "C:\puppet-install.ps1"',
    # Enable and start puppet service
    'Start-Service -Name puppet',
    'Set-Service -Name puppet -StartupType Automatic',
    # Install Hiera YAML support
    'gem install yaml',
  ]
}

6.2 Puppet Hiera Configuration on Windows

The Hiera configuration is the same as Linux — Puppet reads YAML files from the hierarchy:

# C:\puppet\hiera.yaml (in base image)
---
version: 5
defaults:
  datadir: hieradata
  data_backend: yaml

hierarchy:
  - name: "Per-VM override (Crossplane)"
    paths:
      - "vm-override.yaml"
    data_backend: yaml

  - name: "Environment"
    paths:
      - "%{environment}.yaml"
    data_backend: yaml

  - name: "Common"
    paths:
      - "common.yaml"
    data_backend: yaml

6.3 Puppet Module Usage on Windows

Inside a Puppet module, consume the config data via Hiera lookups (same as Linux):

# manifests/init.pp (same as Linux)
class vm_config (
  Array[String] $profile = [],
  Hash        $app       = {},
  Hash        $features  = {},
) {

  $profiles = $profile ? {
    []    => hiera('puppet::profile', ['base']),
    default => $profile,
  }

  include $profiles

  if !empty($app) {
    file { 'C:/vm-config/app-config.yaml':
      content => yaml_encode($app),
      mode    => '0644',
    }
  }
}

7. PowerShell Custom Data Script

For complex bootstrapping, use a PowerShell script as the Guest Custom Data. This runs at first boot and handles all configuration.

# compositions/xvirtualmachine-pipeline.yaml — Guest Custom Data
- step: inject-guestcustomdata
  functionRef:
    name: crossplane-function-patch-and-transform
  inputs:
    - apiVersion: pt.fn.crossplane.io/v1
      kind: Resources
      resources:
        - name: guestcustomdata-secret
          base:
            apiVersion: v1
            kind: Secret
            metadata:
              name: '{{ .XR.metadata.name }}-guestcustomdata'
              namespace: '{{ .XR.metadata.namespace }}'
            type: Opaque
            stringData:
              userData: |
                #cloud-config
                hostname: {{ .XR.spec.guest.hostname | default .XR.metadata.name }}
                password: {{ .XR.spec.secretData.admin_password | default "P@ssw0rd!" }}
                chpasswd:
                  expire: false
                runcmd:
                  - powershell -Command "& {
                      Write-Host 'Creating puppet directory...';
                      New-Item -ItemType Directory -Path 'C:\puppet' -Force | Out-Null;

                      Write-Host 'Writing profiles.yaml...';
                      $yaml = @'
                      # Auto-generated by Crossplane
                      # Source: {{ .XR.metadata.namespace }}/{{ .XR.metadata.name }}
                      ---
                      puppet_profile:
                      {{- range .XR.spec.windowsConfig.puppet_profile }}
                        - {{ . | quote }}
                      {{- end }}
                      {{- if .XR.spec.windowsConfig.app_environment }}
                      app_environment: {{ .XR.spec.windowsConfig.app_environment | quote }}
                      {{- end }}
                      {{- if .XR.spec.windowsConfig.app_config }}
                      app_config:
                      {{- range $key, $value := .XR.spec.windowsConfig.app_config }}
                        {{ $key }}: {{ $value | toPrettyJson }}
                      {{- end }}
                      {{- end }}
                      {{- if .XR.spec.windowsConfig.features }}
                      features:
                      {{- range $key, $value := .XR.spec.windowsConfig.features }}
                        {{ $key }}: {{ $value }}
                      {{- end }}
                      {{- end }}
                      '@;
                      Set-Content -Path 'C:\puppet\profiles.yaml' -Value $yaml -Encoding UTF8;

                      Write-Host 'Starting puppet service...';
                      if (Get-Service -Name 'puppet' -ErrorAction SilentlyContinue) {
                        Set-Service -Name 'puppet' -StartupType Automatic;
                        Start-Service -Name 'puppet';
                      } else {
                        Write-Warning 'Puppet service not found';
                      }

                      Write-Host 'VM configured at $(Get-Date)';
                      'configured' | Out-File -FilePath 'C:\VM_READY' -Encoding UTF8;
                    }"
                final_message: "Windows VM {{ .XR.metadata.name }} configured"

8. Handling Sensitive Data

Windows VMs need sensitive data (admin password, DB credentials, API keys) handled securely.

8.1 Simple: inline in Guest Custom Data (dev/low-risk)

For non-production or low-risk secrets, inline them in the Guest Custom Data. They're base64-encoded in K8s but not encrypted at rest.

                password: {{ .XR.spec.secretData.admin_password | default "P@ssw0rd!" }}
                runcmd:
                  - powershell -Command "$env:DB_PASSWORD = '{{ .XR.spec.secretData.db_password }}'"
                  - powershell -Command "Set-Item -Path 'env:DB_PASSWORD' -Value '{{ .XR.spec.secretData.db_password }}'"

8.2 Production: External Secrets Operator (recommended)

For production, use the OpenShift External Secrets Operator to pull from Vault, AWS Secrets Manager, etc.

# compositions/xvirtualmachine-pipeline.yaml
- step: external-secrets
  functionRef:
    name: crossplane-function-patch-and-transform
  inputs:
    - apiVersion: pt.fn.crossplane.io/v1
      kind: Resources
      resources:
        - name: admin-password-secret
          base:
            apiVersion: external-secrets.io/v1beta1
            kind: ExternalSecret
            metadata:
              name: '{{ .XR.metadata.name }}-admin-password'
              namespace: '{{ .XR.metadata.namespace }}'
            spec:
              secretStoreRef:
                name: vault-backend
                kind: SecretStore
              target:
                name: '{{ .XR.metadata.name }}-admin-password'
                creationPolicy: Owner
              data:
                - secretKey: admin_password
                  remoteRef:
                    key: production/windows/admin-password
        - name: db-password-secret
          base:
            apiVersion: external-secrets.io/v1beta1
            kind: ExternalSecret
            metadata:
              name: '{{ .XR.metadata.name }}-db-password'
              namespace: '{{ .XR.metadata.namespace }}'
            spec:
              secretStoreRef:
                name: vault-backend
                kind: SecretStore
              target:
                name: '{{ .XR.metadata.name }}-db-password'
                creationPolicy: Owner
              data:
                - secretKey: db_password
                  remoteRef:
                    key: production/db/password

Then reference the created K8s Secret in Guest Custom Data:

                runcmd:
                  - powershell -Command "$env:DB_PASSWORD = (Get-Content 'C:\secrets\db_password.txt')"
                  - powershell -Command "Set-Item -Path 'env:DB_PASSWORD' -Value (Get-Content 'C:\secrets\db_password.txt')"

9. Updating Config Without Reboot

If you need to update config on a running Windows VM, you have two options:

9.1 Puppet runs periodically (default)

If Puppet is running on a schedule (every 30 min default), it will re-read the config file on each run. To trigger an immediate run:

# Via guest agent (requires guest agent running)
oc virt guest-exec <vm-name> -n workloads-dev -- powershell -Command "puppet agent -t"

# Via WinRM port-forward
oc port-forward <vm-name> 5985:5985 -n workloads-dev
# Then use Invoke-Command or psexec to run puppet agent -t

9.2 Update the XR → ArgoCD syncs → Puppet picks it up

# Update the VirtualMachineClaim
oc edit virtualmachineclaim win-server-01 -n workloads-dev
# Change windowsConfig.puppet_profile, save
# ArgoCD detects drift, syncs
# Puppet runs on next cycle, reads new file
# VM reconfigured — no reboot needed

10. Troubleshooting

10.1 Guest Custom Data not applied

# Check if Guest Custom Data was injected
oc get secret <vm-name>-guestcustomdata -n workloads-dev -o jsonpath='{.data.userData}'

# Check cloud-init status inside the VM
oc rsh <vm-debug-pod>
# Then into the guest:
Get-Content 'C:\var\log\cloud-init.log'
Get-Content 'C:\var\log\cloud-init-output.log'

# Check if VM_READY file exists
Get-Content 'C:\VM_READY'

10.2 Puppet not reading the file

# Verify file exists and is valid YAML
Get-Content 'C:\puppet\profiles.yaml'
# Check YAML syntax (PowerShell)
Import-Module -Name YAML
$data = Get-Content 'C:\puppet\profiles.yaml' | ConvertFrom-Yaml

# Check Puppet logs
Get-Content 'C:\ProgramData\PuppetLabs\Puppet\var\log\puppet.log' | Select-String -Pattern 'profile'

# Force Puppet to run
puppet agent -t --verbose

10.3 File permissions wrong

# Check permissions (icacls for Windows)
icacls 'C:\puppet\profiles.yaml'

# Should allow Administrators full control
# If wrong, fix:
icacls 'C:\puppet\profiles.yaml' /grant 'Administrators:F' /T

10.4 PowerShell errors in runcmd

# Check PowerShell error output
oc rsh <vm-debug-pod>
Get-Content 'C:\Windows\Temp\puppet-bootstrap.log'
