Application-Level VM Workload Development

Now that OS image preparation and guest tools are in scope, this explainer covers how VM workloads are built, configured, and managed within the Crossplane + ArgoCD + OpenShift Virtualization stack. You'll learn about OS images, cloud-init, QEMU guest agent, and how these integrate with your XRD and Composition design.

What this adds to the stack
Previously, we treated VMs as black boxes — the Composition created a VirtualMachine CR and KubeVirt ran it. Now we're looking inside the VM: how it boots, how it's configured on first boot, how it communicates with the host, and how images are managed at scale.

1. The VM Workload Stack

A VM workload is more than just a VirtualMachine CR. It's a stack of layers:

VM Workload Layers
Git: Claim YAML + image config
↓
ArgoCD syncs Claim
↓
Crossplane creates VM CR + DataVolume
↓
Disk Image (qcow2) with cloud-init support
↓
QEMU Guest Agent + virtio drivers
↓
Guest OS (CentOS, RHEL, Ubuntu, Windows)

Each layer has decisions that affect how the VM behaves, is managed, and integrates with the stack.

2. OS Image Preparation

Image Format: qcow2

KubeVirt's preferred image format is qcow2. It supports:

  • Copy-on-write — multiple VMs can share a base image, saving storage
  • Compression — smaller disk images reduce download time and storage
  • Snapshots — KubeVirt snapshots require qcow2 format
  • Cloud-init — the cloud-init drive is injected as a secondary qcow2 disk

Cloud-Init Support

cloud-init is the standard tool for configuring VMs on first boot. It runs when the VM starts for the first time and configures:

  • Hostname, SSH keys, users
  • Network configuration (static IP, DNS)
  • Package installation, script execution
  • Yum/DNF repository configuration

For cloud-init to work, the base image must have cloud-init installed and the cloud-init service enabled:

# On the base image (built with Packer, osc image-tee, or manual):
dnf install -y cloud-init
systemctl enable cloud-init
systemctl enable cloud-config
systemctl enable cloud-final

# Disable cloud-init's network configuration if OpenShift SDN handles networking
# (otherwise cloud-init may override the pod network settings)
echo "network_config_source: [cloud-init]" >> /etc/cloud/cloud.cfg.d/99-custom-networking.cfg
Critical: cloud-init vs. OpenShift networking
On OpenShift, the pod network (SDN/OVN-Kubernetes) handles VM networking. If cloud-init's network configuration runs, it may try to set up its own network interfaces, conflicting with OpenShift's pod network. Either:
  • Disable cloud-init networking: Set network: {config: disabled} in cloud-init config
  • Use a custom cloud-init config that only configures hostname, SSH keys, and user data — not networking
This is one of the most common gotchas when running VMs on OpenShift.

Building Base Images

There are several approaches to building base images for KubeVirt:

3. QEMU Guest Agent

The QEMU guest agent (qga) runs inside the VM and provides a communication channel between the VM and the host (KubeVirt controller). It enables:

  • File system freeze/thaw — for consistent snapshots (runs fsfreeze before taking a snapshot)
  • Guest OS info — reports guest OS type, hostname, IP addresses, running processes
  • Guest shutdown — graceful OS shutdown via the KubeVirt API
  • Guest network config — read IP addresses from inside the guest
  • Guest user/password changes — change passwords from the host

For the guest agent to work, the VM must have:

  1. Guest agent installed in the OS:
# RHEL/CentOS
dnf install -y qemu-guest-agent
systemctl enable --now qemu-guest-agent

# Ubuntu
apt install -y qemu-guest-agent
systemctl enable --now qemu-guest-agent
  1. Serial console configured in the VM spec:
spec:
  template:
    spec:
      domain:
        devices:
          interfaces:
          - masquerade: {}
            name: default
          disks:
          - name: rootdisk
            disk:
              bus: virtio
          - name: cloudinitdisk
            disk:
              bus: virtio
        serials:
        - name: serial0
          serial: ttyS0
        console:
        - name: serial0
          consoleType: serial
          port: 0
  1. Guest agent enabled in the VirtualMachine spec:
spec:
  running: true
  template:
    spec:
      domain:
        devices:
          interfaces:
          - masquerade: {}
            name: default
          # Guest agent channel
          autoattachGuestAgentInstance: true
Why this matters for your XRD
The guest agent is what enables Crossplane to report accurate status back to the Claim (IP addresses, running state). Without it, Crossplane can only report the VirtualMachine CR's phase (Running, Stopped) but not the guest's actual state (IP, hostname, disk usage). Consider adding guest-agent status fields to your XRD's status schema.

4. Cloud-Init in a Crossplane Composition

Cloud-init configuration is passed to the VM via a cloud-init user-data drive. In a Crossplane Composition, this is typically handled by the Go-Templating function:

# In the Composition's Go-Templating step, inject cloud-init user-data:
apiVersion: v1
kind: ConfigMap
metadata:
  name: vm-cloudinit-template
data:
  user-data.yaml: |
    #cloud-config
    hostname: {{ .XR.metadata.name }}
    fqdn: {{ .XR.metadata.name }}.{{ .XR.metadata.namespace }}.svc.cluster.local
    users:
    - name: {{ .XR.spec.guest.adminUser | default "cloud-user" }}
      ssh-authorized-keys:
      {{- range .XR.spec.guest.sshKeys }}
      - {{ . }}
      {{- end }}
      sudo: ['ALL=(ALL)']
      shell: /bin/bash
    chpasswd:
      expire: true
    package_update: true
    package_upgrade: false
    runcmd:
    {{- if .XR.spec.guest.postInstallCommands }}
    {{- range .XR.spec.guest.postInstallCommands }}
    - {{ . }}
    {{- end }}
    {{- end }}
    cloud_final_modules:
    - [scripts-user, always]
  meta-data.yaml: |
    local-hostname: {{ .XR.metadata.name }}
    instance-id: {{ .XR.metadata.name }}-{{ .XR.metadata.namespace }}

The Composition then creates a Secret or ConfigMap with this user-data and references it in the VirtualMachine's cloud-init disk:

spec:
  template:
    spec:
      domain:
        devices:
          disks:
          - name: cloudinitdisk
            disk:
              bus: virtio
          - name: rootdisk
            disk:
              bus: virtio
      volumes:
      - name: cloudinitdisk
        cloudInitNoCloud:
          secretRef: cloud-init-user-data-{{ .XR.metadata.name }}
      - name: rootdisk
        dataVolume:
          name: dv-{{ .XR.metadata.name }}
XRD schema extension
To support this, extend your XRD's spec to include guest configuration: 
properties:
  guest:
    type: object
    properties:
      adminUser:
        type: string
        description: Username for the admin account
      sshKeys:
        type: array
        items:
          type: string
        description: SSH public keys for the admin user
      postInstallCommands:
        type: array
        items:
          type: string
        description: Commands to run after initial setup

5. Windows VM Support

Windows VMs require additional configuration beyond Linux:

VirtIO Drivers

Windows needs VirtIO drivers for network, disk, and paravirtualized devices. These must be available at boot time:

spec:
  template:
    spec:
      domain:
        devices:
          disks:
          - name: rootdisk
            disk:
              bus: virtio
          - name: virtio-drivers
            cdrom:
              bus: sata
            readonly: true
          - name: cloudinitdisk
            disk:
              bus: sata
        interfaces:
        - masquerade: {}
          name: default
        volumes:
        - name: virtio-drivers
          dataVolume:
            name: dv-virtio-drivers
        - name: cloudinitdisk
          cloudInitNoCloud:
            secretRef: cloud-init-user-data-{{ .XR.metadata.name }}
        - name: rootdisk
          dataVolume:
            name: dv-{{ .XR.metadata.name }}

Windows cloud-init Alternative: Sysprep

Windows doesn't use cloud-init. Instead, use sysprep for first-boot configuration:

spec:
  template:
    spec:
      domain:
        devices:
          disks:
          - name: windows-sysprep
            cdrom:
              bus: sata
            readonly: true
        volumes:
        - name: windows-sysprep
          cloudInitConfigDrive:
            secretRef: sysprep-config

The sysprep ConfigDrive contains an autounattend.xml file that configures the Windows VM on first boot.

6. Image Management at Scale

As your team provisions more VMs, managing base images becomes important:

Base Image Registry

Store base images in a registry accessible from the OpenShift cluster:

# Option 1: OpenShift Image Registry (for containerized images)
# Convert qcow2 to a container and push to the internal registry
oc image export registry.internal.example.com/base-images/centos-stream-9:latest /tmp/centos.qcow2

# Option 2: HTTP-accessible image store
# Host images on an internal HTTP server
image: "https://images.internal.example.com/centos-stream-9.qcow2"

# Option 3: PVC-based image store
# Pre-provision PVCs with images and reference them directly
storageClass:
  existingPvc: base-images/centos-stream-9

Image Templates

Define image templates in your XRD to standardize which images teams can use:

spec:
  parameters:
    image:
      type: object
      properties:
        name:
          type: string
          enum:
          - centos-stream-9
          - rhel-9
          - ubuntu-22.04
        version:
          type: string
          enum:
          - latest
          - "20260601"  # date-based versioning

This prevents teams from referencing arbitrary URLs and ensures only approved, security-scanned images are used.

Image Update Workflow

# 1. Build new base image (via Packer, OSBuild, etc.)
# 2. Scan for vulnerabilities (Trivy, Grype, etc.)
# 3. Push to image registry
# 4. Update the image template in Git
# 5. ArgoCD syncs the new template
# 6. New VMs use the new image
# 7. Existing VMs are NOT affected (they have their own disk)
# 8. To update existing VMs: snapshot → delete → recreate with new image
Image update caveat
Existing VMs have their own persistent disk. Changing the base image in the Composition does NOT update running VMs. To update an existing VM:
  1. Take a snapshot of the VM's disk
  2. Delete the VM (and its Claim)
  3. Update the Claim with the new image reference
  4. Create a new Claim — Crossplane provisions a new VM with the new image
This is a fundamental difference from container workloads where images are pulled fresh on every deployment.

7. Try This: Design the Guest Config Schema

Exercise
Your team wants to support both Linux and Windows VMs. Design the guest section of your XRD spec that supports:
  1. Linux: SSH keys, admin user, cloud-init user-data, post-install commands
  2. Windows: admin password (hashed), sysprep configuration, VirtIO driver reference
  3. A shared field for both: hostname, tags for organization
How would you structure the schema? How would the Composition handle the different cloud-init vs. sysprep paths?

Exercise 006: Guest Config Schema Design — Answer Key

XRD Schema

spec:
  parameters:
    guest:
      type: object
      properties:
        # Shared fields
        hostname:
          type: string
          description: Hostname for the VM
        tags:
          type: object
          additionalProperties:
            type: string
          description: Key-value tags for organization

        # Linux-specific
        linux:
          type: object
          properties:
            adminUser:
              type: string
              default: cloud-user
            sshKeys:
              type: array
              items:
                type: string
              description: SSH public keys
            postInstallCommands:
              type: array
              items:
                type: string
              description: Commands to run after initial setup
            disablePasswordAuth:
              type: boolean
              default: true
              description: Disable password-based SSH login

        # Windows-specific
        windows:
          type: object
          properties:
            adminPasswordSecretRef:
              type: object
              properties:
                name:
                  type: string
                key:
                  type: string
              description: Reference to a Secret containing the admin password
            domainJoin:
              type: object
              properties:
                domainName:
                  type: string
                username:
                  type: string
                passwordSecretRef:
                  type: object
                  properties:
                    name:
                      type: string
                    key:
                      type: string
              description: Active Directory domain join configuration

Composition Handling

The Composition uses Go-Templating to branch based on whether linux or windows is specified:

# Go-Templating template for cloud-init vs sysprep
{{- if .XR.spec.guest.linux }}
# Linux: cloud-init NoCloud
spec:
  template:
    spec:
      volumes:
      - name: cloudinitdisk
        cloudInitNoCloud:
          secretRef: cloud-init-user-data-{{ .XR.metadata.name }}
{{- else if .XR.spec.guest.windows }}
# Windows: cloud-init ConfigDrive for sysprep
spec:
  template:
    spec:
      volumes:
      - name: sysprep
        cloudInitConfigDrive:
          secretRef: sysprep-config-{{ .XR.metadata.name }}
      - name: virtio-drivers
        dataVolume:
          name: dv-virtio-drivers
{{- end }}

The Composition also conditionally includes the VirtIO driver CD-ROM only for Windows VMs, and sets different disk buses (virtio for Linux, SATA for Windows cloud-init disk).

Secret Management

Passwords and sensitive data should be stored in Kubernetes Secrets, not in the Claim YAML. The Composition creates the cloud-init secret from the XR's parameters:

# Composition creates a Secret with cloud-init user-data
apiVersion: v1
kind: Secret
metadata:
  name: cloud-init-user-data-{{ .XR.metadata.name }}
type: Opaque
stringData:
  user-data: |
    #cloud-config
    hostname: {{ .XR.spec.guest.hostname | default .XR.metadata.name }}
    users:
    - name: {{ .XR.spec.guest.linux.adminUser | default "cloud-user" }}
      ssh-authorized-keys:
      {{- range .XR.spec.guest.linux.sshKeys }}
      - {{ . }}
      {{- end }}
  meta-data: |
    local-hostname: {{ .XR.spec.guest.hostname | default .XR.metadata.name }}
Security note
For production, consider using an external secrets manager (HashiCorp Vault, AWS Secrets Manager, OpenShift External Secrets Operator) instead of storing passwords in Kubernetes Secrets. The Composition would reference the external secret and inject the value at runtime.
