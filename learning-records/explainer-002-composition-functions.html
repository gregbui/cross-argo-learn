Composition Functions

Crossplane Compositions can be written two ways: the old declarative style (patches + connections) and the new pipeline style (Composition Functions). This explainer covers the function-based approach — what functions are, the three built-in functions, and how to choose between them.

Why functions exist
Declarative patches can do field-level mapping and simple transforms, but they cannot express loops, conditionals, or complex data manipulation. Composition Functions are small programs that run during composition rendering, giving you full programming capability within the reconciliation loop.

1. The Composition Function Model

A Composition Function is a container image that implements the Composition Function Runtime gRPC protocol. Crossplane calls it during reconciliation, passes it the XR and any inputs, and expects back a list of managed resources to create or update.

Function Invocation During Reconciliation
XR (reconciling) → Composition Controller → Function Runtime → Function Container → Managed Resources

The Composition declares a mode: Pipeline with ordered steps. Each step references a function and optionally passes inputs (manifests, function config, or XR data):

spec:
  mode: Pipeline
  pipeline:
  - step: render-template
    functionRef:
      name: crossplane-function-go-templating
    inputs:
    - name: vm-base
      base:
        apiVersion: kubevirt.io/v1
        kind: VirtualMachine
        metadata:
          labels:
            managed-by: crossplane
        spec:
          running: true
          template:
            spec:
              domain:
                devices: {}
  - step: apply-patches
    functionRef:
      name: crossplane-function-patch-and-transform
    inputs:
      functionConfigRef:
        name: vm-patch-config
  - step: finalize
    functionRef:
      name: crossplane-function-go-templating
    inputs:
    - name: vm-final
      base:
        apiVersion: kubevirt.io/v1
        kind: VirtualMachine
        metadata:
          labels:
            managed-by: crossplane
Key concept
Each function step receives the output of the previous step. The first step gets the XR + its own inputs. The last step's output becomes the Composition's final list of managed resources. This is why steps are ordered — data flows through the pipeline.

2. The Three Built-in Functions

Declarative Patch-and-Transform

What it does: Applies YAML patches to resources in the pipeline. Think of it as "find and replace" for Kubernetes manifests.

Best for: Field mapping, adding labels/annotations, simple value transforms.

Limitations: No loops, no conditionals, no dynamic resource creation. One input resource → one output resource (per patch set).

Config: Uses a FunctionConfig CRD with patches and transforms arrays.

Imperative Go-Templating

What it does: Renders resources from Go templates (Go template language + Sprig functions). Supports loops, conditionals, and data manipulation.

Best for: Dynamic resource counts (e.g., N disks), conditional fields, complex value computation.

Limitations: Template syntax can get complex. Debugging template errors requires reading function logs.

Config: Template strings inline in the Composition or stored as ConfigMaps.

The third function — crossplane-function-utam (Useful Templating And Mapping) — is a newer addition that combines declarative patching with limited templating. It's worth knowing about but less mature than the other two.

3. Patch-and-Transform in Detail

This is the most common function for VM provisioning. It uses a FunctionConfig to declare patches that transform input resources.

The FunctionConfig

apiVersion: pkg.crossplane.io/v1beta1
kind: FunctionConfig
metadata:
  name: vm-patch-config
spec:
  parameters:
    compute:
      vcpus: 4
      memory: "8Gi"
    storage:
      size: "50Gi"
      storageClass: ocs.external.ceph.rook.io

The Patches in the FunctionConfig

apiVersion: pkg.crossplane.io/v1beta1
kind: FunctionConfig
metadata:
  name: vm-patch-config
spec:
  patches:
  # Map XR spec.compute.vcpus → VM spec.template.spec.domain.cpu.cores
  - from: XR
    to: spec.template.spec.domain.cpu.cores
    policy:
      optional: false
    transforms:
    - type: integerToCore
      integerToCore: {}

  # Map XR spec.compute.memory → VM spec.template.spec.domain.resources.requests.memory
  - from: XR
    to: spec.template.spec.domain.resources.requests.memory
    policy:
      optional: false

  # Map XR spec.storage.size → DataVolume spec.pvc.resources.requests.storage
  - from: XR
    to: spec.pvc.resources.requests.storage
    transforms:
    - type: parseDuration
      parseDuration:
        from: spec.storage.size

  # Add labels to all managed resources
  - type: mapFrom
    mapFrom:
    - path: metadata.labels
      ref:
        kind: XR
    to: metadata.labels

  # Set VM running state from XR
  - from: XR
    to: spec.running
    transforms:
    - type: const
      const:
        val: true
Patch policy
optional: true — skip the patch if the source field is missing (don't error).
optional: false — fail the composition if the source field is missing (strict validation).
fromField/toField — map between specific fields.
fromValue/toField — set a constant value.
mapFrom/mapTo — copy entire objects or arrays.

Transforms Available

TransformPurposeExample
const Set a fixed value {val: "true"} — always set running: true
math Arithmetic on numeric fields {op: "multiply", factor: 1024} — convert Gi to Mi
string String manipulation {op: "format", format: "vm-%s"} — prefix with name
date Date formatting {format: "2006-01-02"} — RFC 3339 date
parseDuration Parse Kubernetes duration strings {} — converts "5m" to nanoseconds
converter Convert between types {kind: "toString"} or {kind: "toInt"}
password Generate random passwords {length: 32} — generates 32-char random string

4. Go-Templating in Detail

Go-Templating uses Go's template engine with Sprig function extensions. It's the choice when you need logic that patches alone can't express.

Use cases for Go-Templating

  • Dynamic resource counts: Create N number of disks based on spec.storage.disks array
  • Conditional fields: Only set spec.evictionStrategy if the XR specifies live-migration support
  • Computed values: Generate a unique VM name from the XR name + namespace
  • Complex networking: Build a list of network definitions from an XR array input

Example: Conditional disk configuration

# Composition step using Go-Templating
- step: render-disks
  functionRef:
    name: crossplane-function-go-templating
  inputs:
  - name: disk-template
    ref:
      kind: ConfigMap
      apiVersion: v1
      name: vm-disk-template
    mode: Rendered
# ConfigMap: vm-disk-template
apiVersion: v1
kind: ConfigMap
metadata:
  name: vm-disk-template
data:
  template.yaml: |
    apiVersion: kubevirt.io/v1
    kind: VirtualMachine
    metadata:
      name: {{ .XR.metadata.name }}-vm
      labels:
        infra.example.com/xr-name: {{ .XR.metadata.name }}
    spec:
      running: {{ if .XR.spec.compute.liveMigrate }}true{{ else }}false{{ end }}
      template:
        spec:
          domain:
            cpu:
              cores: {{ .XR.spec.compute.vcpus }}
            devices:
              disks:
              {{- range $i, $disk := .XR.spec.storage.disks }}
              - name: disk-{{ $i }}
                disk:
                  bus: {{ $disk.bus | default "virtio" }}
              {{- end }}
              {{- if .XR.spec.storage.bootDevice }}
              bootOrder: {{ .XR.spec.storage.bootDevice.order }}
              {{- end }}
            resources:
              requests:
                memory: {{ .XR.spec.compute.memory }}
          networks:
          {{- range .XR.spec.network.interfaces }}
          - name: {{ .name }}
            {{- if .pod }}
            pod: {}
            {{- end }}
            {{- if .multus }}
            multus:
              networkName: {{ .multus.networkName }}
            {{- end }}
          {{- end }}
          volumes:
          {{- range $i, $disk := .XR.spec.storage.disks }}
          - name: disk-{{ $i }}
            {{- if $disk.dataVolume }}
            dataVolume:
              name: dv-{{ $.XR.metadata.name }}-{{ $i }}
            {{- else if $disk.existingPvc }}
            persistentVolumeClaim:
              claimName: {{ $disk.existingPvc }}
            {{- end }}
          {{- end }}
Template data available
The template receives .XR (the composite resource), .Params (composition parameters), and .Resources (output from previous pipeline steps). Sprig functions like default, join, replace, regexReplaceAll are all available.

5. Choosing Between Patch-and-Transform and Go-Templating

ScenarioFunctionWhy
Map 5-10 fields from XR to VM spec Patch-and-Transform Simple, declarative, easy to review. No templates needed.
Create a disk for each item in an XR array Go-Templating Patches can't loop. Templates handle range natively.
Conditionally set evictionStrategy Go-Templating Templates have if/else. Patches have optional but can't branch.
Add labels, set constant values Patch-and-Transform These are single-field operations. Overkill for templates.
Generate random passwords for VM root user Patch-and-Transform Has a built-in password transform. No template needed.
Complex multi-resource orchestration Both (pipeline) Use Patch-and-Transform for field mapping, Go-Templating for dynamic resources.
Practical recommendation for your VM stack
Start with Patch-and-Transform for the basic VM spec (CPU, memory, disk size, storage class). Add Go-Templating only when you need dynamic features: multiple disks, conditional networking, or computed field values. You can mix both in the same pipeline — use Patch-and-Transform for simple mapping, then Go-Templating for the complex parts.

6. Composition Functions: Installation on OpenShift

Composition Functions run as pods in the cluster. They need to be installed before you can use them in Compositions.

Installation via xpkg

# Install the Patch-and-Transform function
crossplane xpkg install-function \
  crossplane/function-patch-and-transform:v0.14.0

# Install the Go-Templating function
crossplane xpkg install-function \
  crossplane/function-go-templating:v0.40.0

This creates a CompositionFunction CR and deploys the function as a Deployment in the crossplane-system namespace. The function pods run as sidecars during composition rendering.

Permissions needed

Composition Functions need permissions to read XRs and write managed resources. Crossplane's function runtime handles most of this, but on OpenShift you need to ensure:

  • The crossplane-system ServiceAccount has view access to XRs and Claims
  • The function pods run with a ServiceAccount that has permissions to create/update the managed resource types (VirtualMachine, DataVolume, etc.)
  • OpenShift's SecurityContextConstraints (SCC) allow the function pods (usually restricted SCC is fine)

7. Try This: Design a Pipeline

Exercise
Design the pipeline steps for a VM Composition that needs to:
  1. Create a VirtualMachine CR with CPU, memory, and basic disk
  2. Create a DataVolume CR for the disk image download
  3. Link the DataVolume's PVC name to the VirtualMachine's volume reference
  4. Only set evictionStrategy: LiveMigrate if the XR has spec.compute.liveMigrate: true
For each step, say which function you'd use and why. What would the patch config or template look like for the tricky parts?

Exercise 002: VM Composition Pipeline — Answer Key

Step 1: render-vm-template — Function: Go-Templating

Why Go-Templating: We need a conditional (evictionStrategy only when liveMigrate: true) and we're building the full VM spec from scratch. Templates handle if/else natively; Patch-and-Transform cannot branch.

- step: render-vm-template
  functionRef:
    name: crossplane-function-go-templating
  inputs:
  - name: vm-base
    ref:
      kind: ConfigMap
      apiVersion: v1
      name: vm-render-template
    mode: Rendered
# ConfigMap: vm-render-template
apiVersion: v1
kind: ConfigMap
metadata:
  name: vm-render-template
data:
  template.yaml: |
    apiVersion: kubevirt.io/v1
    kind: VirtualMachine
    metadata:
      name: {{ .XR.metadata.name }}-vm
      labels:
        infra.example.com/managed-by: crossplane
    spec:
      running: true
      {{- if .XR.spec.compute.liveMigrate }}
      evictionStrategy: LiveMigrate
      {{- end }}
      template:
        spec:
          domain:
            cpu:
              cores: {{ .XR.spec.compute.vcpus }}
            resources:
              requests:
                memory: {{ .XR.spec.compute.memory }}
            devices:
              disks:
              - name: disk0
                disk:
                  bus: virtio
          networks:
          - name: default-network
            pod: {}
          volumes:
          - name: disk0
            dataVolume:
              name: dv-placeholder  # patched in step 2

Step 2: patch-dv-reference — Function: Patch-and-Transform

Why Patch-and-Transform: This is a simple field mapping — take the XR's storage image URL and put it into the DataVolume spec, then set the PVC size and storage class. No loops, no conditionals, just map fields.

- step: patch-dv-reference
  functionRef:
    name: crossplane-function-patch-and-transform
  inputs:
  - name: datavolume
    base:
      apiVersion: cdi.kubevirt.io/v1beta1
      kind: DataVolume
      metadata:
        labels:
          infra.example.com/managed-by: crossplane
        name: dv-{{ .XR.metadata.name }}
      spec:
        pvc:
          accessModes:
          - ReadWriteMany
          resources:
            requests:
              storage: 1Gi
          storageClassName: default
        source:
          http:
            url: placeholder
    functionConfigRef:
      name: vm-dv-patch-config
# FunctionConfig: vm-dv-patch-config
apiVersion: pkg.crossplane.io/v1beta1
kind: FunctionConfig
metadata:
  name: vm-dv-patch-config
spec:
  patches:
  - from: XR
    to: spec.source.http.url
    policy:
      optional: false
  - from: XR
    to: spec.pvc.resources.requests.storage
    policy:
      optional: false
  - from: XR
    to: spec.pvc.storageClassName
    policy:
      optional: false
  - type: const
    const:
      to: spec.pvc.accessModes[0]
      val: ReadWriteMany

Step 3: link-dv-to-vm — Function: Patch-and-Transform

Why Patch-and-Transform: This is a cross-resource connection — take the DataVolume's PVC name (output of step 2) and inject it into the VirtualMachine's volume reference. This is exactly what Patch-and-Transform's cross-resource patching is designed for.

- step: link-dv-to-vm
  functionRef:
    name: crossplane-function-patch-and-transform
  inputs:
  - ref:
      name: dv-{{ .XR.metadata.name }}
      kind: DataVolume
    policy:
      optional: false
  - ref:
      name: {{ .XR.metadata.name }}-vm
      kind: VirtualMachine
    policy:
      optional: false
  functionConfigRef:
    name: vm-link-patch-config
Implementation note
The exact field to use for the DV→VM link depends on the KubeVirt provider version. In some versions, the DataVolume's status.pvc.name is the field to reference. In others, you may need to use Crossplane's connections mechanism (declarative style) instead of pipeline steps.

Alternative: Pure Declarative (No Functions)

If the VM spec is simple enough (no conditionals, no dynamic resources), you can skip Composition Functions entirely and use the declarative style with a single Composition YAML:

apiVersion: apiextensions.crossplane.io/v1
kind: Composition
metadata:
  name: vm-kubevirt
  labels:
    crossplane.io/xrd: xvirtualmachines.infra.example.com
spec:
  mode: Declarative
  resources:
  - name: virtualmachine
    base:
      apiVersion: kubevirt.io/v1
      kind: VirtualMachine
      spec:
        running: true
        template:
          spec:
            domain:
              cpu:
                cores: 1
              resources:
                requests:
                  memory: 1Gi
              devices:
                disks:
                - name: disk0
                  disk:
                    bus: virtio
            volumes:
            - name: disk0
              dataVolume:
                name: placeholder
    patches:
    - fromFieldPath: spec.parameters.compute.vcpus
      toFieldPath: spec.template.spec.domain.cpu.cores
    - fromFieldPath: spec.parameters.compute.memory
      toFieldPath: spec.template.spec.domain.resources.requests.memory
    - fromFieldPath: spec.parameters.compute.liveMigrate
      toFieldPath: spec.evictionStrategy
      transforms:
      - type: map
        map:
          true: LiveMigrate
    - fromFieldPath: metadata.name
      toFieldPath: metadata.name
      transforms:
      - type: string
        string:
          fmt: "%s-vm"

  - name: datavolume
    base:
      apiVersion: cdi.kubevirt.io/v1beta1
      kind: DataVolume
      spec:
        pvc:
          accessModes:
          - ReadWriteMany
          resources:
            requests:
              storage: 1Gi
          storageClassName: default
        source:
          http:
            url: https://placeholder
    patches:
    - fromFieldPath: metadata.name
      toFieldPath: metadata.name
      transforms:
      - type: string
        string:
          fmt: "dv-%s"
    - fromFieldPath: spec.parameters.storage.size
      toFieldPath: spec.pvc.resources.requests.storage
    - fromFieldPath: spec.parameters.storage.storageClass
      toFieldPath: spec.pvc.storageClassName
    - fromFieldPath: spec.parameters.storage.image
      toFieldPath: spec.source.http.url
    connections:
    - fromFieldPath: metadata.name
      toFieldPath: spec.template.spec.volumes[0].dataVolume.name
      from: datavolume
      to: virtualmachine

Decision Matrix

RequirementFunction ChoiceRationale
Build VM spec from XR fields Go-Templating (step 1) Template handles the full spec in one render; conditional evictionStrategy
Create DataVolume with XR values Patch-and-Transform (step 2) Simple field mapping, no loops or branching needed
Link DV PVC → VM volume Patch-and-Transform (step 3) Cross-resource field injection is its specialty
Simple VM, no conditionals Declarative (no functions) Simpler and more auditable for straightforward mappings
