XRD Creation & Processing Pipeline

This explainer walks through the complete lifecycle of a Crossplane-managed VM — from the moment an XRD is defined to the moment a VirtualMachine CR runs on OpenShift. We'll follow the data flow through every component in the stack.

Prerequisite knowledge
You know what CRDs are and have some Crossplane reading under your belt. This explainer bridges that knowledge into a concrete, end-to-end mental model of the XRD pipeline.

1. The Big Picture: Where Each Component Lives

Stack Architecture
You / Git → ArgoCD → Crossplane → KubeVirt Provider → OpenShift Virtualization

Four layers, each with a distinct responsibility:

LayerRoleWhat it manages
Git (source of truth) Declarative intent Claims, XRDs, Compositions, ArgoCD Application manifests
ArgoCD GitOps sync Watches Git, applies manifests to the cluster, ensures drift is corrected
Crossplane Abstraction & orchestration Reconciles Claims → XRs → Compositions → Managed Resources
KubeVirt Provider + OpenShift Virt Infrastructure API Creates and manages VirtualMachine CRs, DataVolumes, networking on OpenShift
Key insight
ArgoCD never touches VMs directly. It syncs Crossplane manifests (XRDs, Compositions, Claims). Crossplane then translates those into KubeVirt resources. ArgoCD is the GitOps conveyor belt; Crossplane is the factory.

2. Step-by-Step: The XRD Processing Pipeline

1

Author the XRD (Composite Resource Definition)

The XRD is the contract. It defines the shape of the infrastructure API you're building. It declares:

  • Names — the API group and kind (e.g., vm.infra.example.com)
  • Spec schema — what fields users fill in (CPU, memory, disk size, image)
  • Status shape — what fields Crossplane reports back (IP, running state, conditions)
  • Composition reference — which Composition handles instances of this XRD
apiVersion: apiextensions.crossplane.io/v1
kind: CompositeResourceDefinition
metadata:
  name: xvirtualmachines.infra.example.com
spec:
  group: infra.example.com
  names:
    kind: XVirtualMachine
    plural: xvirtualmachines
  claimNames:
    kind: VirtualMachineClaim
    plural: virtualmachineclaims
  versions:
  - name: v1alpha1
    served: true
    referenceable: true   # ← XRs of this version can be referenced by other resources
    schema:
      openAPIV3Schema:
        type: object
        properties:
          spec:
            type: object
            required:
            - compute
            - storage
            properties:
              compute:
                type: object
                properties:
                  vcpus:
                    type: integer
                    description: Number of virtual CPUs
                  memory:
                    type: string
                    description: e.g., "8Gi"
              storage:
                type: object
                properties:
                  size:
                    type: string
                    description: e.g., "50Gi"
                  storageClass:
                    type: string
                    description: OpenShift StorageClass name
                  image:
                    type: string
                    description: DataVolume source URL or PVC name
              network:
                type: object
                properties:
                  macAddress:
                    type: string
                    description: Optional fixed MAC address
          status:
            type: object
            properties:
              conditions:
                type: array
                items:
                  type: object
                  properties:
                    type:
                      type: string
                    status:
                      type: string
                    reason:
                      type: string
              endpoint:
                type: string
                description: VM IP or hostname
XRD design note
The claimNames field is what makes this an XRD rather than a plain CRD. It tells Crossplane: "users create VirtualMachineClaim resources, and I'll create the underlying XVirtualMachine XRs automatically." Without claimNames, users would need to create XRs directly — bypassing the abstraction layer.
2

ArgoCD syncs the XRD to the cluster

ArgoCD watches your Git repository. When it sees the XRD manifest, it applies it to the cluster's API server. Crossplane's xrd-controller then picks it up:

  • Crossplane creates the corresponding managed CRD in the cluster (e.g., xvirtualmachines.infra.example.com)
  • Crossplane creates the claim CRD (e.g., virtualmachineclaims.infra.example.com)
  • Both CRDs are now available for users to create instances of
ArgoCD's role here
ArgoCD is only ensuring the XRD manifest exists in the cluster. It has no understanding of Crossplane semantics — it just applies YAML. The actual CRD generation is done entirely by Crossplane's xrd-controller reconciling the XRD object.
3

Author the Composition

The Composition is the recipe. It tells Crossplane what concrete resources to create when an XR is reconciled. For a VM, the Composition typically creates:

  • A KubeVirt VirtualMachine CR (the VM definition)
  • A DataVolume CR (the disk image source)
  • Optionally a PersistentVolumeClaim for additional storage
  • Optionally networking resources (Service, Ingress, or OpenShift Route)

The Composition uses patches to map fields from the XR's spec to the managed resource specs, and connections to pass outputs from one resource to another.

apiVersion: apiextensions.crossplane.io/v1
kind: Composition
metadata:
  name: vm-kubevirt
  labels:
    crossplane.io/xrd: xvirtualmachines.infra.example.com
    infra.example.com/kubevirt: "true"
spec:
  compositeResourceSelector:
    matchLabels:
      infra.example.com/kubevirt: "true"
  mode: Pipeline
  pipeline:
  - step: render-virtualmachine
    functionRef:
      name: crossplane-kubevirt-function
    inputs:
    - name: vm-spec
      ref:
        name: vm-template
        kind: Manifest
        apiVersion: delivery.crossplane.io/v1alpha1
      base:
        apiVersion: kubevirt.io/v1
        kind: VirtualMachine
        metadata:
          labels:
            infra.example.com/managed-by: crossplane
        spec:
          running: true
          template:
            spec:
              domain:
                devices:
                  disks:
                  - name: disk0
                    disk:
                      bus: virtio
              evictionStrategy: LiveMigrate
              networks:
              - name: default-network
                pod: {}
              terminationGracePeriodSeconds: 30
              volumes:
              - name: disk0
                dataVolume:
                  name: placeholder
    - step: patch-vm-from-xr
      functionRef:
        name: crossplane-function-patch-and-transform
      inputs:
        functionConfigRef:
          name: vm-patch-config
    - step: create-datavolume
      functionRef:
        name: crossplane-function-patch-and-transform
      inputs:
        functionConfigRef:
          name: dv-patch-config
    - step: reconcile
      functionRef:
        name: crossplane-function-go-templating
      inputs:
        resources:
        - name: vm
          base:
            apiVersion: kubevirt.io/v1
            kind: VirtualMachine
        - name: dv
          base:
            apiVersion: cdi.kubevirt.io/v1beta1
            kind: DataVolume
Composition modes
Declarative (old style): uses resources array with patches and connections — pure YAML, no functions. Simpler but limited to field-level transforms.
Pipeline (new style): uses steps with functionRef — allows imperative logic, loops, conditionals. More flexible, requires Composition Function controllers.
4

ArgoCD syncs the Composition to the cluster

Same pattern as the XRD — ArgoCD applies the Composition manifest. Crossplane's controller reads the crossplane.io/xrd label to know which XRD this Composition belongs to. The Composition is now "attached" and ready to process XRs.

5

User creates a Claim (or ArgoCD syncs it from Git)

The user (or an ArgoCD Application) creates a VirtualMachineClaim. This is the user-facing resource — it uses the simplified schema defined in the XRD, not the full KubeVirt spec.

apiVersion: infra.example.com/v1alpha1
kind: VirtualMachineClaim
metadata:
  name: web-server
  namespace: workloads
spec:
  compositionSelector:
    matchLabels:
      infra.example.com/kubevirt: "true"
  parameters:
    compute:
      vcpus: 4
      memory: "8Gi"
    storage:
      size: "50Gi"
      storageClass: ocs.external.ceph.rook.io
      image: "https://example.com/centos-stream-9.qcow2"
    network:
      macAddress: "52:54:00:ab:cd:ef"

When the Claim is created, Crossplane's controller:

  • Creates an XVirtualMachine (the XR) in the crossplane-system namespace (or the Claim's namespace, depending on configuration)
  • Copies the Claim's spec.parameters into the XR's spec
  • Links the XR to the Composition via the selector label
6

Crossplane reconciles the XR against its Composition

This is where the magic happens. Crossplane's XR controller calls the Composition's pipeline (or applies its declarative resources):

  1. Render phase: The Composition template is instantiated with the XR's spec values. Patches map spec.compute.vcpus → VirtualMachine spec's domain.cpu.cores, spec.storage.size → PVC size, etc.
  2. Apply phase: Crossplane creates/updates the managed resources (VirtualMachine CR, DataVolume CR) in the cluster using the KubeVirt provider's controller.
  3. Observe phase: Crossplane reads the status of the managed resources and writes it back to the XR's status.
  4. Propagate phase: The XR's status is copied back to the Claim's status, so the user sees the result.
Reconciliation Loop
XR Controller → Composition Pipeline → KubeVirt Provider → KubeVirt Controller → libvirt / VM
7

KubeVirt runs the VM on OpenShift

The KubeVirt controller (part of OpenShift Virtualization) observes the VirtualMachine CR created by Crossplane and:

  • Resolves the DataVolume (downloads or clones the disk image)
  • Creates a VirtualMachineInstance (VMI) CR — this is the actual running VM
  • Provisions the VM using libvirt on an OpenShift node
  • Manages the VM lifecycle (start, stop, migrate, snapshot)

The VM is now running. Crossplane continues to reconcile periodically, ensuring the actual state matches the desired state declared in the Claim.

4. Where ArgoCD Fits in the Full GitOps Flow

ArgoCD operates at two levels in this stack:

Level 1: Infrastructure Bootstrap (one-time or periodic)

ArgoCD syncs the Crossplane infrastructure manifests from Git:

  • crossplane/ — Provider installations, ProviderConfigs, CompositionFunctions
  • xrd/ — XRD definitions
  • compositions/ — Composition manifests
  • providers/ — ProviderConfig (credentials, connection info)

Level 2: Workload Provisioning (per-VM)

ArgoCD syncs individual VM Claims from Git. Each VM is a separate manifest in a namespaced directory:

infrastructure/
├── crossplane/
│   ├── provider-kubevirt.yaml
│   ├── provider-config.yaml
│   └── composition-functions.yaml
├── xrd/
│   └── xvirtualmachine.yaml
├── compositions/
│   └── vm-kubevirt.yaml
└── workloads/
    ├── web-server/
    │   └── claim.yaml
    ├── db-server/
    │   └── claim.yaml
    └── app-server/
        └── claim.yaml

An ArgoCD App-of-Apps Application points to the infrastructure/ and workloads/ directories, creating child Applications for each.

GitOps flow summary
Developer commits a Claim YAML to Git → ArgoCD detects the change → ArgoCD applies the Claim to the cluster → Crossplane reconciles the Claim → Crossplane creates the XR → Composition renders managed resources → KubeVirt provider applies them → VM runs on OpenShift. The entire chain is driven by a Git commit.

5. Key Design Decisions for Your Task Plan

As you build your task plan, these decisions will come up:

DecisionOptionsConsiderations
Composition style Declarative (patches) vs. Pipeline (functions) Pipeline is more powerful but adds operator complexity. Start declarative if your VM spec is straightforward.
Claim namespace Single namespace vs. multi-tenant Crossplane supports RBAC per namespace. For on-prem with multiple teams, plan namespace isolation early.
ProviderConfig auth ServiceAccount vs. kubeconfig secret On OpenShift, ServiceAccount with appropriate RBAC is the recommended approach. The KubeVirt provider needs permissions to manage VirtualMachine, DataVolume, and CDI resources.
Storage provisioning DataVolume downloads vs. pre-provisioned PVCs DataVolume downloads are flexible but slow. Pre-provisioned PVCs with base images are faster for standardized VMs.
ArgoCD sync strategy Auto-sync vs. manual approval For production VMs, manual approval for the workloads layer prevents accidental VM creation/deletion.

Exercise 001: XRD Pipeline Trace — Answer Key

Step 1: Claim → XR (Crossplane's claim controller)

When the VirtualMachineClaim is created, Crossplane's claim controller detects it and automatically creates an XVirtualMachine (the composite resource). The controller copies the Claim's spec.parameters block directly into the XR's spec block.

# Claim (user-facing)
apiVersion: infra.example.com/v1alpha1
kind: VirtualMachineClaim
metadata:
  name: web-server
  namespace: workloads
spec:
  compositionSelector:
    matchLabels:
      infra.example.com/kubevirt: "true"
  parameters:
    storage:
      image: "https://example.com/centos-stream-9.qcow2"
      size: "50Gi"
      storageClass: ocs.external.ceph.rook.io

# XR (created by Crossplane, hidden from user)
apiVersion: infra.example.com/v1alpha1
kind: XVirtualMachine
metadata:
  name: xvirtualmachine-web-server
  namespace: crossplane-system
spec:
  compositionSelector:
    matchLabels:
      infra.example.com/kubevirt: "true"
  storage:
    image: "https://example.com/centos-stream-9.qcow2"
    size: "50Gi"
    storageClass: ocs.external.ceph.rook.io
Key point
The claim controller does a shallow copy of spec.parameters into spec. No transformation happens here — it's a direct field migration.

Step 2: Composition Patch (XR → DataVolume)

The Composition's patch maps the XR field path to the DataVolume field path. The mapping is not 1:1 — the XRD uses spec.storage.image while the CDI API uses spec.source.http.url.

# FunctionConfig referenced by the Composition step
apiVersion: pkg.crossplane.io/v1beta1
kind: FunctionConfig
metadata:
  name: vm-dv-patch-config
spec:
  patches:
  - from: XR
    to: spec.source.http.url
    policy:
      optional: false
Key point
The patch bridges two different API schemas. This translation layer is the core job of the Composition.

Step 3: Resulting DataVolume CR

apiVersion: cdi.kubevirt.io/v1beta1
kind: DataVolume
metadata:
  name: dv-web-server-centos-stream-9
  labels:
    infra.example.com/managed-by: crossplane
spec:
  source:
    http:
      url: "https://example.com/centos-stream-9.qcow2"
  pvc:
    accessModes:
    - ReadWriteMany
    resources:
      requests:
        storage: 50Gi
    storageClassName: ocs.external.ceph.rook.io

Complete Field Flow

Claim.spec.parameters.storage.image
  │
  │  (claim controller: shallow copy spec.parameters → spec)
  ▼
XR.spec.storage.image
  │
  │  (Composition patch: spec.storage.image → spec.source.http.url)
  ▼
DataVolume.spec.source.http.url
  │
  │  (CDI controller: downloads image, creates PVC)
  ▼
PVC backed by downloaded qcow2 image

Common Pitfalls

  • URL accessibility: The URL must be reachable from OpenShift cluster nodes, not the user's machine.
  • TLS verification: CDI verifies TLS by default. Self-signed certs require CA injection or insecure: true.
  • Image format: CDI supports qcow2, raw, and vmdk. Unsupported formats must be converted first.
  • Download timeout: Large images may exceed CDI's default timeout. Adjust spec.individualPVDownloadTimeout for images larger than a few GB.
