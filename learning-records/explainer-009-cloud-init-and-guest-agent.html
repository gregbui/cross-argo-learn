Cloud-init & Guest Agent for VM Workloads

Two pieces make VMs actually useful: cloud-init (provisioning at first boot) and the guest agent (runtime visibility and management). In a Crossplane + KubeVirt stack, these need careful configuration — especially on OpenShift where the SDN, RBAC, and networking differ from public clouds.

1. Cloud-init: First-Boot Provisioning

Cloud-init is the de facto standard for VM initialization. It runs once at first boot (or every boot with runcmd scripts) and configures:

  • Networking — static IPs, DNS, routes, bond interfaces
  • Users & auth — SSH keys, passwords, sudo access
  • Packages — install software, apply patches
  • Services — start/enable daemons
  • Files — drop config files, certificates, scripts
  • Hostname — set the system hostname

In KubeVirt, cloud-init data is injected via the guestInfo or cloudInitNoCloud properties on the VirtualMachine spec. Crossplane compositions generate these from XR spec fields.

2. Cloud-init Document Structure

Cloud-init supports YAML or MIME multi-part documents. YAML is simpler and more common for straightforward use cases:

# Basic cloud-init document
#cloud-config

hostname: web-server-01
fqdn: web-server-01.example.org

users:
  - name: cloud-user
    sudo: ALL=(ALL) NOPASSWD:ALL
    ssh-authorized-keys:
      - ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAI... user@host

ssh_pwauth: false

packages:
  - htop
  - tmux
  - jq

runcmd:
  - systemctl enable qemu-guest-agent
  - systemctl start qemu-guest-agent
  - echo "VM provisioned at $(date)" > /etc/VM_PROVISIONED
  - curl -s http://metadata.internal/ready > /dev/null

final_message: "Cloud-init finished. VM is ready."
write_files:
  - path: /etc/network-config
    content: |
      network:
        version: 2
        ethernets:
          eth0:
            dhcp4: false
            addresses:
              - {{ .XR.spec.network.staticIP }}/24
            gateway4: {{ .XR.spec.network.gateway }}
            nameservers:
              addresses:
                - 10.0.0.10
                - 10.0.0.11

manage_etc_hosts: true

MIME multi-part documents let you mix cloud-init with custom scripts — useful when you need shell scripts that run before cloud-init:

MIME-Version: 1.0
Content-Type: multipart/mixed; boundary="==MYBOUNDARY=="

--==MYBOUNDARY==
Content-Type: text/cloud-config; charset="us-ascii"

#cloud-config
hostname: web-server-01
users:
  - name: cloud-user
    ssh-authorized-keys:
      - ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAI... user@host

--==MYBOUNDARY==
Content-Type: text/x-shellscript; charset="us-ascii"

#!/bin/bash
# Pre-cloud-init setup
echo "Configuring custom NTP..."
timedatectl set-timezone America/Chicago

--==MYBOUNDARY==
Content-Type: text/x-shellscript; charset="us-ascii"

#!/bin/bash
# Post-cloud-init
echo "VM ready at $(date)"
curl -s http://monitoring.internal/register -d '{"name":"{{ .XR.metadata.name }}","env":"{{ .XR.metadata.namespace }}"}'

--==MYBOUNDARY==--

3. Cloud-init on OpenShift

OpenShift adds complexity to cloud-init. The OpenShift SDN (OVN-Kubernetes) handles networking differently than a flat network, and the default user is cloud-user (not ec2-user like AWS).

3.1 Networking on OpenShift

VMs on OpenShift typically use the multus CNI for secondary network interfaces. The primary interface connects to the OpenShift SDN; secondary interfaces connect to workload networks:

┌─────────────────────────────────────────────────┐ │ VirtualMachine │ │ │ │ ┌───────────────────────────────────────────┐ │ │ │ Primary NIC (OVN-Kubernetes) │ │ │ │ - DHCP from OpenShift SDN │ │ │ │ - Pod network namespace │ │ │ │ - IP from OpenShift IP pool │ │ │ └───────────────────────────────────────────┘ │ │ ┌───────────────────────────────────────────┐ │ │ │ Secondary NIC (Multus) │ │ │ │ - Connected to workload network │ │ │ │ - Static IP from cloud-init │ │ │ │ - Isolated from OpenShift SDN │ │ │ └───────────────────────────────────────────┘ │ │ │ │ Multus NetworkAttachmentDefinition: │ │ - workload-net (L2 isolated) │ │ - management-net (management VLAN) │ └─────────────────────────────────────────────────┘

Configure secondary networks via NetworkAttachmentDefinition CRs:

# networks/workload-network.yaml
apiVersion: k8s.cni.cncf.io/v1
kind: NetworkAttachmentDefinition
metadata:
  name: workload-net
  namespace: workloads-dev
spec:
  config: '{
    "cniVersion": "0.3.1",
    "type": "macvlan",
    "master": "ens4f0",
    "mode": "bridge",
    "ipam": {
      "type": "static",
      "ranges": [[{"subnet": "10.100.0.0/24"}]],
      "routes": [{"dst": "10.100.0.0/16", "gateway": "10.100.0.1"}]
    }
  }'

Reference the network in your VirtualMachine composition:

# composition: inject multus network into VM spec
- name: virtualmachine
  patches:
    - type: FromCompositeFieldPath
      fromFieldPath: spec.network.workloadNetwork
      toFieldPath: spec.template.spec.networks
      patchesStringData: |
        - name: workload-net
          multus:
            networkName: workloads-dev/workload-net
    - type: FromCompositeFieldPath
      fromFieldPath: spec.network.interfaces
      toFieldPath: spec.template.spec.domain.devices.interfaces

3.2 OpenShift-Specific cloud-init

On OpenShift, cloud-init needs to handle the fact that the VM is actually a pod. Key differences:

  • No direct SSH to VM IP — the VM IP is a pod IP, accessible only within the cluster. Use oc rsh or set up SSH port-forwarding.
  • Root access requires special handling — OpenShift runs containers as non-root by default. The guest agent runs as root inside the VM, but the pod's init container handles the setup.
  • Firewall — OpenShift's firewalld is managed by the cluster. Don't try to configure it via cloud-init.
#cloud-config
# OpenShift-specific cloud-init

# OpenShift uses cloud-user by default
users:
  - name: cloud-user
    groups: wheel
    sudo: ['ALL=(ALL) NOPASSWD:ALL']
    shell: /bin/bash
    ssh-authorized-keys:
      - {{ .XR.spec.guest.sshKey | default "" }}

# Disable cloud-init's network configuration — OpenShift handles it
network:
  version: 1
  config: disabled

# Use runcmd for OpenShift-specific setup
runcmd:
  # Disable unnecessary services that conflict with OpenShift
  - systemctl disable firewalld 2>/dev/null || true
  - systemctl stop firewalld 2>/dev/null || true

  # Enable guest agent
  - systemctl enable qemu-guest-agent
  - systemctl start qemu-guest-agent

  # Set hostname (OpenShift respects this for VM identification)
  - hostnamectl set-hostname {{ .XR.spec.guest.hostname }}

  # Register with internal monitoring
  - |
    curl -s -X POST http://monitoring.internal/register \
      -H "Content-Type: application/json" \
      -d '{
        "name": "{{ .XR.metadata.name }}",
        "namespace": "{{ .XR.metadata.namespace }}",
        "image": "{{ .XR.spec.imageTemplate }}",
        "env": "{{ .XR.metadata.namespace }}"
      }'

  # Create a ready marker file for ArgoCD health checks
  - echo "ready" > /etc/VM_READY

write_files:
  - path: /etc/ssh/sshd_config.d/openshift.conf
    content: |
      PermitRootLogin prohibit-password
      PasswordAuthentication no
      PubkeyAuthentication yes
      AuthorizedKeysFile .ssh/authorized_keys

  - path: /usr/local/bin/vm-status
    content: |
      #!/bin/bash
      # Simple VM status reporter
      echo "VM: {{ .XR.metadata.name }}"
      echo "Env: {{ .XR.metadata.namespace }}"
      echo "Image: {{ .XR.spec.imageTemplate }}"
      echo "Started: $(uptime)"
      echo "Guest agent: $(systemctl is-active qemu-guest-agent)"
    permissions: '0755'
Warning: cloud-init networking conflicts
If you configure network.config in cloud-init on OpenShift, it can conflict with the OVN-Kubernetes SDN and break VM networking. Always set network: { version: 1, config: disabled } and let OpenShift handle primary networking. Use multus for secondary interfaces only.

4. Guest Agent: Runtime Visibility

The QEMU guest agent runs inside the VM and provides the hypervisor with runtime information. Without it, KubeVirt sees VMs as black boxes — no filesystem access, no graceful shutdown, no metrics.

4.1 What the Guest Agent Provides

Capability Without Agent With Agent
Guest OS name/version Unknown Reported to KubeVirt
IP addresses Only MAC address All guest IPs reported
Filesystem access None virt-copy-in/out, file operations
Graceful shutdown Force power-off only ACPI shutdown signal sent
Live migration Blocked Enabled, with freeze/unfreeze
Disk freeze/thaw Not possible For consistent snapshots
Run commands inside VM Not possible guest-exec support
Memory usage Only host-side metrics Guest-reported memory stats

4.2 Installing and Configuring the Guest Agent

Include guest agent installation in your base image build (see explainer 008):

# In Packer provisioner or cloud-init runcmd
sudo dnf install -y qemu-guest-agent
sudo systemctl enable qemu-guest-agent
sudo systemctl start qemu-guest-agent

# Verify it's running
sudo systemctl status qemu-guest-agent
sudo qemu-ga --version

In the VirtualMachine spec, enable the guest agent device:

# composition: add guest agent to VM spec
- name: virtualmachine
  patches:
    - type: Inline
      patches: |
        spec:
          template:
            spec:
              domain:
                devices:
                  autoattachPodDevice: true
                  autoattachSerialConsole: true
                  # Enable guest agent
                  serials:
                    - name: guestagent
                      devicePath: /dev/virtio-ports/org.qemu.guest_agent.0
                  interfaces:
                    - name: default
                      masquerade: {}
                  disks:
                    - name: rootdisk
                      disk:
                        bus: virtio
                    - name: cloudinitdisk
                      disk:
                        bus: virtio
              guest:
                # Require guest agent for accurate status
                agent: true
              networks:
                - name: default
                  pod: {}

5. Guest Agent Status in Crossplane

The guest agent's status flows through KubeVirt → Crossplane → your application. Understanding this flow is critical for monitoring and health checks.

┌──────────┐ ┌──────────────┐ ┌───────────────┐ │ VM │────>│ KubeVirt │────>│ Crossplane │ │ Guest │ │ VMI Status │ │ XR Status │ │ Agent │ │ │ │ │ │ │ │ guestInfo: │ │ status.atProvider: │ │ • OS │ │ running: │ │ ready: true │ │ • IPs │ │ true │ │ guestInfo: │ │ • FS │ │ os: │ │ os: {...} │ │ • Cmds │ │ name: │ │ ips: [...]│ │ │ │ ... │ │ │ └──────────┘ └──────────────┘ └───────────────┘ │ ▼ ┌────────────────┐ │ ArgoCD │ │ Health Check │ │ │ │ Ready if: │ │ • VM Running │ │ • Agent.Connected │ │ • Guest has IPs │ └────────────────┘

Query guest agent status from the command line:

# Check if guest agent is connected
oc get vmi <vm-name> -n workloads-dev -o jsonpath='{.status.guestOSInfo}'

# Get guest IP addresses (more accurate than pod IPs)
oc get vmi <vm-name> -n workloads-dev -o jsonpath='{.status.interfaces}'

# Run a command inside the guest via the agent
oc virt guest-exec <vm-name> -n workloads-dev -- ls -la /etc

# Get guest filesystem contents
oc virt-copy-out -d <vm-name> /etc/hostname .

# Check agent version inside guest
oc virt-guest-agent-command <vm-name> -n workloads-dev \
  --query '{"execute":"guest-info"}'

In your composition, you can use the guest agent status in conditional logic:

# composition: check guest agent status before proceeding
- step: check-guest-agent
  functionRef:
    name: crossplane-function-go-templating
  inputs:
    - apiVersion: apps.crossplane.io/v1alpha1
      kind: RuntimeConfig
      resources:
        - name: vm-status
          base:
            apiVersion: kubevirt.io/v1
            kind: VirtualMachineInstance
            metadata:
              name: '{{ .XR.metadata.name }}'
              namespace: '{{ .XR.metadata.namespace }}'
          patches:
            - type: FromCompositeFieldPath
              fromFieldPath: metadata.name
              toFieldPath: metadata.name
            - type: FromCompositeFieldPath
              fromFieldPath: metadata.namespace
              toFieldPath: metadata.namespace

6. Injecting Cloud-init via Composition

The composition generates the cloud-init user-data and passes it to KubeVirt via the cloudInitNoCloud or guestInfo source. Here's the full composition pattern:

# compositions/xvirtualmachine-pipeline.yaml
- step: inject-cloudinit
  functionRef:
    name: crossplane-function-patch-and-transform
  inputs:
    - apiVersion: pt.fn.crossplane.io/v1
      kind: Resources
      resources:
        - name: cloudinit-dv
          base:
            apiVersion: cdi.kubevirt.io/v1beta1
            kind: DataVolume
            metadata:
              name: '{{ .XR.metadata.name }}-cloudinit'
              namespace: '{{ .XR.metadata.namespace }}'
            spec:
              pvc:
                accessModes:
                  - ReadWriteOnce
                resources:
                  requests:
                    storage: 1Mi
                storageClassName: 'local-storage'
              source:
                cloudInit:
                  noCloud:
                    secretRef: '{{ .XR.metadata.name }}-cloudinit-secret'
        - name: cloudinit-secret
          base:
            apiVersion: v1
            kind: Secret
            metadata:
              name: '{{ .XR.metadata.name }}-cloudinit-secret'
              namespace: '{{ .XR.metadata.namespace }}'
            type: kubernetes.io/opaque
            stringData:
              user-data: |
                #cloud-config
                hostname: {{ .XR.spec.guest.hostname | default .XR.metadata.name }}
                fqdn: {{ .XR.spec.guest.hostname | default .XR.metadata.name }}.{{ .XR.metadata.namespace }}.svc.cluster.local

                users:
                  - name: cloud-user
                    groups: wheel
                    sudo: ['ALL=(ALL) NOPASSWD:ALL']
                    ssh-authorized-keys:
                    {{- range .XR.spec.guest.linux.sshKeys }}
                      - {{ . }}
                    {{- end }}

                packages:
                  - htop
                  - tmux
                  - jq
                  - qemu-guest-agent

                runcmd:
                  - systemctl enable qemu-guest-agent
                  - systemctl start qemu-guest-agent
                  - echo "VM {{ .XR.metadata.name }} provisioned at $(date)" > /etc/VM_PROVISIONED
                  - echo "ready" > /etc/VM_READY

                write_files:
                  - path: /etc/ssh/sshd_config.d/openshift.conf
                    content: |
                      PermitRootLogin prohibit-password
                      PasswordAuthentication no
                      PubkeyAuthentication yes

                final_message: "Cloud-init finished. VM {{ .XR.metadata.name }} is ready."

The composition creates two resources: a DataVolume that references a Secret containing the cloud-init user-data, and the Secret itself. KubeVirt reads the cloud-init data from the Secret and injects it into the VM at boot via a cloud-init disk.

7. Windows Guest Support

Windows VMs use a different initialization mechanism: Guest Custom Data via the guestInfo cloud-init source, or unattend.xml for Windows-specific provisioning.

7.1 Windows Guest Custom Data (cloud-init for Windows)

KubeVirt supports cloud-init for Windows 2016+ via the guestInfo source. This uses a subset of cloud-init features:

# composition: Windows cloud-init
- name: cloudinit-secret
  base:
    apiVersion: v1
    kind: Secret
    metadata:
      name: '{{ .XR.metadata.name }}-cloudinit-secret'
      namespace: '{{ .XR.metadata.namespace }}'
    type: kubernetes.io/opaque
    stringData:
      user-data: |
        #cloud-config
        hostname: win-server-01
        users:
          - name: Administrator
            ssh-authorized-keys: []
        password: '{{ .XR.spec.guest.windows.adminPassword }}'
        chpasswd:
          expire: false
        runcmd:
          - powershell -Command "Set-NetAdapter -Name 'Ethernet' | Set-NetIPInterface -Dhcp Enabled"
          - powershell -Command "Install-WindowsFeature -Name Web-Server"
        packages: []
        final_message: "Windows VM {{ .XR.metadata.name }} is ready."

7.2 Windows Guest Agent

Windows VMs need the QEMU Guest Agent for Windows installed. Include it in your Windows base image:

# In Packer Windows provisioner
provisioner "powershell" {
  inline = [
    # Download and install QEMU guest agent
    'Invoke-WebRequest -Uri "https://download.qemu.org/qga/qemu-ga-win-*.msi" -OutFile "C:\qemu-ga.msi"',
    'msiexec /i C:\qemu-ga.msi /quiet /norestart',
    # Enable and start the service
    'Start-Service qemu-ga',
    'Set-Service qemu-ga -StartupType Automatic',
  ]
}

In the VirtualMachine spec, enable the Windows guest agent:

# composition: Windows guest agent config
- name: virtualmachine
  patches:
    - type: Inline
      patches: |
        spec:
          template:
            spec:
              domain:
                devices:
                  autoattachPodDevice: true
                  autoattachSerialConsole: false
                  autoattachGraphicsDevice: false
                  # Windows uses virtio-serial for guest agent
                  serials:
                    - name: guestagent
                      devicePath: 'COM1'
              guest:
                agent: true
Note
Windows cloud-init support in KubeVirt is available from version 1.1+. Check your KubeVirt version before using guest custom data for Windows VMs.

8. Troubleshooting

8.1 Cloud-init not running

# Check cloud-init status inside the VM
oc rsh <vm-debug-pod>
# Then into the guest:
systemctl status cloud-init
journalctl -u cloud-init

# Common issue: cloud-init ran but failed
cat /var/log/cloud-init.log
cat /var/log/cloud-init-output.log

# Check if cloud-init disk was attached
oc get vmi <vm-name> -n workloads-dev -o yaml | grep -A5 cloudInit

8.2 Guest agent not connecting

# Check VMI status for agent connection
oc get vmi <vm-name> -n workloads-dev -o jsonpath='{.status.conditions[?(@.type=="AgentConnected")].status}'

# Should return "True". If "False" or missing:
# 1. Check guest agent is installed and running
oc rsh <vm-debug-pod>
systemctl status qemu-guest-agent

# 2. Check virtio-serial device is present
ls -la /dev/virtio-ports/

# 3. Check KubeVirt logs for agent errors
oc logs -n kubevirt $(oc get pods -n kubevirt -l kubevirt.io=virt-handler -o name) | grep agent

8.3 Network conflicts

# If VM lost networking after cloud-init:
# 1. Check if cloud-init modified network config
cat /etc/sysconfig/network-scripts/ifcfg-eth0
# or on newer systems:
cat /etc/NetworkManager/system-connections/*.nmconnection

# 2. Check OVN network status
oc get network-attachment-definitions -n workloads-dev

# 3. Verify multus is working
oc get pods -n multus
oc debug node/<worker-node>
# chroot into the node and check CNI plugins
ls /var/lib/cni/

8.4 SSH not working

# Common causes:
# 1. SSH key not injected — check cloud-init secret
oc get secret <vm-name>-cloudinit-secret -n workloads-dev -o jsonpath='{.data.user-data}' | base64 -d

# 2. SSH service not started
oc rsh <vm-debug-pod>
systemctl status sshd

# 3. Firewall blocking (shouldn't happen on OpenShift, but check)
oc rsh <vm-debug-pod>
firewall-cmd --list-ports  # Should be empty on OpenShift

# 4. Using pod IP to SSH (won't work from outside cluster)
# Solution: use oc rsh or port-forward
oc port-forward <vm-name> 2222:22 -n workloads-dev
ssh -p 2222 cloud-user@localhost
