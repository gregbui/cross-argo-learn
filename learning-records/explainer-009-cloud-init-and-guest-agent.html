Cloud-init, Guest Agent & K8s-Native VM Configuration

VMs need two things to be useful: configuration at boot (how they start) and runtime visibility (how you manage them). In a Crossplane + KubeVirt stack, you have a choice of patterns — traditional cloud-init for first-boot provisioning, and Kubernetes-native patterns (ConfigMaps, Secrets, init containers, readiness probes) that align with GitOps and your existing stack. This explainer covers both, showing when each pattern fits best.

1. Cloud-init: First-Boot Provisioning

Cloud-init is the de-facto standard for first-boot VM provisioning. It runs once when a VM boots for the first time, executing commands, writing files, and configuring users. KubeVirt supports cloud-init via two mechanisms:

Method How It Works Best For
cloudInit.noCloud Injects a cloud-config as a disk (CD-ROM) via a DataVolume or Secret User-data scripts, SSH keys, hostname
cloudInit.configDrive Injects a config-drive ISO (OpenStack-compatible metadata service) Complex networking, metadata service simulation

In KubeVirt, cloud-init is always first-boot only — it runs once and never re-executes on subsequent boots. For runtime configuration changes, you need a different pattern (see Section 3).

2. Cloud-init Document Structure

A cloud-init document is a YAML file with named stages. Each stage runs in a defined order:

cloud-config stages (ordered): │ ├─ #cloud-config │ users → Create users, set SSH keys, add to groups │ ssh_authorized → Inject public keys for passwordless SSH │ packages → Install packages (apt/dnf/yum) │ write_files → Write arbitrary files to the filesystem │ runcmd → Run shell commands (final stage) │ final_message → Log a message when cloud-init finishes │ └─ network: → Configure networking (⚠️ avoid on OpenShift)

Example — a minimal cloud-init that sets a hostname and injects an SSH key:

#cloud-config
hostname: my-vm
users:
  - name: cloud-user
    groups: wheel
    sudo: ['ALL=(ALL) NOPASSWD:ALL']
    ssh-authorized-keys:
      - ssh-ed25519 AAAA... user@host
runcmd:
  - systemctl enable qemu-guest-agent
  - systemctl start qemu-guest-agent
final_message: "VM {{ .XR.metadata.name }} is ready."
K8s-native contrast
Cloud-init is imperative and runs once. In Kubernetes, the equivalent concern — "ensure this resource exists" — is solved declaratively by controllers that reconcile continuously. Cloud-init is a procedural bootstrap; Kubernetes-native patterns are declarative state. For VMs, cloud-init is often the right tool for the bootstrap step, but K8s-native patterns are better for ongoing configuration.

3. Kubernetes-Native Alternatives

Cloud-init is a traditional VM tool. In a Kubernetes-native stack, you have patterns that solve the same problems — config injection, boot-time setup, credential management, health checks — using resources and mechanisms your cluster already understands. This section covers each alternative and when to use it.

Decision guide
  • Use cloud-init for OS-level setup that must happen before the main workload starts: user creation, package installation, firewall rules, hostname, guest agent enablement.
  • Use K8s-native patterns for configuration that should be Git-tracked, version-controlled, and updatable without reimaging: app config, feature flags, connection strings, service registration.
  • Use both when you need OS bootstrap (cloud-init) plus runtime config (K8s-native). This is the most common pattern.

3a. ConfigMaps & Secrets for Configuration Data

Instead of using cloud-init write_files to drop config files into the VM, use Kubernetes ConfigMaps and mount them as volumes. This makes config version-controlled in Git, visible to ArgoCD, and updatable at runtime.

cloud-init Traditional approach — write_files in cloud-init:

#cloud-config
write_files:
  - path: /etc/myapp/config.yaml
    content: |
      database_url: postgres://db:5432/myapp
      feature_flags:
        enable_beta: true
  - path: /etc/myapp/secrets.yaml
    content: |
      api_key: super-secret-key

K8s-native ConfigMap + Secret approach:

# configmaps/myapp-config.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: myapp-config
  namespace: workloads-dev
data:
  config.yaml: |
    database_url: postgres://db:5432/myapp
    feature_flags:
      enable_beta: true
---
# secrets/myapp-secrets.yaml
apiVersion: v1
kind: Secret
metadata:
  name: myapp-secrets
  namespace: workloads-dev
type: Opaque
stringData:
  api_key: super-secret-key

Mount these in the VirtualMachine spec via the KubeVirt guestInfo cloud-init source or a DataVolume:

# composition: mount ConfigMap/Secret into VM
- name: virtualmachine
  patches:
    - type: Inline
      patches: |
        spec:
          template:
            spec:
              domain:
                devices:
                  disks:
                    - name: rootdisk
                      disk: { bus: virtio }
                    - name: config-volume
                      disk: { bus: virtio }
                    - name: secrets-volume
                      disk: { bus: virtio }
              volumes:
                - dataVolume:
                    name: '{{ .XR.metadata.name }}-config-dv'
                  name: config-volume
                - secret:
                    secretName: '{{ .XR.metadata.name }}-secrets'
                  name: secrets-volume
Why this is better for GitOps
ConfigMaps and Secrets are Kubernetes resources — they live in Git, are managed by ArgoCD, and can be updated without touching the VM's disk image. cloud-init write_files embeds config in the VM at first boot; changing it requires either a cloud-init re-run (not supported after first boot) or SSH into the VM and edit manually.

3b. Init Containers for Boot-Time Tasks

Cloud-init runcmd runs shell commands at first boot. The Kubernetes-native equivalent is init containers — they run before the main workload starts and can perform setup tasks like creating directories, downloading files, or running migrations.

cloud-init runcmd in cloud-init:

#cloud-config
runcmd:
  - mkdir -p /etc/myapp/data
  - curl -s https://config-server/config | jq '.production' > /etc/myapp/config.json
  - chmod 600 /etc/myapp/config.json

K8s-native Init container approach:

# compositions/vm-config-init.yaml
- name: virtualmachine
  patches:
    - type: Inline
      patches: |
        spec:
          template:
            spec:
              initContainers:
                - name: setup-config
                  image: curlimages/curl:latest
                  command: ['sh', '-c']
                  args:
                    - |
                      mkdir -p /etc/myapp/data
                      curl -s https://config-server/config \
                        --cacert /etc/tls/ca.crt \
                        | jq '.production' > /etc/myapp/config.json
                      chmod 600 /etc/myapp/config.json
                  volumeMounts:
                    - name: config-data
                      mountPath: /etc/myapp
                  envFrom:
                    - secretRef:
                        name: config-server-credentials
              containers:
                - name: workload
                  image: my-registry/myapp:latest
              volumes:
                - name: config-data
                  configMap:
                    name: myapp-config
                - name: tls-ca
                  secret:
                    secretName: config-server-ca
Important: KubeVirt init containers
KubeVirt VMs run as pods, so init containers are supported. However, the init container shares the VM's filesystem with the main workload — changes persist across the VM lifecycle. For ephemeral setup (run once per boot), this works well. For persistent config, prefer ConfigMap volume mounts (Section 3a).

3c. Crossplane Connection Secrets for Credentials

Instead of injecting SSH keys and passwords via cloud-init, use Crossplane's built-in connection secret mechanism. This is the most K8s-native way to handle credentials in a Crossplane stack.

cloud-init SSH key via cloud-init:

#cloud-config
ssh_authorized_keys:
  - {{ .XR.spec.guest.sshKey }}
write_files:
  - path: /etc/myapp/credentials.env
    content: |
      DB_HOST={{ .XR.spec.database.host }}
      DB_PASS={{ .XR.spec.database.password }}

K8s-native Crossplane connection secret:

# composition: create connection secret from XR spec
- name: connection-secret
  base:
    apiVersion: v1
    kind: Secret
    metadata:
      name: '{{ .XR.metadata.name }}-connection'
      namespace: '{{ .XR.metadata.namespace }}'
      annotations:
        crossplane.io/composition-resource-name: connection-secret
    type: Opaque
  patches:
    - from: XR
      to: metadata
      transforms:
        - type: string
          string:
            fmt: '%s-connection'
    - type: ConnectionSecretKeyRefs
      fieldPath: spec.database
    - type: ConnectionSecretValues
      fieldPath: spec.credentials

The VirtualMachine composition then references this secret:

# composition: inject connection secret into VM
- name: virtualmachine
  patches:
    - type: FromCompositeFieldPath
      from: connection-secret
      toFieldPath: spec.template.spec.volumes[*].secret.secretName
Why connection secrets are better
  • Credentials are managed by Crossplane, not baked into cloud-init YAML
  • Secret rotation is a Crossplane concern — update the secret, and it propagates
  • ArgoCD can manage secret resources in Git without exposing plaintext (use SealedSecrets or SOPS)
  • No SSH into VM needed to rotate credentials

3d. Readiness Probes for Health Checks

Cloud-init has no concept of health checks. Kubernetes-native pods have readiness probes that determine whether a workload is ready to receive traffic. For VMs, you can achieve the same effect through the guest agent and Crossplane XR status conditions.

The flow:

VM guest agent │ │ Reports: OS info, IPs, disk usage, processes ▼ KubeVirt VMI status │ │ .status.guestOSInfo, .status.interfaces, .status.conditions ▼ Crossplane XR status │ │ status.atProvider.ready, status.atProvider.guestInfo ▼ ArgoCD health check │ │ Ready if: │ • VM Running │ • Agent.Connected │ • Guest has IPs │ • Custom readiness marker file

For a custom readiness marker (the closest equivalent to a readiness probe on a VM), use cloud-init or an init container to create a file that ArgoCD checks:

# Init container: signal readiness after setup completes
initContainers:
  - name: readiness-signal
    image: busybox:latest
    command: ['sh', '-c']
    args:
      - |
        # Wait for guest agent to be running
        until systemctl is-active qemu-guest-agent; do
          sleep 2
        done
        # Signal that setup is complete
        echo "ready" > /etc/myapp/READY
        echo "Setup complete at $(date)" >> /var/log/setup.log
    volumeMounts:
      - name: config-data
        mountPath: /etc/myapp

ArgoCD can then check for this file using a custom health check script:

# ArgoCD application health check
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-vm-workload
spec:
  destination:
    namespace: workloads-dev
  source:
    path: apps/my-vm
    repoURL: 'https://git.example.com/infra.git'
  healthChecks:
    - issuer: custom
      version: v1
      tool:
        name: vm-health
        version: v1
        settings:
          valueDirs:
            - ready
K8s-native vs cloud-init health
Readiness probes are the standard K8s pattern for this concern. On VMs, you can't use TCP/HTTP probes directly, so you either use the guest agent (KubeVirt-native) or a readiness marker file (init container + ArgoCD custom check). Both are more GitOps-friendly than cloud-init's implicit "first boot completed" state.

3e. Composition Functions for Dynamic Config

Hardcoding cloud-init YAML in a Composition's patches section doesn't scale. Composition Functions (covered in explainer 002) let you generate config dynamically from XR spec inputs — the most K8s-native way to handle variable VM configuration.

cloud-init Patch-and-Transform (limited):

# composition: hardcoded cloud-init, string interpolation only
- name: cloudinit-secret
  base:
    apiVersion: v1
    kind: Secret
    data:
      user-data: |
        #cloud-config
        hostname: {{ .XR.metadata.name }}
        runcmd:
          - echo "env={{ .XR.spec.environment }}" > /etc/env.conf
  patches:
    - type: FromCompositeFieldPath
      from: environment
      toFieldPath: data.user-data

K8s-native Go-Templating Function (full logic):

# composition: Go-Templating Function generates cloud-init dynamically
- step: generate-config
  functionRef:
    name: crossplane-function-go-templating
  inputs:
    - apiVersion: apps.crossplane.io/v1alpha1
      kind: FunctionRequest
      spec:
        functions:
          - name: go-templating
            configRef:
              name: vm-config-template
  outputs:
    - apiVersion: v1
      kind: Secret
      metadata:
        name: '{{ .XR.metadata.name }}-config'
      data:
        user-data: |
          {{- template "vmUserdata" . }}
          {{- define "vmUserdata" }}
          #cloud-config
          hostname: {{ .xr.metadata.name }}
          users:
            - name: {{ .xr.spec.guest.username | default "cloud-user" }}
          {{- if .xr.spec.guest.sshKey }}
          ssh_authorized_keys:
            - {{ .xr.spec.guest.sshKey }}
          {{- end }}
          runcmd:
            {{- range .xr.spec.guest.packages }}
            - dnf install -y {{ . }}
            {{- end }}
            - systemctl enable qemu-guest-agent
          write_files:
            {{- range .xr.spec.configFiles }}
            - path: {{ .path }}
              content: |
                {{ .content | indent 14 }}
            {{- end }}
          {{- end }}
Why Composition Functions are better
  • Full templating: conditionals, loops, defaults — no string interpolation hacks
  • XR spec drives config: change the XR, the config changes, no Composition edits needed
  • Version-controlled in Git: the template is a Composition Function Config, managed by ArgoCD
  • Reusable across XRDs: same template function can serve multiple resource types

4. Cloud-init on OpenShift

When you do use cloud-init on OpenShift, there are important differences from public cloud environments:

  • No direct SSH to VM IP — the VM IP is a pod IP, accessible only within the cluster. Use oc rsh or SSH port-forwarding.
  • No cloud-init network config — OpenShift's OVN-Kubernetes SDN handles primary networking. Setting network.config in cloud-init will break VM networking.
  • Secondary networks via Multus — use NetworkAttachmentDefinition CRs for additional interfaces.
  • Firewall is cluster-managed — don't try to configure firewalld via cloud-init on OpenShift.

OpenShift-safe cloud-init (minimal, no network config):

#cloud-config
# OpenShift: disable network config, let SDN handle it
network:
  version: 1
  config: disabled

users:
  - name: cloud-user
    groups: wheel
    sudo: ['ALL=(ALL) NOPASSWD:ALL']
    ssh-authorized-keys:
      - {{ .XR.spec.guest.sshKey | default "" }}

runcmd:
  # Enable guest agent
  - systemctl enable qemu-guest-agent
  - systemctl start qemu-guest-agent

  # Set hostname
  - hostnamectl set-hostname {{ .XR.spec.guest.hostname }}

  # Create readiness marker for ArgoCD
  - echo "ready" > /etc/VM_READY

final_message: "VM {{ .XR.metadata.name }} is ready."

Configure secondary networks via NetworkAttachmentDefinition CRs:

# networks/workload-network.yaml
apiVersion: k8s.cni.cncf.io/v1
kind: NetworkAttachmentDefinition
metadata:
  name: workload-net
  namespace: workloads-dev
spec:
  config: '{
    "cniVersion": "0.3.1",
    "type": "macvlan",
    "master": "ens4f0",
    "mode": "bridge",
    "ipam": {
      "type": "static",
      "ranges": [[{"subnet": "10.100.0.0/24"}]],
      "routes": [{"dst": "10.100.0.0/16", "gateway": "10.100.0.1"}]
    }
  }'

Reference the network in your VirtualMachine composition:

# composition: inject multus network into VM spec
- name: virtualmachine
  patches:
    - type: FromCompositeFieldPath
      fromFieldPath: spec.network.workloadNetwork
      toFieldPath: spec.template.spec.networks
      patchesStringData: |
        - name: workload-net
          multus:
            networkName: workloads-dev/workload-net
    - type: FromCompositeFieldPath
      fromFieldPath: spec.network.interfaces
      toFieldPath: spec.template.spec.domain.devices.interfaces
Warning: cloud-init networking conflicts
If you configure network.config in cloud-init on OpenShift, it can conflict with the OVN-Kubernetes SDN and break VM networking. Always set network: { version: 1, config: disabled } and let OpenShift handle primary networking. Use Multus for secondary interfaces only.

5. Guest Agent: Runtime Visibility

The QEMU guest agent runs inside the VM and provides the hypervisor with runtime information. Without it, KubeVirt sees VMs as black boxes — no filesystem access, no graceful shutdown, no metrics.

5.1 What the Guest Agent Provides

Capability Without Agent With Agent
Guest OS name/version Unknown Reported to KubeVirt
IP addresses Only MAC address All guest IPs reported
Filesystem access None virt-copy-in/out, file operations
Graceful shutdown Force power-off only ACPI shutdown signal sent
Live migration Blocked Enabled, with freeze/unfreeze
Disk freeze/thaw Not possible For consistent snapshots
Run commands inside VM Not possible guest-exec support
Memory usage Only host-side metrics Guest-reported memory stats

5.2 Installing and Configuring the Guest Agent

Include guest agent installation in your base image build (see explainer 008):

# In Packer provisioner or cloud-init runcmd
sudo dnf install -y qemu-guest-agent
sudo systemctl enable qemu-guest-agent
sudo systemctl start qemu-guest-agent

# Verify it's running
sudo systemctl status qemu-guest-agent
sudo qemu-ga --version

In the VirtualMachine spec, enable the guest agent device:

# composition: add guest agent to VM spec
- name: virtualmachine
  patches:
    - type: Inline
      patches: |
        spec:
          template:
            spec:
              domain:
                devices:
                  autoattachPodDevice: true
                  autoattachSerialConsole: true
                  serials:
                    - name: guestagent
                      devicePath: /dev/virtio-ports/org.qemu.guest_agent.0
                  interfaces:
                    - name: default
                      masquerade: {}
                  disks:
                    - name: rootdisk
                      disk: { bus: virtio }
                    - name: cloudinitdisk
                      disk: { bus: virtio }
              guest:
                agent: true
              networks:
                - name: default
                  pod: {}

6. Guest Agent Status in Crossplane

The guest agent's status flows through KubeVirt → Crossplane → your application. Understanding this flow is critical for monitoring and health checks.

┌──────────┐ ┌──────────────────┐ ┌─────────────────────┐ │ VM │────>│ KubeVirt │────>│ Crossplane │ │ Guest │ │ VMI Status │ │ XR Status │ │ Agent │ │ │ │ │ │ │ │ guestInfo: │ │ status.atProvider: │ │ • OS │ │ running: true │ │ ready: true │ │ • IPs │ │ os: │ │ guestInfo: │ │ • FS │ │ name: │ │ os: {...} │ │ • Cmds │ │ ... │ │ ips: [...] │ │ │ │ │ │ │ └──────────┘ └──────────────────┘ └─────────────────────┘ │ ▼ ┌────────────────────┐ │ ArgoCD │ │ Health Check │ │ │ │ Ready if: │ │ • VM Running │ │ • Agent.Connected │ │ • Guest has IPs │ └────────────────────┘

Query guest agent status from the command line:

# Check if guest agent is connected
oc get vmi <vm-name> -n workloads-dev -o jsonpath='{.status.guestOSInfo}'

# Get guest IP addresses (more accurate than pod IPs)
oc get vmi <vm-name> -n workloads-dev -o jsonpath='{.status.interfaces}'

# Run a command inside the guest via the agent
oc virt guest-exec <vm-name> -n workloads-dev -- ls -la /etc

# Get guest filesystem contents
oc virt-copy-out -d <vm-name> /etc/hostname .

# Check agent version inside guest
oc virt-guest-agent-command <vm-name> -n workloads-dev \
  --query '{"execute":"guest-info"}'

In your composition, you can use the guest agent status in conditional logic:

# composition: check guest agent status before proceeding
- step: check-guest-agent
  functionRef:
    name: crossplane-function-go-templating
  inputs:
    - apiVersion: apps.crossplane.io/v1alpha1
      kind: RuntimeConfig
      resources:
        - name: vm-status
          base:
            apiVersion: kubevirt.io/v1
            kind: VirtualMachineInstance
            metadata:
              name: '{{ .XR.metadata.name }}'
              namespace: '{{ .XR.metadata.namespace }}'
          patches:
            - type: FromCompositeFieldPath
              fromFieldPath: metadata.name
              toFieldPath: metadata.name
            - type: FromCompositeFieldPath
              fromFieldPath: metadata.namespace
              toFieldPath: metadata.namespace

7. Injecting Config via Composition

The composition generates cloud-init user-data and passes it to KubeVirt via the cloudInitNoCloud or guestInfo source. Here's the full composition pattern, combining cloud-init with K8s-native config injection:

# compositions/xvirtualmachine-pipeline.yaml
- step: inject-cloudinit
  functionRef:
    name: crossplane-function-patch-and-transform
  inputs:
    - apiVersion: pt.fn.crossplane.io/v1
      kind: Resources
      resources:
        - name: cloudinit-dv
          base:
            apiVersion: cdi.kubevirt.io/v1beta1
            kind: DataVolume
            metadata:
              name: '{{ .XR.metadata.name }}-cloudinit'
              namespace: '{{ .XR.metadata.namespace }}'
            spec:
              pvc:
                accessModes:
                  - ReadWriteOnce
                resources:
                  requests:
                    storage: 1Mi
                storageClassName: 'local-storage'
              source:
                cloudInit:
                  noCloud:
                    secretRef: '{{ .XR.metadata.name }}-cloudinit-secret'
        - name: cloudinit-secret
          base:
            apiVersion: v1
            kind: Secret
            metadata:
              name: '{{ .XR.metadata.name }}-cloudinit-secret'
              namespace: '{{ .XR.metadata.namespace }}'
            type: Opaque
          patches:
            - type: Inline
              patches: |
                stringData:
                  user-data: |
                    #cloud-config
                    hostname: {{ .XR.metadata.name }}
                    users:
                      - name: cloud-user
                        groups: wheel
                        sudo: ['ALL=(ALL) NOPASSWD:ALL']
                    runcmd:
                      - systemctl enable qemu-guest-agent
                      - systemctl start qemu-guest-agent
                      - echo "ready" > /etc/VM_READY
                    final_message: "VM {{ .XR.metadata.name }} is ready."

For K8s-native config injection alongside cloud-init, add ConfigMap-based volumes:

# compositions/xvirtualmachine-k8s-config.yaml
- step: inject-k8s-config
  functionRef:
    name: crossplane-function-patch-and-transform
  inputs:
    - apiVersion: pt.fn.crossplane.io/v1
      kind: Resources
      resources:
        - name: config-dv
          base:
            apiVersion: cdi.kubevirt.io/v1beta1
            kind: DataVolume
            metadata:
              name: '{{ .XR.metadata.name }}-config'
              namespace: '{{ .XR.metadata.namespace }}'
            spec:
              pvc:
                accessModes:
                  - ReadWriteOnce
                resources:
                  requests:
                    storage: 10Mi
                storageClassName: 'local-storage'
              source:
                registry:
                  url: 'docker://my-registry/config-init:latest'
        - name: virtualmachine
          base:
            apiVersion: kubevirt.io/v1
            kind: VirtualMachine
          patches:
            - type: Inline
              patches: |
                spec:
                  template:
                    spec:
                      initContainers:
                        - name: apply-config
                          image: busybox:latest
                          command: ['sh', '-c']
                          args:
                            - cp /config/* /data/ && echo "Config applied"
                          volumeMounts:
                            - name: config-volume
                              mountPath: /config
                            - name: data-volume
                              mountPath: /data
                      volumes:
                        - dataVolume:
                            name: '{{ .XR.metadata.name }}-config'
                          name: config-volume
                        - emptyDir: {}
                          name: data-volume

8. Windows Guest Support

Windows VMs need special handling for both cloud-init and the guest agent.

8.1 Windows Cloud-init (guest custom data)

KubeVirt 1.1+ supports Windows guest custom data via cloudInit.guestCustomData. This is a base64-encoded blob that Windows processes via Guest Agent or a custom provisioner:

# composition: Windows guest custom data
- name: virtualmachine
  patches:
    - type: Inline
      patches: |
        spec:
          template:
            spec:
              domain:
                devices:
                  autoattachPodDevice: true
                  autoattachSerialConsole: false
                  autoattachGraphicsDevice: false
                  serials:
                    - name: guestagent
                      devicePath: 'COM1'
              guest:
                agent: true
              cloudInit:
                guestCustomData: |
                  <powershell>
                  New-Item -ItemType Directory -Path "C:\setup" -Force
                  Set-Content -Path "C:\setup\config.json" -Value '{"env":"{{ .XR.spec.environment }}"}'
                  Start-Service qemu-ga
                  </powershell>

8.2 Windows Guest Agent

Windows VMs need the QEMU Guest Agent for Windows installed. Include it in your Windows base image:

# In Packer Windows provisioner
provisioner "powershell" {
  inline = [
    # Download and install QEMU guest agent
    'Invoke-WebRequest -Uri "https://download.qemu.org/qemu-ga-win-*.msi" -OutFile "C:\qemu-ga.msi"',
    'msiexec /i C:\qemu-ga.msi /quiet /norestart',
    # Enable and start the service
    'Start-Service qemu-ga',
    'Set-Service qemu-ga -StartupType Automatic',
  ]
}

In the VirtualMachine spec, enable the Windows guest agent:

# composition: Windows guest agent config
- name: virtualmachine
  patches:
    - type: Inline
      patches: |
        spec:
          template:
            spec:
              domain:
                devices:
                  autoattachPodDevice: true
                  autoattachSerialConsole: false
                  autoattachGraphicsDevice: false
                  # Windows uses virtio-serial for guest agent
                  serials:
                    - name: guestagent
                      devicePath: 'COM1'
              guest:
                agent: true
Note
Windows cloud-init support in KubeVirt is available from version 1.1+. Check your KubeVirt version before using guest custom data for Windows VMs.

9. Troubleshooting

9.1 Cloud-init not running

# Check cloud-init status inside the VM
oc rsh <vm-debug-pod>
# Then into the guest:
systemctl status cloud-init
journalctl -u cloud-init

# Common issue: cloud-init ran but failed
cat /var/log/cloud-init.log
cat /var/log/cloud-init-output.log

# Check if cloud-init disk was attached
oc get vmi <vm-name> -n workloads-dev -o yaml | grep -A5 cloudInit

9.2 Guest agent not connecting

# Check VMI status for agent connection
oc get vmi <vm-name> -n workloads-dev -o jsonpath='{.status.conditions[?(@.type=="AgentConnected")].status}'

# Should return "True". If "False" or missing:
# 1. Check guest agent is installed and running
oc rsh <vm-debug-pod>
systemctl status qemu-guest-agent

# 2. Check virtio-serial device is present
ls -la /dev/virtio-ports/

# 3. Check KubeVirt logs for agent errors
oc logs -n kubevirt $(oc get pods -n kubevirt -l kubevirt.io=virt-handler -o name) | grep agent

9.3 Network conflicts

# If VM lost networking after cloud-init:
# 1. Check if cloud-init modified network config
cat /etc/sysconfig/network-scripts/ifcfg-eth0
# or on newer systems:
cat /etc/NetworkManager/system-connections/*.nmconnection

# 2. Check OVN network status
oc get network-attachment-definitions -n workloads-dev

# 3. Verify multus is working
oc get pods -n multus
oc debug node/<worker-node>
# chroot into the node and check CNI plugins
ls /var/lib/cni/

9.4 SSH not working

# Common causes:
# 1. SSH key not injected — check cloud-init secret
oc get secret <vm-name>-cloudinit-secret -n workloads-dev -o jsonpath='{.data.user-data}' | base64 -d

# 2. SSH service not started
oc rsh <vm-debug-pod>
systemctl status sshd

# 3. Firewall blocking (shouldn't happen on OpenShift, but check)
oc rsh <vm-debug-pod>
firewall-cmd --list-ports  # Should be empty on OpenShift

# 4. Using pod IP to SSH (won't work from outside cluster)
# Solution: use oc rsh or port-forward
oc port-forward <vm-name> 2222:22 -n workloads-dev
ssh -p 2222 cloud-user@localhost

9.5 K8s-native config not applying

# If using init containers or ConfigMaps and config isn't appearing:
# 1. Check init container logs
oc logs <vm-pod> -n workloads-dev -c setup-config

# 2. Check if ConfigMap was created
oc get configmap myapp-config -n workloads-dev -o yaml

# 3. Check if the volume was mounted
oc rsh <vm-pod>
ls -la /etc/myapp/

# 4. Check ArgoCD sync status
oc get applications -n openshift-gitops | grep my-vm

# 5. Check XR status for composition errors
oc get xr <vm-name> -n workloads-dev -o yaml | grep -A10 status
