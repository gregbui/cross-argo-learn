Provider Installation on OpenShift

Crossplane Providers are the bridge between Crossplane's abstraction layer and the infrastructure API. For our VM stack, the KubeVirt provider translates Crossplane managed resources into KubeVirt VirtualMachine and DataVolume CRs. This explainer covers the full installation and configuration flow on OpenShift.

Prerequisite knowledge
You understand XRDs, Compositions, and how Crossplane reconciles resources. Now we're installing the pieces that let Crossplane actually talk to OpenShift's KubeVirt API.

1. What a Crossplane Provider Is

A Crossplane Provider is a container image that bundles two things:

  1. CRDs — the Custom Resource Definitions for the infrastructure resources it manages (e.g., VirtualMachine, DataVolume for KubeVirt)
  2. A controller — a Kubernetes controller that watches those CRDs and reconciles them against the infrastructure API
Provider Architecture
Crossplane creates Managed Resource CR watches Provider Controller calls API KubeVirt API (OpenShift)

For the KubeVirt provider specifically:

  • CRDs installed: VirtualMachine, DataVolume, VirtualMachineInstance, VirtualMachineClusterInstance, and related types from kubevirt.io and cdi.kubevirt.io
  • Controller: reconciles these CRs by calling the OpenShift/KubeVirt API server
  • Authentication: uses a ProviderConfig to determine how to connect (ServiceAccount, kubeconfig, or credentials)

2. Installation Methods

There are two ways to install a Crossplane provider on OpenShift:

Method A: xpkg (recommended for initial setup)

Use the Crossplane CLI to install the provider package. This is a one-time setup command:

# Install the KubeVirt provider
crossplane xpkg install-function \
  crossplane/function-patch-and-transform:v0.14.0

# Install the KubeVirt provider (not a function — a provider)
crossplane xpkg install-provider \
  crossplane/community-provider-kubevirt:v0.18.0

This creates:

  • A Provider CR in the crossplane-system namespace
  • A Deployment running the provider controller
  • All the managed resource CRDs in the cluster

Method B: ArgoCD GitOps (for ongoing management)

For production, manage the provider through GitOps. Create an ArgoCD Application that installs the provider from a manifest:

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: crossplane-provider-kubevirt
  namespace: argocd
spec:
  project: infrastructure
  source:
    repoURL: https://github.com/crossplane-community/provider-kubevirt
    targetRevision: v0.18.0
    path: package/crds
  destination:
    server: https://kubernetes.default.svc
    namespace: crossplane-system
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
GitOps consideration
When managing providers via ArgoCD, be careful with CRD updates. ArgoCD's prune: true will delete CRDs when the provider is removed, which can cause issues if other resources depend on those CRDs. Consider using ignoreDifferences for CRDs that Crossplane manages.

3. RBAC: What the Provider Needs

The KubeVirt provider controller runs as a Deployment in crossplane-system and needs permissions to manage KubeVirt resources. On OpenShift, this requires a ServiceAccount, Role, and RoleBinding.

ServiceAccount

apiVersion: v1
kind: ServiceAccount
metadata:
  name: crossplane-kubevirt-provider
  namespace: crossplane-system

Role

The provider needs permissions to manage all KubeVirt and CDI resources:

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: crossplane-kubevirt-provider
  namespace: crossplane-system
rules:
# VirtualMachine lifecycle
- apiGroups:
  - kubevirt.io
  resources:
  - virtualmachines
  - virtualmachines/finalizers
  - virtualmachines/status
  verbs:
  - get
  - list
  - watch
  - create
  - update
  - patch
  - delete
# VirtualMachineInstance (created by KubeVirt controller, but provider may need to observe)
- apiGroups:
  - kubevirt.io
  resources:
  - virtualmachineinstances
  verbs:
  - get
  - list
  - watch
# DataVolume lifecycle
- apiGroups:
  - cdi.kubevirt.io
  resources:
  - datavolumes
  - datavolumes/finalizers
  - datavolumes/status
  verbs:
  - get
  - list
  - watch
  - create
  - update
  - patch
  - delete
# PVC management (DataVolumes create PVCs)
- apiGroups:
  - ""
  resources:
  - persistentvolumeclaims
  verbs:
  - get
  - list
  - watch
  - create
  - update
  - patch
  - delete

RoleBinding

apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: crossplane-kubevirt-provider
  namespace: crossplane-system
subjects:
- kind: ServiceAccount
  name: crossplane-kubevirt-provider
  namespace: crossplane-system
roleRef:
  kind: Role
  name: crossplane-kubevirt-provider
  apiGroup: rbac.authorization.k8s.io
Namespace scope vs. cluster scope
The Role/RoleBinding above are namespace-scoped to crossplane-system. If your VMs span multiple namespaces, you need either:
  • A ClusterRole + ClusterRoleBinding (broader permissions, simpler setup)
  • Multiple Role + RoleBinding pairs (one per namespace where VMs are created)
For a single-tenant setup, namespace-scoped is fine. For multi-tenant, use ClusterRole.

4. Attaching the ServiceAccount to the Provider

When you install the provider via crossplane xpkg install-provider, it creates a default ServiceAccount. To use our custom ServiceAccount, update the Provider resource:

apiVersion: pkg.crossplane.io/v1beta1
kind: Provider
metadata:
  name: crossplane-community-provider-kubevirt
spec:
  package: crossplane/community-provider-kubevirt:v0.18.0
  revisionActivationPolicy: Automatic
  serviceAccountName: crossplane-kubevirt-provider

This tells Crossplane to run the provider controller pod with our ServiceAccount instead of the default one.

5. ProviderConfig: Connection Configuration

The ProviderConfig tells the provider how to connect to the infrastructure API. For the KubeVirt provider on OpenShift, there are two approaches:

Approach A: In-cluster (ServiceAccount-based)

Since the provider runs inside the OpenShift cluster and already has API access through its ServiceAccount, the ProviderConfig can be minimal:

apiVersion: kubevirt.io.crossplane.io/v1alpha1
kind: ProviderConfig
metadata:
  name: default
spec:
  credentials:
    source: Secret  # or "Injected" for in-cluster auth
    secretRef:
      name: kubevirt-credentials  # optional — may not be needed for in-cluster
      namespace: crossplane-system
      key: credentials
In-cluster auth
On OpenShift, the provider controller can use the cluster's built-in service account token to authenticate to the API server. In many cases, the ProviderConfig's credentials section can be omitted entirely if the ServiceAccount has the right RBAC permissions. The key is RBAC — not credentials.

Approach B: External cluster (kubeconfig-based)

If your Crossplane runs outside the OpenShift cluster (managing it as a managed cluster), you need a kubeconfig:

# Create a secret with the kubeconfig
apiVersion: v1
kind: Secret
metadata:
  name: kubevirt-credentials
  namespace: crossplane-system
type: Opaque
stringData:
  credentials: |
    apiVersion: v1
    kind: Config
    clusters:
    - cluster:
        server: https://kubernetes.default.svc
        certificate-authority-data: LS0tLS1CRUdJTi...
    users:
    - name: provider
      user:
        token: eyJhbGciOiJSUzI1NiIsImtpZCI6...
    contexts:
    - context:
        cluster: openshift
        user: provider
        namespace: crossplane-system
      name: default
    current-context: default

---
apiVersion: kubevirt.io.crossplane.io/v1alpha1
kind: ProviderConfig
metadata:
  name: default
spec:
  credentials:
    source: Secret
    secretRef:
      name: kubevirt-credentials
      namespace: crossplane-system
      key: credentials

6. Installing CompositionFunctions

CompositionFunctions are separate from Providers. They run as sidecar pods during composition rendering and need their own ServiceAccount and permissions.

Patch-and-Transform Function

apiVersion: pkg.crossplane.io/v1beta1
kind: CompositionFunction
metadata:
  name: crossplane-function-patch-and-transform
spec:
  revisionActivationPolicy: Automatic
  package: crossplane/function-patch-and-transform:v0.14.0

Go-Templating Function

apiVersion: pkg.crossplane.io/v1beta1
kind: CompositionFunction
metadata:
  name: crossplane-function-go-templating
spec:
  revisionActivationPolicy: Automatic
  package: crossplane/function-go-templating:v0.40.0

These create Deployments in crossplane-system. The pods run as the default ServiceAccount in that namespace, which needs permissions to read XRs and write managed resources.

OpenShift SCC note
OpenShift's SecurityContextConstraints (SCC) may restrict what containers can do. The default restricted SCC should work for both the provider and function pods. If you see pod startup failures, check the pod events with oc describe pod -n crossplane-system for SCC denials.

7. Verification Checklist

After installation, verify everything is working:

CheckCommandExpected
Provider is installed oc get providers -n crossplane-system Shows crossplane-community-provider-kubevirt with READY = True
Provider pods are running oc get pods -n crossplane-system -l pkg.crossplane.io/provider=name Pod status is Running, no restarts
Managed CRDs exist oc get crd | grep kubevirt Shows VirtualMachine, DataVolume, VirtualMachineInstance CRDs
CompositionFunctions are installed oc get compositionfunctions -n crossplane-system Shows both patch-and-transform and go-templating functions
ProviderConfig is ready oc get providerconfig -n crossplane-system Shows default with READY = True
Crossplane controllers are healthy oc get pods -n crossplane-system All pods Running, 0/0 restarts

8. Try This: Design the RBAC

Exercise
Your team has three namespaces for VM workloads: dev, staging, and prod. Each namespace has its own team that manages VM Claims.
  1. Would you use a namespace-scoped Role or a ClusterRole for the KubeVirt provider? Why?
  2. What additional RBAC might you need for the CompositionFunctions (not the provider itself)?
  3. How would you structure the ProviderConfig if you want to support different authentication per namespace?
Think through each question, then reveal the answers.

Exercise 003: Provider RBAC Design — Answer Key

Q1: Namespace-scoped Role vs. ClusterRole?

ClusterRole is the right choice here. Reasons:

  • The provider controller runs in a single namespace (crossplane-system) but needs to manage resources across multiple namespaces (dev, staging, prod)
  • A namespace-scoped Role only grants permissions in its own namespace — it wouldn't work across dev, staging, and prod
  • A ClusterRole + ClusterRoleBinding gives the provider the needed cross-namespace access
# Use ClusterRole for multi-namespace access
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: crossplane-kubevirt-provider
rules:
- apiGroups:
  - kubevirt.io
  resources:
  - virtualmachines
  - virtualmachines/finalizers
  - virtualmachines/status
  verbs:
  - get
  - list
  - watch
  - create
  - update
  - patch
  - delete
# ... (same resource rules, but ClusterRole is cluster-wide)

---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: crossplane-kubevirt-provider
subjects:
- kind: ServiceAccount
  name: crossplane-kubevirt-provider
  namespace: crossplane-system
roleRef:
  kind: ClusterRole
  name: crossplane-kubevirt-provider
  apiGroup: rbac.authorization.k8s.io

Q2: Additional RBAC for CompositionFunctions?

CompositionFunctions need permissions to:

  • Read XRs and Claims — to access the spec parameters during composition rendering
  • Create/Update managed resources — the output of the composition pipeline
  • Read managed resources — to observe the results of previous pipeline steps

In practice, Crossplane's function runtime handles most of this automatically. The function pods run with the crossplane-system default ServiceAccount, which Crossplane's controller already configures with the necessary permissions. You typically don't need to add extra RBAC for functions.

Q3: ProviderConfig per namespace?

ProviderConfigs are namespaced resources. You can create one per namespace:

# ProviderConfig in each namespace
apiVersion: kubevirt.io.crossplane.io/v1alpha1
kind: ProviderConfig
metadata:
  name: dev
  namespace: dev
spec:
  credentials:
    source: ServiceAccount  # use the namespace's own SA

---
apiVersion: kubevirt.io.crossplane.io/v1alpha1
kind: ProviderConfig
metadata:
  name: staging
  namespace: staging
spec:
  credentials:
    source: ServiceAccount

---
apiVersion: kubevirt.io.crossplane.io/v1alpha1
kind: ProviderConfig
metadata:
  name: prod
  namespace: prod
spec:
  credentials:
    source: ServiceAccount

Each Composition can then reference the appropriate ProviderConfig via a providerConfigRef in the managed resource. However, for the KubeVirt provider on OpenShift, in-cluster auth usually means a single ProviderConfig is sufficient — the RBAC permissions are what matters, not per-namespace credentials.

Practical recommendation
For most OpenShift deployments, a single ClusterRole + ClusterRoleBinding + ProviderConfig in crossplane-system is sufficient. The RBAC gives the provider access to all namespaces, and the ProviderConfig uses in-cluster auth. Per-namespace ProviderConfigs add complexity without much benefit unless you have strict credential isolation requirements.
